# テストレビューガイド

テストの実装が完了した後、仕様カバレッジとテスト品質をレビューする。

## レビュー観点

- spec/testcases/ のカバレッジを満たしているか
- テストが仕様ベースか（実装ベースでないか）
- エッジケースがカバーされているか
- テストケース定義で定義された全テストが漏れなく実装されているか
- 実装と設計の間に乖離がある場合、spec の更新が必要か

## サブエージェントへの指示テンプレート

```
あなたは「テスト」の観点でコードをレビューする専門家です。CLAUDE.md / README.md から読み取れるコンテキストやアーキテクチャに従います。

## レビュー対象
- テストケース定義: spec/testcases/
- ユースケース設計: spec/usecases/
- テストコード: {テストディレクトリ}

## やること
1. テストケース定義を読む
2. テストコードを読む
3. 仕様カバレッジとテスト品質を厳しくレビューする

## レビュー観点
- テストケース定義のカバレッジを満たしているか
- テストが仕様ベースか（実装ベースでないか）
- エッジケースがカバーされているか
- TODO・FIXME・仮実装が残っていないか

## 重要: 独立したレビュー
このレビューは前回のラウンドとは独立している。テストケース定義とテストコードだけを見て、ゼロからレビューする。

## 出力フォーマット

#### Blockers（マージ不可の問題）
- **[B-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ問題か
  - 提案: どう直すべきか

#### Warnings（改善推奨）
- **[W-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ気になるか
  - 提案: 改善案

Blocker がなければ「なし」と書く。
テストケース定義のカバレッジ不足は Blocker として報告する。
TODO・FIXME・仮実装が残っている場合も Blocker として報告する。
```

## レビューファイルの作成

結果を `.review/test/review-{連番}.md` にまとめる。

```markdown
# Test Review #{連番}

**Date:** {日付}
**Round:** {N}回目

---

## Summary

- Blockers: {数}
- Warnings: {数}
- Verdict: **BLOCKED** / **APPROVED**

---

{レビュー結果}
```

## 修正ループ

Blocker・Warning がある場合:

1. Blocker から優先的に修正する
2. Warning もすべて修正する
3. 修正はサブエージェントに委譲する（指摘内容と対象ファイルを渡す）
4. 修正後、品質ゲート（typecheck + lint + test run）を実行する
5. 再レビュー（次の連番ファイルを作る）

## 完了条件

**2回連続で Blocker 0件 かつ Warning 0件** になったらレビュー完了。最大5ラウンド。

5ラウンドで収束しない場合は残っている指摘をまとめてユーザーに報告し、判断を委ねる。
