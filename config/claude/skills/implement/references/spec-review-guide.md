# Phase 1: Spec レビューガイド

設計ドキュメントの品質を実装前に確認する。レイヤーごとに並列でサブエージェントを起動し、独立してレビューさせる。

## レビュー対象と観点

### シナリオ（spec/scenario/）

レビュー出力先: `spec/scenario/review/{連番}.md`

観点:
- 利用シナリオが要件を網羅しているか
- 利用者の視点で具体的なフローが分かるように記載されているか
- フローに不自然な迂回や無駄なステップがないか

### ページ設計（spec/pages/）

レビュー出力先: `spec/pages/review/{連番}.md`

観点:
- 要件を実現するために適切なページ構成になっているか
- レイアウトやコンポーネントの粒度が適切か
- UXが考慮されているか（ページ遷移、タブ、モーダル等の使い分け）

### ドメイン設計（spec/domains/）

レビュー出力先: `spec/domains/review/{連番}.md`

観点:
- エンティティや値オブジェクトが適切に定義され、型安全であるか
- 集約ルートが適切に定義され、整合性が保たれているか
- ビジネスロジックがドメインモデルに適切に配置されているか
- 外部依存は適切なポートとして定義されているか

### ユースケース設計（spec/usecases/）

レビュー出力先: `spec/usecases/review/{連番}.md`

観点:
- ユースケースが要件を満たしているか
- ユースケースが適切に分割され、単一責任の原則に従っているか
- フロントエンドで行うべき処理がユースケースに含まれていないか

### DB設計（spec/database/）

レビュー出力先: `spec/database/review/{連番}.md`

観点:
- 適切に正規化されているか
- パフォーマンスに問題がないか
- データ整合性が保証されているか

## サブエージェントへの指示テンプレート

```
あなたは「{対象レイヤー}」の設計品質をレビューする専門家です。

## レビュー対象
- 設計ファイル: {spec/対象パス}

## やること
1. 設計ファイルを読む
2. 上記の観点でレビューする
3. 対応が必要な問題点のみ記載する（良い点の列挙は不要）

## 出力フォーマット
問題がある場合:
- **[P-001]** 問題の説明
  - 場所: 該当箇所
  - 理由: なぜ問題か
  - 提案: どう修正すべきか

問題がない場合: 「問題なし」とだけ書く。
```

## レビュー → 修正ループ

問題があればサブエージェントで修正させ、再レビューする。クリーンになるまで繰り返す（最大3回）。
