# Implement — 設計ドキュメントからの段階的実装オーケストレーター

`spec/` の設計ドキュメントに基づき、レイヤーごとに実装を進める。各レイヤーの実装とレビューはサブスキルに委譲し、本スキルは全体の進行管理と統合品質ゲートに集中する。

**メインエージェントの役割はオーケストレーションのみ。** コードの読み書きはすべてサブエージェントに委譲する。メインエージェントがやるのは、サブスキルの手順に従った実装の指揮・進捗管理・次フェーズへの判断だけ。

## なぜこの順序か

ドメインモデル → アダプター → ユースケース → テスト → フロントエンドの順で実装する。内側のレイヤーから外側へ向かうことで、各レイヤーが依存先の安定した基盤の上に構築される。テストはユースケース実装後に書くことで、仕様に忠実なテストが書ける。フロントエンドは最後に回すことで、バックエンドの API が固まった状態でページを組める。

各レイヤーは実装直後にレビューサイクルを回す。問題を早期に発見・修正することで、後続レイヤーが不安定な基盤の上に積み上がるリスクを防ぐ。

## Workflow Overview

```
Phase 0: 準備
  spec/ の存在確認 → プロジェクトコンテキスト把握 → 進捗ファイル確認 → 途中再開 or 新規開始

Phase 1: Spec レビュー（省略可）
  設計ドキュメントの品質を並列レビュー → 問題があれば修正

Phase 2: ドメイン実装＋レビュー
  implement-domain の手順に従う → レビューサイクル完了まで

Phase 3: アダプター実装＋レビュー
  implement-adapter の手順に従う → レビューサイクル完了まで

Phase 4: ユースケース実装＋レビュー
  implement-usecase の手順に従う → レビューサイクル完了まで

Phase 5: テスト実装＋レビュー
  implement-test の手順に従う → レビューサイクル完了まで

Phase 6: フロントエンド実装＋レビュー
  implement-frontend の手順に従う → レビューサイクル完了まで

Phase 7: 統合品質ゲート
  全体 typecheck + lint + test run

Phase 8: 完了報告
```

途中のフェーズから再開する場合は、進捗ファイルを確認して該当フェーズから開始する。

---

## Phase 0: 準備

### spec/ の確認

以下の設計ドキュメントが存在するか確認する:

- `spec/index.md` — 設計インデックス
- `spec/scenario/` — 利用シナリオ
- `spec/domains/` — ドメイン設計
- `spec/usecases/` — ユースケース設計
- `spec/pages/` — ページ設計（フロントエンドがある場合）
- `spec/database/` — DB 設計（アダプターがある場合）
- `spec/testcases/` — テストケース定義
- `spec/design/` — UIデザイン成果物（design-flow スキルの出力。フロントエンドがある場合）

存在しない場合はユーザーに確認する。spec スキルで設計を先に作成する必要があるかもしれない。

### プロジェクトコンテキストの把握

CLAUDE.md と README.md を読み、以下を確認する:

- 使用言語・フレームワーク
- ディレクトリ構成（ドメイン・アダプター・ユースケース・フロントエンドの配置先）
- typecheck・lint・test の実行コマンド
- コーディング規約

これらの情報はサブエージェントへの指示に含める。プロジェクトごとに言語もディレクトリ構成も異なるため、spec とプロジェクトコンテキストの両方に従って実装する。

### 進捗管理

`spec/progress.md` で実装進捗を管理する。なければ新規作成する。

```markdown
# 実装進捗

## ドメイン
- [ ] {domain名}
- [ ] {domain名}

## アダプター
- [ ] {adapter名}

## ユースケース
- [ ] {domain名}
- [ ] {domain名}

## テスト
- [ ] {domain名}

## フロントエンド
- [ ] {page名}
```

各フェーズの開始時に `[ ]` → `[-]`（進行中）、完了時に `[x]` に更新する。備考があれば行末に追記する。

### 実装例の確認

`docs/backend_implementation_example.md` や `docs/frontend_implementation_example.md` が存在するか確認する。存在する場合はサブエージェントに参照先として渡す。

### 共通

- Phase 0 の確認が完了したら、以降は最後まで自走する（フェーズ間でユーザー承認を挟まない）
- ただし設計に問題がある場合や判断に迷う場合はユーザーに確認する

## Phase 1: Spec レビュー（省略可）

`references/spec-review-guide.md` を読み、その手順に従って設計ドキュメントをレビューする。

既にレビュー済み（`spec/*/review/` にレビューファイルが存在し、問題がクリーンになっている）の場合はスキップする。

成果物:
- `spec/{対象}/review/{連番}.md` — レビュー記録

問題がなくなるまでレビュー → 修正を繰り返す。クリーンになったら Phase 2 へ。

---

## Phase 2: ドメイン実装＋レビュー

`../implement-domain/SKILL.md` を読み、その Phase 2（実装）と Phase 3（レビューサイクル）の手順に従う。Phase 1（準備）は Phase 0 で完了済みのため省略する。

レビューサイクルが完了したら Phase 3 へ。

---

## Phase 3: アダプター実装＋レビュー

`../implement-adapter/SKILL.md` を読み、その Phase 2（実装）と Phase 3（レビューサイクル）の手順に従う。

レビューサイクルが完了したら Phase 4 へ。

---

## Phase 4: ユースケース実装＋レビュー

`../implement-usecase/SKILL.md` を読み、その Phase 2（実装）と Phase 3（レビューサイクル）の手順に従う。

レビューサイクルが完了したら Phase 5 へ。

---

## Phase 5: テスト実装＋レビュー

`../implement-test/SKILL.md` を読み、その Phase 2（実装）と Phase 3（レビューサイクル）の手順に従う。

レビューサイクルが完了したら Phase 6 へ。

---

## Phase 6: フロントエンド実装＋レビュー

`../implement-frontend/SKILL.md` を読み、その Phase 2（実装）と Phase 3（レビューサイクル）の手順に従う。

フロントエンドがないプロジェクトではスキップする。レビューサイクルが完了したら Phase 7 へ。

---

## Phase 7: 統合品質ゲート

全レイヤーの実装とレビューが完了したら、プロジェクト全体で品質チェックを実行する。

```
1. typecheck（CLAUDE.md のコマンドに従う）
2. lint（CLAUDE.md のコマンドに従う）
3. test run（CLAUDE.md のコマンドに従う）
```

エラーがある場合はサブエージェントを起動して修正させる。サブエージェントにはエラー内容と関連ファイルを渡す。

すべてパスするまで繰り返す（最大3回）。3回で収束しない場合はユーザーに報告して判断を仰ぐ。

---

## Phase 8: 完了報告

すべてのフェーズが完了したら、サマリーを出す。

```
実装が完了しました！

## 進捗
- spec/progress.md を参照

## 実装レイヤー
- ドメイン: {数} ドメイン
- アダプター: {数} 個
- ユースケース: {数} 個
- テスト: {数} ファイル
- フロントエンド: {数} ページ

## 品質
- typecheck: PASS
- lint: PASS
- test: PASS ({数} tests)

## レビュー
- 各レイヤーのレビュー記録: .review/{layer}/ を参照
```

## 原則

- **メインエージェントはオーケストレーションのみ** — コードの読み書きはサブエージェントに委譲する
- **プロジェクトコンテキストに従う** — 言語・ディレクトリ構成・規約は CLAUDE.md / README.md から読み取る。ハードコードされたパスを前提にしない
- 設計ドキュメントに忠実に実装する — スコープ外の改善を含めない
- 各フェーズは前フェーズの成果に基づいて進める — 飛ばさない（部分実行の場合を除く）
- **並列実行時はドメイン間の依存関係を尊重する** — 依存先を先に実装してから依存元を起動する
- サブエージェントが失敗・タイムアウトした場合は、成功した結果だけで柔軟に進める
- 品質チェック（typecheck + lint + test）はごまかさない — すべてパスするまで修正する
- **TODO・FIXME・仮実装を残さない** — サブエージェントには設計で定義された全機能を完全に実装させる。「後で実装」は許可しない。実装が困難な場合はその旨を報告させ、メインエージェントが判断する
- **フェーズ完了ごとに git commit する** — 途中で中断しても成果が失われないようにする
- `spec/progress.md` はフェーズ完了ごとに更新し、進捗を反映する
- **spec ドキュメントを実装と同期させる** — 実装中に設計の誤りや不足を発見したら、実装に合わせて spec を更新する。設計と実装が乖離したまま放置しない
- **設計判断は `.adr/` に記録する** — 実装中に設計を変更・拡張した場合、その判断と理由を `.adr/{連番}-{topic}.md` に ADR（Architecture Decision Record）として残す。後から「なぜこうなっているか」を追えるようにする
- **サブエージェントには必ずプロジェクトルートの CLAUDE.md を最初に読むよう指示する** — コマンド・規約・構成の情報源として
