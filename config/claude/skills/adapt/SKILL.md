既存のデザインを異なるコンテキスト（画面サイズ、デバイス、プラットフォーム、ユースケース）で効果的に機能するように適応させます。

## 視覚的な評価

HTMLファイルを対象とする場合、スクリーンショットを撮影して現在の各viewportでの表示を確認する。

```bash
bash <skill-path>/../critique/scripts/screenshot.sh /path/to/target.html /tmp/adapt-screenshots
```

`Read` ツールで各画像（desktop.png, tablet.png, mobile.png）を読み込んでから作業を開始する。adaptスキルでは特に複数viewportの比較が重要。

## 適応課題の評価

何を適応させる必要があるか、なぜ必要かを理解する：

1. **元のコンテキストを特定する**：
   - 元々何のために設計されたか？（デスクトップWeb？モバイルアプリ？）
   - どのような前提があったか？（大画面？マウス入力？高速接続？）
   - 現在のコンテキストで何がうまく機能しているか？

2. **ターゲットコンテキストを理解する**：
   - **デバイス**: モバイル、タブレット、デスクトップ、TV、ウォッチ、印刷？
   - **入力方法**: タッチ、マウス、キーボード、音声、ゲームパッド？
   - **画面の制約**: サイズ、解像度、向き？
   - **接続環境**: 高速Wi-Fi、低速3G、オフライン？
   - **使用状況**: 移動中 vs デスクワーク、一瞥 vs 集中読書？
   - **ユーザーの期待**: このプラットフォームでユーザーが期待するものは？

3. **適応の課題を特定する**：
   - 何が収まらないか？（コンテンツ、ナビゲーション、機能）
   - 何が動作しないか？（タッチでのホバー状態、小さなタッチターゲット）
   - 何が不適切か？（モバイルでのデスクトップパターン、デスクトップでのモバイルパターン）

**重要**: 適応はスケーリングだけではありません。新しいコンテキストに合わせて体験を再考することです。

## 適応戦略の計画

コンテキストに適した戦略を作成する：

### モバイル適応（デスクトップ → モバイル）

**レイアウト戦略**：
- マルチカラムではなくシングルカラム
- 横並びではなく縦方向のスタッキング
- 固定幅ではなくフルワイドコンポーネント
- 上部/サイドナビゲーションではなくボトムナビゲーション

**インタラクション戦略**：
- タッチターゲットは最小44x44px（ホバー依存不可）
- 適切な場面でスワイプジェスチャー（リスト、カルーセル）
- ドロップダウンではなくボトムシート
- 親指ファーストデザイン（親指が届く範囲にコントロール）
- より大きなタップ領域とより広い間隔

**コンテンツ戦略**：
- 段階的開示（すべてを一度に表示しない）
- 主要コンテンツを優先（二次的コンテンツはタブ/アコーディオンに）
- より短いテキスト（より簡潔に）
- より大きなテキスト（最小16px）

**ナビゲーション戦略**：
- ハンバーガーメニューまたはボトムナビゲーション
- ナビゲーションの複雑さを軽減
- コンテキスト維持のための固定ヘッダー
- ナビゲーションフローでの戻るボタン

### タブレット適応（ハイブリッドアプローチ）

**レイアウト戦略**：
- 2カラムレイアウト（シングルでも3カラムでもなく）
- 二次的コンテンツ用のサイドパネル
- マスター・ディテールビュー（一覧＋詳細）
- 向きに応じた適応（縦向き vs 横向き）

**インタラクション戦略**：
- タッチとポインターの両方をサポート
- タッチターゲットは44x44pxだが、スマートフォンより密なレイアウトを許容
- サイドナビゲーションドロワー
- 適切な場面でのマルチカラムフォーム

### デスクトップ適応（モバイル → デスクトップ）

**レイアウト戦略**：
- マルチカラムレイアウト（水平スペースを活用）
- サイドナビゲーションを常に表示
- 複数の情報パネルを同時表示
- max-width制約付き固定幅（4Kに引き伸ばさない）

**インタラクション戦略**：
- 追加情報のためのホバー状態
- キーボードショートカット
- 右クリックコンテキストメニュー
- 有用な場面でのドラッグ＆ドロップ
- Shift/Cmdでの複数選択

**コンテンツ戦略**：
- より多くの情報を前面に表示（段階的開示を減らす）
- 多数のカラムを持つデータテーブル
- より豊かなビジュアライゼーション
- より詳細な説明

### 印刷適応（画面 → 印刷）

**レイアウト戦略**：
- 論理的なポイントでの改ページ
- ナビゲーション、フッター、インタラクティブ要素の削除
- モノクロ（または限られた色）
- 製本用の適切なマージン

**コンテンツ戦略**：
- 短縮されたコンテンツの展開（完全なURL、非表示セクションの表示）
- ページ番号、ヘッダー、フッターの追加
- メタデータの含有（印刷日、ページタイトル）
- チャートの印刷向けバージョンへの変換

### メール適応（Web → メール）

**レイアウト戦略**：
- 狭い幅（最大600px）
- シングルカラムのみ
- インラインCSS（外部スタイルシートなし）
- テーブルベースのレイアウト（メールクライアント互換性のため）

**インタラクション戦略**：
- 大きく明確なCTA（テキストリンクではなくボタン）
- ホバー状態なし（信頼性が低い）
- 複雑なインタラクション用のWebアプリへのディープリンク

## 適応の実装

変更を体系的に適用する：

### レスポンシブブレークポイント

適切なブレークポイントを選択：
- モバイル: 320px-767px
- タブレット: 768px-1023px
- デスクトップ: 1024px以上
- またはコンテンツ駆動のブレークポイント（デザインが崩れる場所）

### レイアウト適応テクニック

- **CSS Grid/Flexbox**: レイアウトの自動リフロー
- **コンテナクエリ**: ビューポートではなくコンテナに基づく適応
- **`clamp()`**: 最小値と最大値間の流動的なサイジング
- **メディアクエリ**: 異なるコンテキストに異なるスタイル
- **displayプロパティ**: コンテキストごとの要素の表示/非表示

### タッチ適応

- タッチターゲットサイズの拡大（最小44x44px）
- インタラクティブ要素間のスペースを追加
- ホバー依存のインタラクションを削除
- タッチフィードバックの追加（リップル、ハイライト）
- 親指ゾーンの考慮（上部より下部の方が届きやすい）

### コンテンツ適応

- `display: none`は控えめに使用（ダウンロードは発生する）
- プログレッシブエンハンスメント（まずコアコンテンツ、大画面で拡張）
- 画面外コンテンツの遅延読み込み
- レスポンシブ画像（`srcset`、`picture`要素）

### ナビゲーション適応

- モバイルで複雑なナビをハンバーガー/ドロワーに変換
- モバイルアプリ用のボトムナビバー
- デスクトップでの常時表示サイドナビゲーション
- 小さい画面でのコンテキスト用パンくずリスト

**重要**: ブラウザのDevToolsだけでなく、実際のデバイスでテストしてください。デバイスエミュレーションは有用ですが完璧ではありません。

**絶対にしてはいけないこと**：
- モバイルでコア機能を隠す（重要なら機能させる）
- デスクトップ＝高性能デバイスと仮定する（アクセシビリティ、古いマシンを考慮）
- コンテキスト間で異なる情報アーキテクチャを使用する（混乱の原因）
- プラットフォームのユーザー期待を裏切る（モバイルユーザーはモバイルパターンを期待）
- モバイル/タブレットの横向きを忘れる
- 汎用的なブレークポイントを盲目的に使用する（コンテンツ駆動のブレークポイントを使用）
- デスクトップでのタッチを無視する（多くのデスクトップデバイスにタッチがある）

## 適応の検証

コンテキスト間で徹底的にテストする：

- **実際のデバイス**: 実際のスマートフォン、タブレット、デスクトップでテスト
- **異なる向き**: 縦向きと横向き
- **異なるブラウザ**: Safari、Chrome、Firefox、Edge
- **異なるOS**: iOS、Android、Windows、macOS
- **異なる入力方法**: タッチ、マウス、キーボード
- **エッジケース**: 非常に小さい画面（320px）、非常に大きい画面（4K）
- **低速接続**: スロットルされたネットワークでテスト

覚えておいてください：あなたはクロスプラットフォームデザインの専門家です。各コンテキストにネイティブに感じる体験を作りながら、ブランドと機能の一貫性を維持してください。意図的に適応し、徹底的にテストしてください。
