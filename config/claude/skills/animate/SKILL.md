機能を分析し、理解を深め、フィードバックを提供し、楽しさを生み出すアニメーションとマイクロインタラクションを戦略的に追加します。

## 必須準備

### コンテキスト収集（最初にこれを行う）

ターゲットオーディエンス（重要）、想定ユースケース（重要）、ブランドの個性/トーン（遊び心 vs 真面目、活発 vs 穏やか）、パフォーマンス制約などの必要なコンテキストなしには、優れた仕事はできません。

現在のスレッドやコードベースからこれらを収集してください。

1. *正確な*情報が見つからず、既存のデザインや機能から推測する必要がある場合は、必ず停止し、AskUserQuestionToolを呼び出して確認してください。
2. それ以外の場合、完全に推測できないか、確信度が中程度以下の場合は、必ず停止し、AskUserQuestionToolを呼び出して明確化のための質問をしてください。

回答を得るまで進めないでください。推測は不適切または過剰なアニメーションにつながります。

### frontend-designスキルを使用する

デザイン原則とアンチパターンについてはfrontend-designスキルを使用してください。実行され、すべてのDOとDON'Tを把握するまで進めないでください。

## アニメーション機会の評価

モーションが体験を向上させる箇所を分析する：

1. **静的な領域を特定する**：
   - **フィードバック不足**: 視覚的な確認がないアクション（ボタンクリック、フォーム送信など）
   - **唐突なトランジション**: 突然の状態変化（表示/非表示、ページ読み込み、ルート変更）
   - **不明確な関係**: 明確でない空間的または階層的な関係
   - **楽しさの欠如**: 機能的だが喜びのないインタラクション
   - **ガイダンス不足**: 注意を誘導したり動作を説明する機会

2. **コンテキストを理解する**：
   - 個性は？（遊び心 vs 真面目、活発 vs 穏やか）
   - パフォーマンス予算は？（モバイルファースト？複雑なページ？）
   - オーディエンスは？（モーション敏感なユーザー？速度を求めるパワーユーザー？）
   - 最も重要なことは？（一つのヒーローアニメーション vs 多くのマイクロインタラクション？）

コードベースからこれらが不明な場合は、停止してAskUserQuestionToolを呼び出してください。

**重要**: `prefers-reduced-motion`を尊重してください。モーション軽減が必要なユーザーのために、常にアニメーションなしの代替を提供してください。

## アニメーション戦略の計画

意図的なアニメーション計画を作成する：

- **ヒーローモーメント**: 代表的なアニメーションは何か？（ページ読み込み？ヒーローセクション？キーインタラクション？）
- **フィードバックレイヤー**: どのインタラクションに確認が必要か？
- **トランジションレイヤー**: どの状態変化にスムージングが必要か？
- **楽しさレイヤー**: どこでサプライズと楽しさを提供できるか？

**重要**: 一つの統合された体験が、散在するアニメーションに勝ります。インパクトの高い瞬間に集中してください。

## アニメーションの実装

これらのカテゴリーにわたって体系的にモーションを追加する：

### エントランスアニメーション
- **ページ読み込みの演出**: 要素の段階的表示（100-150msの遅延）、フェード＋スライドの組み合わせ
- **ヒーローセクション**: プライマリコンテンツの劇的な登場（スケール、パララックス、クリエイティブなエフェクト）
- **コンテンツの表示**: Intersection Observerを使用したスクロールトリガーアニメーション
- **モーダル/ドロワーの登場**: スムーズなスライド＋フェード、背景のフェード、フォーカス管理

### マイクロインタラクション
- **ボタンフィードバック**：
  - ホバー: 微妙なスケール（1.02-1.05）、色の変化、シャドウの増加
  - クリック: 素早いスケールダウンからアップ（0.95 → 1）、リップルエフェクト
  - ローディング: スピナーまたはパルス状態
- **フォームインタラクション**：
  - 入力フォーカス: ボーダーの色遷移、微妙なスケールまたはグロー
  - バリデーション: エラー時のシェイク、成功時のチェックマーク、スムーズな色遷移
- **トグルスイッチ**: スムーズなスライド＋色遷移（200-300ms）
- **チェックボックス/ラジオ**: チェックマークアニメーション、リップルエフェクト
- **いいね/お気に入り**: スケール＋回転、パーティクルエフェクト、色遷移

### 状態トランジション
- **表示/非表示**: フェード＋スライド（即座でなく）、適切なタイミング（200-300ms）
- **展開/折りたたみ**: オーバーフロー処理付き高さ遷移、アイコンの回転
- **ローディング状態**: スケルトンスクリーンのフェード、スピナーアニメーション、プログレスバー
- **成功/エラー**: 色遷移、アイコンアニメーション、穏やかなスケールパルス
- **有効/無効**: 不透明度遷移、カーソル変更

### ナビゲーション＆フロー
- **ページ遷移**: ルート間のクロスフェード、共有要素のトランジション
- **タブ切り替え**: スライドインジケーター、コンテンツのフェード/スライド
- **カルーセル/スライダー**: スムーズなtransform、スナップポイント、モメンタム
- **スクロールエフェクト**: パララックスレイヤー、状態変化を伴う固定ヘッダー、スクロール進捗インジケーター

### フィードバック＆ガイダンス
- **ホバーヒント**: ツールチップのフェードイン、カーソル変更、要素のハイライト
- **ドラッグ＆ドロップ**: リフトエフェクト（シャドウ＋スケール）、ドロップゾーンのハイライト、スムーズな再配置
- **コピー/ペースト**: ペースト時の短いハイライトフラッシュ、「コピーしました」の確認
- **フォーカスフロー**: フォームやワークフロー内のハイライトパス

### 楽しさの瞬間
- **空の状態**: イラストの微妙な浮遊アニメーション
- **完了アクション**: 紙吹雪、チェックマークの華やかな表示、成功のお祝い
- **イースターエッグ**: 発見のための隠しインタラクション
- **コンテキストアニメーション**: 天気エフェクト、時間帯テーマ、季節の演出

## 技術的な実装

各アニメーションに適切な技法を使用する：

### タイミングとイージング

**目的別の持続時間：**
- **100-150ms**: 即時フィードバック（ボタン押下、トグル）
- **200-300ms**: 状態変化（ホバー、メニューオープン）
- **300-500ms**: レイアウト変更（アコーディオン、モーダル）
- **500-800ms**: エントランスアニメーション（ページ読み込み）

**イージングカーブ（CSSデフォルトではなくこれらを使用）：**
```css
/* 推奨 - 自然な減速 */
--ease-out-quart: cubic-bezier(0.25, 1, 0.5, 1);    /* スムーズで洗練 */
--ease-out-quint: cubic-bezier(0.22, 1, 0.36, 1);   /* やや素早い */
--ease-out-expo: cubic-bezier(0.16, 1, 0.3, 1);     /* 自信のある、決定的な */

/* 避けるべき - 時代遅れでダサい */
/* bounce: cubic-bezier(0.34, 1.56, 0.64, 1); */
/* elastic: cubic-bezier(0.68, -0.6, 0.32, 1.6); */
```

**退出アニメーションは登場より速く。** 登場時間の約75%を使用。

### CSSアニメーション
```css
/* シンプルで宣言的なアニメーションに最適 */
- 状態変化にはtransition
- 複雑なシーケンスには@keyframes
- transformとopacityのみ（GPU加速対応）
```

### JavaScriptアニメーション
```javascript
/* 複雑でインタラクティブなアニメーションに使用 */
- プログラム制御にはWeb Animations API
- ReactにはFramer Motion
- 複雑なシーケンスにはGSAP
```

### パフォーマンス
- **GPU加速**: `transform`と`opacity`を使用、レイアウトプロパティを避ける
- **will-change**: 既知の高コストアニメーションに控えめに追加
- **再描画の削減**: リペイントを最小化、適切な場所で`contain`を使用
- **FPSの監視**: ターゲットデバイスで60fpsを確保

### アクセシビリティ
```css
@media (prefers-reduced-motion: reduce) {
  * {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

**絶対にしてはいけないこと**：
- bounceやelasticイージングカーブを使用する—時代遅れでアニメーション自体に注意を引く
- レイアウトプロパティ（width、height、top、left）をアニメーション化する—代わりにtransformを使用
- フィードバックに500ms以上の持続時間を使用する—遅延して感じる
- 目的なくアニメーション化する—すべてのアニメーションに理由が必要
- `prefers-reduced-motion`を無視する—アクセシビリティ違反
- すべてをアニメーション化する—アニメーション疲れはインターフェースを疲れさせる
- 意図的でない限りアニメーション中のインタラクションをブロックする

## 品質の検証

アニメーションを徹底的にテストする：

- **60fpsでスムーズ**: ターゲットデバイスでカクつきなし
- **自然に感じる**: イージングカーブが有機的で、ロボット的でない
- **適切なタイミング**: 速すぎず（唐突）、遅すぎない（遅延的）
- **モーション軽減が機能**: アニメーションが適切に無効化または簡略化される
- **ブロックしない**: アニメーション中/後もユーザーがインタラクションできる
- **価値を追加する**: インターフェースをより明確にまたはより楽しくする

覚えておいてください：モーションは理解を深め、フィードバックを提供するためのものであり、装飾を追加するためだけのものではありません。目的を持ってアニメーション化し、パフォーマンス制約を尊重し、常にアクセシビリティを考慮してください。優れたアニメーションは見えません—すべてが正しく感じられるだけです。
