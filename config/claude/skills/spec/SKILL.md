# Spec

`spec/idea.md` の初期アイデアをもとに、段階的に設計ドキュメントを作成する。
各フェーズは前フェーズの成果に基づいて進む。レビューで問題が見つかれば即座に修正し、クリーンになってから次へ進む。

## 設計思想

### 実装可能な設計を出す

このスキルのゴールは「設計フェーズを完了すること」ではなく、「実装者が設計書だけを読んで、質問なしにコードを書ける状態を作ること」。エンティティに振る舞いが足りない、ポートのメソッドが未定義、ユースケースの異常系が未検討 — こうした穴が実装時に TODO になる。レビューフェーズはそれを潰すためにある。

### ユーザー体験を起点にする

シナリオ → 画面構成 → ドメイン → ユースケース → DB → テストケースの順で具体化する。体験から先に固めることで、技術的な都合に引きずられた設計を防ぐ。

### 決定を記録する

設計中に下した非自明な判断は ADR として記録する。「なぜそうしたか」が残っていれば、実装者が迷わず、将来の変更時にも意図を追える。

## アーキテクチャ制約

設計全体を通じて以下を守る。レビューではこれらへの違反がないかを必ず確認する。

- **ドメインモデルの独立性**: ドメイン層はインフラ層（DB, HTTP, 外部API）に依存しない
- **ビジネスロジックの配置**: ビジネスルールはエンティティまたはドメインサービスに置く。ユースケース層はオーケストレーションのみ
- **集約境界**: 他の集約のエンティティは ID で参照する。直接の参照を持たない
- **ポートによる抽象化**: 外部依存はすべてポート（インターフェース）経由。メソッドシグネチャまで定義する
- **ドメイン間の非循環**: ドメイン間に循環依存を作らない

## ADR（Architecture Decision Records）

設計中に非自明な判断を下したら `spec/adr/NNN-title.md` に記録する。

各 ADR には以下を記載する:
- **ステータス**: 承認済み
- **コンテキスト**: なぜこの決定が必要になったか
- **決定**: 何を決めたか
- **検討した代替案**: 各案と不採用の理由
- **影響**: この決定がもたらす結果・トレードオフ

ADR を書くべき判断の例:
- ドメイン境界の切り方
- 集約の設計（何をひとつの集約にまとめるか）
- 複数のモデリング手法のうちどれを採用したか
- パフォーマンスや一貫性のトレードオフ

## Phase 0: 準備

### idea.md が存在する場合

`spec/idea.md` を読み、設計を進めるうえで不明な点・曖昧な点を洗い出す。
質問がある場合はすべてまとめて一度にユーザーに確認する。回答を得たら `spec/idea.md` に反映し、Phase 1 へ進む。

### idea.md が存在しない場合

ユーザーにヒアリングを行い、以下を明らかにする:
- プロダクトの目的・解決したい課題
- 想定するユーザー
- 主要な機能・やりたいこと
- 技術的な制約や前提（あれば）

回答をもとに `spec/idea.md` を作成し、ユーザーに確認する。OKが出たら Phase 1 へ進む。

### 共通

- `spec/index.md` を設計のインデックスとして使用する（各フェーズの成果物へのリンクと現在の進捗を記載する）
- なければ作成してから開始する
- Phase 0 の確認が完了したら、以降は最後まで自走する（フェーズ間でユーザー承認を挟まない）

## Workflow Overview

```
Phase 0: 準備
  spec/idea.md の確認（なければヒアリングして作成）
  → 不明点をまとめてユーザーに一括確認
  → 確認完了後、以降は自走

Phase 1: シナリオ設計
  spec/idea.md → spec/scenario/index.md → spec/scenario/${category}.md
  → レビュー・修正 → spec/index.md 更新

Phase 2: ページ設計
  シナリオ → spec/pages/index.md（画面一覧と機能）
  → レビュー・修正 → spec/index.md 更新

Phase 3: ドメイン設計
  シナリオ・ページ設計 → spec/domains/index.md → spec/domains/${domain}.md
  → レビュー・修正 → spec/index.md 更新

Phase 4: ユースケース設計
  ドメイン設計 → spec/usecases/${domain}.md
  → レビュー・修正 → spec/index.md 更新

Phase 5: DB設計
  ドメイン設計 → spec/database/index.md
  → レビュー・修正 → spec/index.md 更新

Phase 6: テストケース定義
  ユースケース → spec/testcases/${domain}/${usecase}.md
  → spec/index.md 更新

Phase 7: クロスフェーズ検証
  全成果物を横断的に検証
  → トレーサビリティ・アーキテクチャ適合性・実装可能性
  → spec/index.md 更新

※ ADR は全フェーズを通じて随時作成 → spec/adr/
```

途中のフェーズから再開する場合は、既存の成果物を確認して該当フェーズから開始する。

## Phase 7: クロスフェーズ検証

すべてのフェーズが完了した後、成果物を横断的に検証する。
個々のフェーズレビューでは見落としやすいフェーズ間の不整合を検出する。

### トレーサビリティ検証

以下のチェーンをたどり、抜け漏れを特定する:
- **シナリオ → ユースケース**: すべてのシナリオの操作がユースケースとして実現されているか
- **ユースケース → ドメインモデル**: ユースケースが呼び出すドメインの振る舞いがすべて定義されているか
- **ドメインエンティティ → DBテーブル**: すべてのエンティティ・値オブジェクトの永続化先が定義されているか
- **ポート → ユースケース**: 定義されたポートが少なくとも1つのユースケースから利用されているか

### アーキテクチャ適合性の最終確認

「アーキテクチャ制約」セクションのすべてのルールについて、全成果物を横断的に確認する。

### 実装可能性の最終確認

各ドメインについて「このドメインの実装に必要な情報はすべて揃っているか」を確認する。
不足があれば該当フェーズに戻って補完する。

出力先: `spec/review/cross-phase/${連番}.md`

問題があれば該当フェーズの成果物を修正し、再検証する。クリーンになるまで繰り返す。

完了後、`spec/index.md` を更新する。

## レビューの共通原則

各フェーズのレビューでは以下を守る:

- 対象ディレクトリの `review/` 配下に連番ファイル（`001.md`, `002.md`, ...）で記録する
- 無駄な情報を省き、対応が必要な問題点のみ記載する
- ドキュメントの責任範囲をそのレビューの対象に限定する
- 問題がなくなるまでレビュー → 修正を繰り返す

### 各ラウンドの独立性

レビューの各ラウンドは独立した完全なレビューとして実施する。前回のレビューファイルを読まずに、成果物とチェックリストだけを見て一から評価する。こうすることで、前回のラウンドで見落とした問題を次のラウンドで拾える。前回の指摘への対応確認ではなく、毎回ゼロベースの新鮮な目で見直すことが目的。

### レビューの3軸

すべてのフェーズで以下の3軸を確認する（各ガイドに詳細なチェック項目がある）:

1. **実装可能性**: 実装者がこの設計書だけ見てコードを書けるか。型、シグネチャ、制約が揃っているか
2. **アーキテクチャ適合性**: 「アーキテクチャ制約」セクションのルールに違反していないか
3. **トレーサビリティ**: 前フェーズの成果物と整合しているか。漏れているシナリオ・要件がないか

## 原則

- 各フェーズは前フェーズの成果に基づいて進める — 飛ばさない
- レビューで見つかった問題は即座に修正する — 先送りにしない
- スコープクリープを避け、idea.md の範囲に集中する
- 実装時に TODO を残さない粒度で設計する。型・シグネチャ・制約・エラーケースまで定義する
- 非自明な判断は ADR に記録する
- `spec/index.md` はフェーズ完了ごとに更新し、成果物と進捗を反映する
