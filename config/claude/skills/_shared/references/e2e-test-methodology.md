# E2Eテスト設計方法論

テスト対象の要件・実装コード・テスト計画を総合的に分析し、仕様を網羅するテストを設計・実装するための共通方法論。

issue-implement（Step 3: E2Eテスト作成）や implement-test（Phase 2: テスト実装）など、テスト作成を行うスキルから参照される。

## テスト情報源の分析

テストを設計する前に、利用可能な情報源をできる限り多く組み合わせて分析する:

1. **要件定義** — テストすべき仕様のスコープを決定する
   - issue-implement の場合: plan.md の要件
   - spec ベースの場合: spec/testcases/ + spec/usecases/ の定義
2. **実装コード** — 変更・追加されたコードを Read で確認し、実際の振る舞い・分岐・エッジケースを把握する
3. **テスト計画・テストケース定義（参考）** — 手動確認向けの観点やテストケース仕様を参考にする。そのまま自動化するのではなく、実装後の視点で改めてテストすべき観点を設計する
4. **プロジェクトの既存テスト構成** — テストフレームワーク、ディレクトリ構成、設定ファイル、既存テストを Glob/Grep/Read で確認する。CLAUDE.md や README.md にテスト関連の記述があればそれも参照する

## テストケース設計の5つのカテゴリ

以下のカテゴリを網羅するようにテストケースを設計する:

### 1. 正常系
- 要件が満たされていることの検証
- 主要なユースケースフローの確認

### 2. 境界値・エッジケース
- 入力の境界値（最小値、最大値、ちょうどの値）
- 空値・null・未定義
- 上限・下限の処理

### 3. 異常系
- エラー時の振る舞い
- バリデーション（不正入力の拒否）
- 例外処理の確認

### 4. 状態遷移
- 前提条件が異なる場合の挙動
- 条件分岐パスの網羅
- 操作順序の影響

### 5. 既存機能への影響
- 変更が既存の動作を壊していないこと
- 関連する既存テストとの整合性

## テスト実装の原則

- **既存のテストフレームワーク・パターンに合わせる** — 新しいフレームワークを導入しない
- **仕様に合わせたテストを作成する** — 実装の内部構造ではなく、仕様上の振る舞いをテストする
- **テストは安定して再現可能であること** — フレイキーなテストは書かない
- **テストファイル名・配置は既存の規約に従う**
- **UIの目視確認など自動化が困難な項目は無理に書かない** — スキップした項目とその理由を報告する

## テスト結果の確認

1. テストがすべて PASS していることを確認する
2. 自動化を見送った項目がある場合、妥当性を確認する
3. テストが失敗している場合は修正する（最大3回のリトライ）

## E2Eテスト環境が存在しない場合

プロジェクトにE2Eテストの既存環境が存在しない場合は、テスト作成をスキップし、その旨を記録する（PR の Test plan やサマリーに記載）。
