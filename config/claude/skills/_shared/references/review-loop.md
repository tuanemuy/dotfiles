# レビュー・修正ループ共通ガイド

レビューサイクルの共通構造を定義する。各スキルのレビューガイドはこの共通構造に従いつつ、レイヤー固有のレビュー観点を追加する。

issue-implement のレビューガイドや implement-* サブスキルのレビューガイドから参照される。

## 出力フォーマット

レビューの指摘は以下のカテゴリに分類する:

### Blockers（マージ不可の問題）

```
- **[B-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ問題か
  - 提案: どう直すべきか
```

### Warnings（改善推奨）

```
- **[W-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ気になるか
  - 提案: 改善案
```

### Notes（参考情報）

```
- **[N-001]** 良い点や参考情報
```

Blocker がなければ「なし」と書く。

### 共通の Blocker 基準

以下は常に Blocker として報告する:
- 設計からの逸脱（spec/ や plan.md との不整合）
- TODO・FIXME・仮実装が残っている
- テストケース定義のカバレッジ不足（テストレビューの場合）

## レビューファイルのテンプレート

```markdown
# {Layer} Review #{連番}

**Date:** {日付}
**Round:** {N}回目

---

## Summary

- Blockers: {数}
- Warnings: {数}
- Verdict: **BLOCKED** / **APPROVED**

---

{レビュー結果}
```

## 修正ループ

Blocker・Warning がある場合:

1. **Blocker から優先的に修正する**
2. **Warning もすべて修正する** — Warning も放置せず潰す
3. 修正はサブエージェントに委譲する（指摘内容と対象ファイルを渡す）
4. 修正後、品質ゲートを実行する（typecheck + lint、テストレビューの場合は + test run）
5. 再レビュー（次の連番ファイルを作る）

### 後回しにする場合（最終手段）

どうしてもスコープ外でしか対応できない場合のみ:
- ADR（`.adr/` または `.issue/{Issue番号}/adr.md`）に判断理由を記録する
- `gh issue create` で別 Issue を起票する
- レビューファイルの該当指摘に `→ 別Issue #{新Issue番号} で対応` と追記する

## 完了条件

**2回連続で Blocker 0件 かつ Warning 0件** になったらレビュー完了。

最大ラウンド数に達しても収束しない場合は、残っている指摘をまとめてユーザーに報告し、判断を委ねる。

## サブエージェントの運用原則

- サブエージェントは**並列で**起動する（複数レイヤーを同時レビューする場合）
- レビューは**厳しく**行う。「まあいいか」は禁止。ただし的外れな指摘はしない
- 修正するのは確信を持てるものだけ。判断に迷うものはユーザーに聞く
- **各ラウンドのレビューは独立して行う** — 前回の指摘を前提にせず、ゼロからレビューする
- レビューファイルは追記ではなく、毎ラウンド新規作成する（履歴が残る）
