# 照合ガイド — レイヤー×ドメインの比較手順

各レイヤーの spec ファイルと実装コードを照合するための手順とサブエージェントテンプレート。

照合の目的は「spec が実装の現状を正しく反映しているか」の検証であり、「実装が spec 通りか」の検証ではない。つまり、乖離が見つかったとき「spec を直すべきか、コードを直すべきか」を判断する材料を集める作業。

---

## 照合の粒度

各レイヤーで以下の粒度で照合する:

| レイヤー | 照合単位 | spec ファイル | 照合対象 |
|---------|---------|-------------|---------|
| ドメイン | ドメインごと | spec/domains/{name}.md | エンティティ、値オブジェクト、ドメインサービス、ポートインターフェース |
| アダプター | テーブル/リポジトリごと | spec/database/*.md | テーブル定義、カラム、リレーション、リポジトリ実装 |
| ユースケース | ユースケースごと | spec/usecases/{name}.md | 関数シグネチャ、入出力型、処理フロー |
| フロントエンド | ページごと | spec/pages/{name}.md | ページ構成、コンポーネント、props、API接続 |
| テスト | テストスコープごと | spec/testcases/{name}.md | テストケースのカバレッジ |

## サブエージェントテンプレート

### ドメイン層の照合

ドメインごとに1つのサブエージェントを起動する。

```
あなたは設計ドキュメントと実装コードの乖離を検出するエンジニアです。

## 前提
- まずプロジェクトルートの CLAUDE.md を読み、ディレクトリ構成と命名規約を把握する

## やること
1. spec/domains/{ドメイン名}.md を読む
2. 対応する実装コードを Glob/Grep で探し、読む
3. 以下の観点で照合する

## 照合観点
- **エンティティ**: spec に定義されたエンティティが実装に存在するか。プロパティ（名前・型）が一致するか。メソッドが一致するか
- **値オブジェクト**: spec に定義された VO が実装に存在するか。バリデーションルールが一致するか
- **ドメインサービス**: spec に定義されたサービスが実装に存在するか。メソッドのシグネチャが一致するか
- **ポートインターフェース**: spec に定義されたポートが実装に存在するか。メソッドのシグネチャが一致するか
- **実装にあるが spec にないもの**: 実装で追加されたプロパティ・メソッド・クラスがないか

## 乖離の報告形式
各乖離について以下を報告する:

- **場所**: spec のセクション ↔ 実装のファイル:行番号
- **内容**: 何が異なるか（具体的に）
- **推定分類**: 設計判断 / 実装誤り / 実装漏れ / 不明
- **推定根拠**: なぜその分類と判断したか（git log のコミットメッセージ、コードコメント等の手がかり）

乖離がない場合は「一致」とだけ報告する。
```

## 推定分類の判断基準

サブエージェントが乖離の分類を推定するための手がかり:

### 設計判断の手がかり
- git log で意図的なコミットメッセージがある（「〜に変更」「〜を追加」等）
- コードコメントに理由が書かれている
- ADR が存在する
- Issue や PR で議論された形跡がある
- 実装がより合理的・効率的な構造になっている

### 実装誤りの手がかり
- 型の不一致（spec: string → 実装: number 等）
- メソッド名のタイポやケース違い
- 引数の順序が異なる
- spec の制約（必須/任意、バリデーション等）が守られていない

### 実装漏れの手がかり
- spec にある要素が実装に見つからない
- TODO コメントで「後で実装」と書かれている
- 空の関数ボディや未実装のメソッドがある

### 不明
- 上記のどれにも当てはまらない、または複数の可能性がある場合
- ユーザーに確認が必要
