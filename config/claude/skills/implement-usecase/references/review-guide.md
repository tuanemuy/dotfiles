# ユースケース層レビューガイド

ユースケース層の実装が完了した後、設計準拠やドメインロジックの漏出をレビューする。

## レビュー観点

- spec/usecases/ の設計通りに実装されているか
- ドメインロジックがアプリケーションサービスに漏れ出していないか
- バリデーション・存在チェックが適切か
- 設計で定義された全ユースケースが漏れなく実装されているか
- 実装と設計の間に乖離がある場合、spec の更新が必要か

## サブエージェントへの指示テンプレート

```
あなたは「ユースケース層」の観点でコードをレビューする専門家です。CLAUDE.md / README.md から読み取れるコンテキストやアーキテクチャに従います。

## レビュー対象
- ユースケース設計: spec/usecases/
- ドメイン設計: spec/domains/
- 実装コード: {ユースケース層ディレクトリ}

## やること
1. 設計ドキュメントを読む
2. 実装コードを読む
3. 設計と実装の整合性、コード品質を厳しくレビューする

## レビュー観点
- 設計通りに実装されているか
- ドメインロジックがアプリケーションサービスに漏れ出していないか
- バリデーション・存在チェックが適切か
- TODO・FIXME・仮実装が残っていないか

## 重要: 独立したレビュー
このレビューは前回のラウンドとは独立している。設計ドキュメントと実装コードだけを見て、ゼロからレビューする。

## 出力フォーマット

#### Blockers（マージ不可の問題）
- **[B-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ問題か
  - 提案: どう直すべきか

#### Warnings（改善推奨）
- **[W-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ気になるか
  - 提案: 改善案

Blocker がなければ「なし」と書く。
設計からの逸脱は Blocker として報告する。
TODO・FIXME・仮実装が残っている場合も Blocker として報告する。
```

## レビューファイルの作成

結果を `.review/usecase/review-{連番}.md` にまとめる。

```markdown
# Usecase Review #{連番}

**Date:** {日付}
**Round:** {N}回目

---

## Summary

- Blockers: {数}
- Warnings: {数}
- Verdict: **BLOCKED** / **APPROVED**

---

{レビュー結果}
```

## 修正ループ

Blocker・Warning がある場合:

1. Blocker から優先的に修正する
2. Warning もできる限り修正する
3. 修正はサブエージェントに委譲する（指摘内容と対象ファイルを渡す）
4. 修正後、品質ゲート（typecheck + lint）を実行する
5. 再レビュー（次の連番ファイルを作る）

## 完了条件

**2回連続で Blocker 0件 かつ Warning 0件** になったらレビュー完了。最大5ラウンド。

5ラウンドで収束しない場合は残っている指摘をまとめてユーザーに報告し、判断を委ねる。
