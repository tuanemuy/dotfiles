# フロントエンド層レビューガイド

フロントエンドの実装が完了した後、設計・デザイン準拠とコンポーネント品質をレビューする。

## レビュー観点

- spec/pages/ の設計通りに実装されているか
- spec/design/ のHTMLデザイン・デザイントークンに忠実か（存在する場合）
- 共通コンポーネントが適切に定義・再利用されているか
- ドメイン固有のコンポーネントやロジックが適切に切り出されているか
- UXがシナリオ通りに動作するか
- 実装と設計の間に乖離がある場合、spec の更新が必要か

## サブエージェントへの指示テンプレート

```
あなたは「フロントエンド」の観点でコードをレビューする専門家です。CLAUDE.md / README.md から読み取れるコンテキストやアーキテクチャに従います。

## レビュー対象
- ページ設計: spec/pages/
- UIデザイン（存在する場合）: spec/design/
- 利用シナリオ: spec/scenario/
- 実装コード: {フロントエンドディレクトリ}

## やること
1. 設計ドキュメントを読む
2. デザイン成果物を読む（存在する場合）
3. 実装コードを読む
4. 設計・デザインと実装の整合性、コード品質を厳しくレビューする

## レビュー観点
- 設計通りに実装されているか
- デザインに忠実か（HTMLデザイン・デザイントークンとの乖離がないか）
- 共通コンポーネントが適切に定義・再利用されているか
- ドメイン固有のコンポーネントやロジックが適切に切り出されているか
- TODO・FIXME・仮実装が残っていないか

## 重要: 独立したレビュー
このレビューは前回のラウンドとは独立している。設計ドキュメントと実装コードだけを見て、ゼロからレビューする。

## 出力フォーマット

#### Blockers（マージ不可の問題）
- **[B-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ問題か
  - 提案: どう直すべきか

#### Warnings（改善推奨）
- **[W-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ気になるか
  - 提案: 改善案

Blocker がなければ「なし」と書く。
設計からの逸脱は Blocker として報告する。
TODO・FIXME・仮実装が残っている場合も Blocker として報告する。
```

## レビューファイルの作成

結果を `.review/frontend/review-{連番}.md` にまとめる。

```markdown
# Frontend Review #{連番}

**Date:** {日付}
**Round:** {N}回目

---

## Summary

- Blockers: {数}
- Warnings: {数}
- Verdict: **BLOCKED** / **APPROVED**

---

{レビュー結果}
```

## 修正ループ

Blocker・Warning がある場合:

1. Blocker から優先的に修正する
2. Warning もできる限り修正する
3. 修正はサブエージェントに委譲する（指摘内容と対象ファイルを渡す）
4. 修正後、品質ゲート（typecheck + lint）を実行する
5. 再レビュー（次の連番ファイルを作る）

## 完了条件

**2回連続で Blocker 0件 かつ Warning 0件** になったらレビュー完了。最大5ラウンド。

5ラウンドで収束しない場合は残っている指摘をまとめてユーザーに報告し、判断を委ねる。
