# ユースケース実装ガイド

`spec/usecases/` の設計に基づいて、アプリケーションサービス（ユースケース）を実装する。

## サブエージェントへの指示テンプレート

ドメインごとにサブエージェントを起動する。独立したドメインは並列で起動してよい。

```markdown
あなたは設計ドキュメントに基づいてユースケースを実装するエンジニアです。

## 前提
- まずプロジェクトルートの CLAUDE.md を読み、使用言語・ディレクトリ構成・規約・コマンドを把握する

## 対象
- ドメイン: {domain名}
- ユースケース設計: spec/usecases/{domain}.md
- ドメインモデル設計: spec/domains/{domain}.md

## やること
1. CLAUDE.md を読み、プロジェクトの構成・規約を把握する
2. spec/usecases/{domain}.md を読み、ユースケースの設計を理解する
3. spec/domains/{domain}.md でドメインモデルとの関係を確認する
4. **設計で定義された全ユースケースを漏れなく洗い出す** — 各ユースケース（関数・メソッド）と、その入出力・バリデーション・存在チェック・エラー分岐を一覧にする。これが実装すべき全項目であり、後でカバレッジ表になる
5. 実装例があれば参照する（{実装例パス}）
6. 既存の実装があれば Read で読んで現状を把握する
7. 手順4で洗い出した全ユースケースを実装する。正常系だけ実装してエラー分岐・バリデーションを省略しない
8. CLAUDE.md に記載の typecheck・lint コマンドを実行し、エラーがなくなるまで修正する

## 重要な原則
- ドメインロジックをアプリケーションサービス内に実装しない
- 入力には適切なバリデーションや存在チェックを行う — 「後で」にしない
- 既存コードのスタイル・規約に合わせる
- ディレクトリ構成は CLAUDE.md とプロジェクトの既存構造に従う
- 設計に問題がある場合は実装せずにその旨を報告する

## 省略を残さない

スタブ（`return []`/固定値返し）・ダミーデータ・`not implemented` 系シグナル・空ボディやエラー握り潰し・正常系だけ実装してバリデーション/存在チェック/エラー分岐を省略・一部のユースケースだけ実装・「// 省略」等のコメントによる省略は、typecheck が通っても「未完成」。設計で要求された振る舞いを本実装する。実装できないユースケースや分岐は黙って省略せず、カバレッジ表に「未実装」と理由付きで明記する（メインエージェントが判断）。

## 出力
- 変更したファイルの一覧と変更内容の要約
- **カバレッジ表（必須）** — 設計の全ユースケースを1行ずつ列挙し、実装箇所と状態を示す。未実装・部分実装も必ず行に残す:

  | ユースケース | 実装ファイル:行 | 状態 |
  |-------------|----------------|------|
  | {ユースケース名（バリデーション/エラー分岐含む）} | path:line | 完全 / 部分 / 未実装 |

  「部分」「未実装」がある行には理由を添える（例: エラー分岐 X が未実装）。
- typecheck・lint の実行結果
- 設計との乖離（あれば: 乖離の内容・理由・実装での対応方針）
- 判断に迷った点や懸念事項（あれば）
```

## 完了確認

サブエージェント完了後、メインエージェントが typecheck + lint を実行して確認する。エラーがあればサブエージェントを再起動して修正させる。
