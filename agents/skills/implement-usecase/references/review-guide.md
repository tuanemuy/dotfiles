# ユースケース層レビューガイド

ユースケース層の実装が完了した後、設計準拠やドメインロジックの漏出をレビューする。レビューサイクルの共通構造（出力フォーマット・修正ループ・完了条件・レビュー記録テンプレート）は `../../_shared/references/review-loop.md` に従う。本ガイドではユースケース層固有の観点とサブエージェントへの指示だけを定義する。

## サブエージェントへの指示テンプレート

```markdown
あなたは「ユースケース層」の観点でコードをレビューする専門家です。CLAUDE.md / README.md から読み取れるコンテキストやアーキテクチャに従います。

## レビュー対象
- ユースケース設計: spec/usecases/
- ドメイン設計: spec/domains/
- 実装コード: {ユースケース層ディレクトリ}

## やること
1. 設計ドキュメントと実装コードを読む
2. 以下の観点で整合性とコード品質を厳しくレビューする

## レビュー観点
- 設計通りに実装されているか
- ドメインロジックがアプリケーションサービスに漏れ出していないか
- バリデーション・存在チェックが適切か
- 設計で定義された全ユースケースが漏れなく実装されているか
- TODO・FIXME・仮実装が残っていないか

## 重要: 独立したレビュー
前回のラウンドとは独立している。設計ドキュメントと実装コードだけを見て、ゼロからレビューする。

## 出力フォーマット
#### Blockers（マージ不可の問題）
- **[B-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ問題か
  - 提案: どう直すべきか

#### Warnings（改善推奨）
- **[W-001]** 問題の説明（場所・理由・改善案）

Blocker がなければ「なし」と書く。設計からの逸脱、TODO・FIXME・仮実装の残存は Blocker とする。
```

## 修正ループと完了条件

`../../_shared/references/review-loop.md` に従う。レビュー記録は `.review/usecase/review-{連番}.md`。最大5ラウンド。
