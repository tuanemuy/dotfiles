コードのコメントから寿命の短い情報を取り除き、長く価値を保つコメントだけを残す。

## 前提となる考え方

コードは「何を」しているかを語る。だからコメントが「何を」を繰り返すと、コードと二重管理になり、片方だけ古くなって嘘をつき始める。残す価値があるのは、コードを読んでも分からない情報だけ。

残すコメントは次の2種類に集約される：

1. **ドキュメンテーションコメント** — 関数・型・クラス・モジュール・公開APIの「契約」を書く。JSDoc / TSDoc / docstring / rustdoc / GoDoc など言語ごとの形式は違っても、役割は同じ：**利用者が実装を読まずに使うための普遍的な情報**。引数・戻り値・例外・副作用・前提条件・使用例など、時間が経っても変わらない仕様を書く場所。

2. **why / why not コメント** — それ以外のすべてのコメント。コードだけでは伝わらない**判断の理由**を書く。「なぜこの実装にしたか」「なぜ自明に見える別の選択肢を避けたか」。読み手が「これ、こうすればよくない？」と思いそうな箇所の先回りの答えになっているのが理想。

この2つに当てはまらないコメントは、原則として削除する。

## 削除対象

以下は寿命が短い、あるいはコードと重複していてノイズになる。見つけたら消す：

- **修正の経緯**: 「○○のバグを修正」「△△から変更」「以前は〜だった」「リファクタリングで追加」など、変更の歴史。これは git が記録している。
- **メタデータ**: 作者名、日付、変更ログ、レビューのやりとりの残骸、`@author` / `@since` のような出自情報。
- **チケット・課題への参照**: `// JIRA-123`、`SPEC-42` のような ID 参照はメタデータなので消す（例: `// 並び順は表示仕様に合わせる（SPEC-42）` → `// 並び順は表示仕様に合わせる`）。
- **自明な言い換え**: コードをそのまま日本語/英語に直しただけ。`// ユーザーを取得` の下に `getUser()`、`// i をインクリメント` の上に `i++` など。
- **コメントアウトされた古いコード**: 「念のため残した」死んだコード。復活させたいなら git から戻せる。
- **冗長な生成コメント**: 1行ずつ動作を説明する過剰な注釈や、内容の薄い定型コメント。

## 残す・書き換える判断

迷ったら「このコメントは半年後も正しいか？コードを読めば分かることか？」で考える。

- ドキュメンテーションコメントは**残す**。ただし上記の削除対象（経緯・メタデータ・自明な羅列）が混ざっていれば、その部分だけ削いで契約情報に絞る。
- why / why not を含むコメントは**残す**。説明が冗長なら要点に圧縮する。
- 「何を」しか言っていないコメントは、その下のコードが自己説明的なら**消す**。コードが複雑で読み解きにくいなら、コメントを足すのではなく**コードを分かりやすくする方を優先**し、それでも残る非自明さを why として書く。
- **TODO / FIXME / HACK / XXX などのマーカーは、理由・文脈を伴うなら残す**。これらは「ここに既知の問題があり、まだ直っていない」という、コードからは決して読み取れない未完了の状態を表す。`FIXME` という語そのものや、問題の内容（O(n²) で遅い等）・保留理由（仕様確定待ち等）はメタデータではなく価値ある情報なので、マーカーごと保持する。消すのは中身のない裸のマーカー（`// TODO: 後で直す`）だけ。これは経緯コメントとは別物で、まだ閉じていない約束だからこそ残す。
- 判断に確信が持てないコメント（暗黙の前提や落とし穴を示唆していそうなもの）は、**消さずに残す**。情報の喪失は誤った削除の方が痛い。

## 進め方

1. **対象を決める**: `target` 引数があればそのファイル/ディレクトリ。なければ `git diff`（および `git diff --staged`）で変更された範囲のコメントを対象にする。
2. **コメントを1つずつ分類する**: 各コメントを「ドキュメンテーション」「why/why not」「削除対象」のどれかに振り分ける。
3. **直接編集する**: 削除対象を消し、残すコメントのうち冗長なものは圧縮する。コードの挙動は一切変えない。コメント（と必要ならコメントに付随する空行）だけを編集する。
4. **報告する**: 何を消し、何を残し、何を書き換えたかを簡潔に伝える。大量に消した場合は件数とカテゴリ（経緯/メタデータ/自明/死にコード）でまとめる。

## 例

**修正の経緯 → 削除**
```
// 2024-03 にタイムアウトを5sから30sに変更（API遅延対応, by tanaka）
const TIMEOUT = 30_000;
```
↓
```
// 外部APIのp99が約20sあるため余裕を持たせる
const TIMEOUT = 30_000;
```
経緯・日付・作者は git に任せ、残った「なぜ30sなのか」という why だけ残す。

**自明な言い換え → 削除**
```
// ユーザーIDで検索する
const user = await repo.findById(id);
```
↓
```
const user = await repo.findById(id);
```
コードが既に同じことを言っている。

**ドキュメンテーションコメント → メタデータを削いで契約に絞る**
```
/**
 * 金額をフォーマットする。
 * 2024/01 作成。最初は円のみ対応、後でドル追加。
 * @param amount 金額
 * @param currency 通貨コード（ISO 4217）
 * @returns ロケールに応じた表示文字列
 */
```
↓
```
/**
 * 金額をロケールに応じた表示文字列にフォーマットする。
 * @param amount 金額（最小単位ではなく主単位）
 * @param currency 通貨コード（ISO 4217）
 * @returns 表示用にフォーマットされた文字列
 */
```
作成日や経緯を削り、普遍的な契約（引数の意味・単位・戻り値）に集中させる。

**why not → 残す**
```
// Set ではなく配列で持つ。要素数は常に数件で、順序を保持する必要があるため。
const handlers = [];
```
読み手が抱く「Set でよくない？」への先回りの答え。こういうコメントこそ価値がある。

**裸の TODO → 削除 / 理由付き FIXME → マーカーごと残す**
```
// TODO: 後で直す
const result = build(rows);

// FIXME: 入力が10万件を超えると O(n²) で遅い。仕様確定まで保留。
const sorted = rows.sort(cmp);
```
↓
```
const result = build(rows);

// FIXME: 入力が10万件を超えると O(n²) で遅い。仕様確定まで保留。
const sorted = rows.sort(cmp);
```
中身のない `TODO: 後で直す` は消す。一方 `FIXME` は「既知の未解決問題」という状態を表すので、マーカーごと残す。
