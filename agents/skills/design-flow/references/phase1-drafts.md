# Phase 1: ドラフト提案

## 目的

代表画面を、「UI構造 × デザイン様式」の異なる複数のパターンでHTMLデザインし、ユーザーに具体的な選択肢を提示する。
言葉で構造や様式を議論するより、実物を見て選ぶほうが圧倒的に精度が高い。

各案が問うのは「この画面をどういうUI構造で成立させ、どの様式の視覚言語で表現するか」。仕上げの水準は SKILL.md のデザイン原則に従い、全案とも洗練されたプロダクトUIとして通用するレベルに揃える。

## Step 1: パターンの設計

`spec/idea.md` の目的・ターゲットと `spec/scenario/` の体験を踏まえ、「UI構造 × デザイン様式」の組として3〜5案考える。それぞれが「この組で全画面を作っても十分いける」と思える完成度と、明確な設計判断を持った提案にする。

### UI構造の軸

- **レイアウト構造**: サイドバー＋メインペイン / トップナビ＋単一カラム / マルチペイン / 中央寄せ集中型 など
- **ナビゲーションモデル**: 常設ナビ / タブ切替 / 階層ドリルダウン / 検索・コマンド起点 など
- **情報密度**: 一覧性を優先する高密度か、集中を優先する低密度か
- **情報階層**: どの情報・アクションを最上位に置き、何を段階的に開示するか
- **コンポーネント選択**: テーブル / カードグリッド / リスト / タイムライン など、主要コンテンツの表現形式

### デザイン様式の軸

視覚言語は、歴史的なデザイン様式または現行のデザイントレンドから選ぶ（以後まとめて「様式」と呼ぶ）。様式として認めるのは以下を満たすもの:

- 固有の名前を持ち、系譜（歴史的様式）または現在の実践の蓄積（トレンド）によって検証されている
- タイポグラフィ・グリッド・カラー・形状にわたる原則の体系を持つ（単一の視覚的特徴の言い換えではない）
- プロダクトUIに適用して可読性・操作性を保てる

選定はプロダクト起点で行う:

1. `spec/idea.md` のプロダクトの性格・ターゲットから「このプロダクトはどう見えるべきか」をまず言語化する
2. それに応えうる様式の候補を、グラフィックデザイン・エディトリアル・デジタルプロダクトなど系譜を問わず、デザイン史と現在のプロダクトデザインの潮流の両方から広く挙げる。有名な様式や流行のトレンドを反射的に選ばない。トレンドは知識が古くなりやすいため、候補に挙げる前にWeb検索で現在の潮流を確認するとよい
3. このプロダクトへの必然性で絞り込む

歴史的様式は経年耐性に、トレンドは鮮度に強みがある。経年耐性型とトレンド型を混ぜて提示すると、時間軸の選択もユーザーに委ねられる。

重要なのは、様式を雰囲気の引用ではなく原則の体系として忠実に実装し、プロダクトUIとして通用する水準に仕上げること。様式に属さない思いつきの味付けで案を作らない。

### 組の作り方

- 構造と様式は独立に選べるが、案としては必然性のある組にする（構造が要求する情報密度・階層表現と、様式が得意とする表現が噛み合っていること）
- すべての軸で違いを作る必要はない。ユーザーの意思決定にとって意味のある違い（一覧性 vs 集中、禁欲 vs 明快な色彩、学習コスト vs 操作効率 など）がはっきり比較できる案を選ぶ。似た案を数合わせで足さない
- 各案には、選んだ様式と構造の性格が伝わる簡潔な名前をつける

ユーザーから参考サイトやイメージの指定があれば、それを出発点にする。

## Step 2: HTMLドラフトの作成

Phase 0 で選定した代表画面それぞれについて、各パターンでHTMLファイルを作成する。
Skill ツールで `/frontend-design` を呼び出す。

出力先: `spec/design/drafts/draft-{番号}-{パターン名}-{画面名}.html`

パターン名は Step 1 でつけた案の名前をケバブケースにしたもの。同じ案は画面が違っても同じ番号・同じパターン名を使う:
- `spec/design/drafts/draft-1-{パターン名}-dashboard.html`
- `spec/design/drafts/draft-1-{パターン名}-detail.html`
- `spec/design/drafts/draft-2-{パターン名}-dashboard.html`

代表画面が1つの場合は画面名を省略してもよい:
- `spec/design/drafts/draft-1-{パターン名}.html`

`/frontend-design` に渡すコンテキスト:
- **デザイン対象**: 代表画面の名前と、`spec/pages/index.md` に記載されたその画面の目的・機能
- **パターン**: Step 1 で設計したパターンの名前・UI構造・デザイン様式とその体系上の原則・設計判断（なぜこの組がこのプロダクトに効くか）
- **ユーザーシナリオ**: `spec/scenario/` からこの画面に関連するシナリオの要約（ユーザーがこの画面で何をしようとしているか）
- **デザイン原則**: SKILL.md のデザイン原則（全文をそのまま伝える）
- **出力先パス**: 上記の出力先
- **作成条件**: 下記の条件をすべて伝える

作成条件:
- 1つのHTMLファイルに CSS をインラインで含め、単体でブラウザ表示できるようにする
- ダミーデータを使ってリアルな見た目にする
- **レスポンシブ対応必須**: モバイルファーストで設計し、標準ブレークポイント（`sm: 640px` / `md: 768px` / `lg: 1024px` / `xl: 1280px` / `2xl: 1536px`）を使用する。最低限モバイル（<640px）・タブレット（≥768px）・デスクトップ（≥1024px）の3レンジで破綻なく表示されること。Flexbox/Grid と `clamp()`、流動的な単位（%, vw, fr）、`min()`/`max()` を活用し、固定幅 px のレイアウトは避ける
- 案ごとの構造と様式の違いが一目でわかるようにする。視覚表現はすべて指定した様式の体系から導き、体系に根拠のない装飾を足さない
- `<meta name="viewport" content="width=device-width, initial-scale=1">` を必ず含める

## Step 3: ユーザーへの提示

**このフェーズではスクリーンショットを撮らない。** パターンを判断するのはユーザーなので、各ドラフトのHTMLをブラウザで直接開いて見比べてもらうのが手早く正確（必要ならその場で幅を変えてレスポンシブ挙動も確認できる）。

各ドラフトのパスを提示し、各案について以下を簡潔に説明する:
- パターンの名前と、UI構造・デザイン様式の組
- 様式の系譜（または潮流としての背景）と視覚的特徴
- 時間軸の性格: 経年耐性を狙う様式か、鮮度を狙うトレンドか。トレンド型は陳腐化のリスクも添える
- 構造上のトレードオフ（何を優先し、何を割り切っているか。どんな使い方・ユーザーに向くか）

ユーザーの手元のブラウザで開いて確認してもらう（macOS なら下記で既定ブラウザに一括で開ける）:

```bash
open spec/design/drafts/*.html
```

ユーザーに選択を求める。「A案の構造で、B案の様式を使って」のような組み合わせの要望も受け付ける。

**ユーザーの選択を得るまで次のフェーズに進まない。**
