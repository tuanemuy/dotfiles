# Phase 0: 準備

## 自走の原則

Phase 0 ではユーザーへの質問を最小限にする。spec スキルの成果物が揃っていれば、画面選定まで自走して Phase 1 のドラフト作成に直接進む。質問が必要なのは情報が本当に不足しているときだけ。

## Step 1: 再開判定

既存のデザイン成果物を確認し、最も進んだフェーズの次から開始する:

| 成果物 | 判定 |
|--------|------|
| `spec/design/review/*.md` が存在 | Phase 4 進行中 → Phase 4 から再開 |
| `spec/design/pages/*.html` が存在 | Phase 3 完了 → Phase 4 から開始 |
| `spec/design/tokens.md` と `spec/design/index.md` が存在 | Phase 2 完了 → Phase 3 から開始 |
| `spec/design/drafts/*.html` が存在 | Phase 1 完了 → ユーザーに方向性の選択状況を確認 |
| 上記いずれもなし | Phase 0 から開始 |

再開する場合は、既存の成果物の内容を確認してからユーザーに現在の状況を伝え、続行してよいか確認する。

## Step 2: spec 成果物の読み込み

以下の spec スキル成果物を読み込む:

- `spec/index.md` — 設計全体のインデックス（どこまで設計が進んでいるか確認）
- `spec/idea.md` — プロダクトの目的・ターゲットユーザー
- `spec/scenario/index.md` → 各 `spec/scenario/${category}.md` — ユーザーシナリオ
- `spec/pages/index.md` — 画面一覧（画面名・目的・機能）

これ以外にもユーザーから提供されたドキュメント、コメント、参考URLなどがあればすべて確認する。

## Step 3: 情報の十分性を判断

デザインに着手するには最低限以下が必要:

- プロダクトの目的とターゲットユーザーが明確（`spec/idea.md` から）
- デザイン対象の画面が特定できる（`spec/pages/index.md` から）
- 各画面に表示すべき情報やアクションの概要がわかる
- ユーザーシナリオから体験の流れが理解できる

`spec/pages/index.md` が存在しない場合は、spec スキルでページ設計を先に行うことを提案する。
それ以外の情報が不足している場合はユーザーにヒアリングする。質問はまとめて一度に行う。

情報が十分であれば、質問せずに Step 4 に進む。

## Step 4: 画面一覧の整理と代表画面の選定

`spec/pages/index.md` の画面一覧をもとに、デザイン対象を整理する。
Phase 1 のドラフト提案に使う代表画面を選定する。

### 選定数の判断

- 画面の種類が少ない（5画面以下）か、性格が似た画面ばかりの場合 → **1画面** で十分
- 画面の種類が多い、または性格が大きく異なる画面群がある場合 → **2-3画面** を選ぶ
  - 例: ダッシュボード + 詳細画面 + フォーム画面のように、レイアウトパターンが異なるもの

### 選定基準

- プロダクトの性格が最も表れる画面（多くの場合メイン画面やダッシュボード）
- 要素のバリエーションが豊富で、デザイン方向性の違いが出やすい画面
- ユーザーが最も長く滞在する画面
- 異なるレイアウトパターンを代表する画面（リスト型、フォーム型、ダッシュボード型など）

選定は自分の判断で行い、そのまま Phase 1 に進む。ユーザーへの確認は不要。
