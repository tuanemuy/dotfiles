# 実装共通原則

implement オーケストレーターと各 implement-* サブスキルが共通で従う原則。各スキルはこれを参照し、レイヤー固有の原則だけを個別に記述する。

- **メインエージェントはオーケストレーションのみ** — コードの読み書きはサブエージェントに委譲する
- **プロジェクトコンテキストに従う** — 言語・ディレクトリ構成・規約は CLAUDE.md / README.md から読み取る。ハードコードされたパスを前提にしない
- **設計ドキュメントに忠実に実装する** — スコープ外の改善を含めない
- **省略を残さない** — 設計で定義された全機能を完全に実装させる。スタブ（`return []`/固定値返し）・ダミーデータ・`not implemented` 系シグナル・空ボディやエラー握り潰し・CRUD/分岐/バリデーションの部分実装・「// 省略」等のコメント・TODO/FIXME は、typecheck が通っても「未完成」。「後で実装」は許可しない。実装が困難なものは黙ってスタブ化せず、カバレッジ表に「未実装」と明記して報告させ、メインエージェントが判断する
- **完全性は構文チェックで担保できない** — typecheck + lint は型・構文しか見ない。完全性は各レイヤーの完全性ゲート（独立したサブエージェントによる spec 照合。`completeness-gate.md`）で担保する。品質チェックがパスしても、ゲートが省略を検出したら埋めるまで次へ進まない
- **品質チェック（typecheck + lint）はごまかさない** — すべてパスするまで修正する
- **spec ドキュメントを実装と同期させる** — 実装中に設計の誤りや不足を発見したら、実装に合わせて spec を更新する。乖離したまま放置しない
- **設計判断は `.adr/` に記録する** — 設計を変更・拡張した場合、その判断と理由を `.adr/{連番}-{topic}.md` に ADR として残す
- 委譲方式・並列実行・失敗時の扱い・委譲時のコンテキストは `subagent-policy.md` に従う
