# 完全性ゲート（レイヤー単位）

実装フェーズの直後・レビューサイクルの前に、実装したレイヤーに省略がないことを検証するゲート。各 implement-* スキルの「実装」と「レビューサイクル」の間で実行する。

## なぜこのゲートが必要か

typecheck + lint は型・構文しか見ない。スタブ（`return []`/固定値）・ダミーデータ・部分実装・spec 機能のまるごと未実装は型が通るため素通りする。だから完全性は別の手段で担保する必要がある。

- **レイヤー単位で前倒しする:** 省略をレイヤーごとに潰せば、後続が未完成の基盤の上に積み上がらない。最終段の implement-audit まで持ち越すと、実装当時のコンテキストが失われて再実装が高コストになる。
- **独立した検証者に任せる:** 実装者自身は自分の省略を「仕様の範囲外」「ダミーで十分」と正当化しがち。ゼロベースの独立したサブエージェントに「何が欠けているか」を探させて捉える（find → adversarially verify）。

## 入力

- 対象レイヤー（domain / adapter / usecase / test / frontend）
- そのレイヤーの spec（`spec/domains/`、`spec/usecases/`、`spec/database/`、`spec/pages/`、`spec/testcases/` 等）
- 実装したファイル群
- 実装サブエージェントが出力した**カバレッジ表**（spec 要素 → 実装箇所 → 状態）

## 手順

### 1. パターンスキャン（機械的・安価）

`../../implement-audit/references/detection-patterns.md` に定義されたパターンで、**対象レイヤーのファイルだけ**を Grep する。TODO/FIXME、`not implemented` 系、`return []`/`return {}` の固定返し、「省略」「ここに〜を追加」等のコメントを検出する。偽陽性フィルタ（テストファイル内の mock/stub、生成ファイル等）は同ドキュメントに従う。

### 2. spec 照合（網羅性）— 最重要

独立したサブエージェントを起動し、対象レイヤーの spec で定義された全要素が実装に存在し、かつスタブでないことを照合させる。実装者のカバレッジ表は出発点として渡すが、鵜呑みにせず実コードで裏取りさせる。

```markdown
あなたは実装の完全性を検証する独立した監査エンジニアです。実装した本人ではないため、忖度せず「何が欠けているか」を探してください。

## 前提
- まずプロジェクトルートの CLAUDE.md を読み、構成・規約を把握する

## 対象
- レイヤー: {domain/adapter/usecase/test/frontend}
- 設計: {そのレイヤーの spec ディレクトリ}
- 実装ファイル: {実装したファイル群}
- 実装者が申告したカバレッジ表: {カバレッジ表を貼る}

## やること
1. spec を読み、このレイヤーで実装されるべき要素を**自分で**漏れなく列挙する（実装者の申告に頼らず、spec から直接洗い出す）
2. 各要素について、実装コードを Read で開いて実在と完全性を確認する:
   - 要素が実装に存在するか（存在しない = まるごと未実装）
   - 設計で要求された振る舞いが実装されているか、それともスタブか
     （`return []`/固定値、ダミーデータ、空ボディ、`not implemented` 系はスタブ）
   - CRUD・分岐・エラー処理・バリデーション等が一部だけで残りが省略されていないか
3. カバレッジ表の各行が実コードと一致するか裏取りする。表が「完全」と言っていても実コードがスタブなら欠落として報告する

## 重要
- 構文が通ること（typecheck/lint がパスすること）は完全性の証明にならない。中身を見て判定する
- 意図的な設計（抽象メソッド、空実装が正しいケース）と省略を区別する。判断に迷うものは「要確認」とする

## 出力
欠落が1件もなければ「欠落なし」と明記する。あれば各項目について:
- spec 上の定義場所（ファイル:セクション）
- 欠落の種類（まるごと未実装 / スタブ / 部分実装 / TODO残り / 要確認）
- 実装側の該当ファイル:行（存在する場合）
- 何を実装すべきか（埋めるべき振る舞いの具体）
```

### 3. 判定とループ

- パターンスキャンと spec 照合で欠落・スタブ・部分実装が**1件でも**見つかったら、ゲート不通過。
- 修正は新しい実装サブエージェントに委譲する。**欠落リストを具体的に明示**して渡す（「全体を見直して」ではなく「この要素のこの振る舞いが未実装、埋めよ」）。実装者本人の正当化バイアスを避けるため、可能なら別のサブエージェントに埋めさせる。
- 修正後、typecheck + lint を実行し、ゲート（手順 1〜2）を再実行する。
- **欠落 0 件**になったらゲート通過 → レビューサイクルへ進む。
- 3 ラウンドで収束しない場合は、残っている欠落をまとめてユーザーに報告し、判断を委ねる。「要確認」だけが残っている場合もユーザーに確認する。

## 出力

- ゲート結果（PASS / 欠落リスト）を `.review/{layer}/completeness-{連番}.md` に記録する
- ゲート通過後、レビューサイクル（各スキルの Phase 3）に進む

## 委譲方式

委譲方式（サブエージェント / OS プロセス / 親内シミュレート）の判定と並列化の原則は `subagent-policy.md` に従う。spec 照合の対象が多い場合はドメイン・ディレクトリ単位でサブエージェントを分割して並列実行してよい。
