# アーキテクチャ監査の観点

architecture-audit が検査する観点のカタログ。各観点は独立しており、観点ごとにサブエージェントを並列起動する。

すべての観点に共通する前提:

- 判定基準は **Phase 1 で確定した規約**。この文書の説明は一般的なレイヤードアーキテクチャを想定した例であり、実際の規約はプロジェクトの CLAUDE.md / spec/ から読み取ったものを使う。レイヤー名・依存方向・配置はプロジェクトに合わせて読み替える。
- 各サブエージェントには必ず Phase 1 の規約メモを渡し、「まず CLAUDE.md を読む」よう指示する。
- 検出は**根拠（import 文・該当コード行）とともに**報告させる。「漏れてそう」ではなく「このファイルのこの import / このコードが規約のここに反する」という形にする。
- 意図的な例外（`.adr/` に記録された設計判断、明示的に許可された依存）は違反としない。判断に迷うものは「要確認」とする。

観点の目次:

1. レイヤー分離・配置
2. 依存方向
3. ポート準拠・依存性逆転
4. ドメインロジックの配置（漏出）
5. 責務の越境
6. 命名・構造規約

## 1. レイヤー分離・配置

**問い:** 各モジュール/ファイルが、規約で定められた正しいレイヤーのディレクトリに置かれているか。1つのファイルに複数レイヤーの関心事が混在していないか。

**検出方法:** ディレクトリ構成と各ファイルの中身を突き合わせる。ドメインのディレクトリに永続化処理が書かれている、ユースケースのファイルに UI 関心事がある、といった配置ミスを探す。

```text
あなたはアーキテクチャ監査を行うエンジニアです。レイヤー分離と配置を検査します。

## 前提
- まず CLAUDE.md を読み、プロジェクトの構成・規約を把握する
- 以下が Phase 1 で確定したアーキテクチャ規約です:
{規約メモを貼る}

## 対象
{対象スコープ}

## やること
1. 各レイヤーのディレクトリに、そのレイヤーに属さない関心事のコードが混ざっていないか確認する
2. 1ファイル内に複数レイヤーの責務が同居していないか確認する
3. 規約で定義されたレイヤーのどれにも属さない宙ぶらりんなコードがないか確認する

## 出力（違反ごと）
- 場所: ファイルパス:行番号
- 違反した規約: 規約メモのどの項目か
- 該当コードの抜粋
- なぜ問題か / あるべき配置
- 確信度（高/中/低）
```

## 2. 依存方向

**問い:** レイヤー間の依存（import / 参照）が規約で許された方向だけになっているか。逆転していないか。

典型的な違反（プロジェクトの規約に読み替える）:

- ドメインがアダプター・ユースケース・フレームワーク・インフラを import している
- ユースケースがアダプターの**具象実装**を直接 import している（ポート経由でない）
- フロントエンドがドメイン内部やアダプターを直接参照している（ユースケース/公開 API を経由していない）
- 内側のレイヤーが外側のレイヤーを知っている

**検出方法:** import 文（および動的参照）を起点に、依存元レイヤーと依存先レイヤーを判定し、規約で禁止された方向を抽出する。これが最も機械的に検出しやすい観点。

```text
あなたはアーキテクチャ監査を行うエンジニアです。レイヤー間の依存方向を検査します。

## 前提
- まず CLAUDE.md を読み、構成・規約を把握する
- Phase 1 で確定した依存方向の規約:
{規約メモを貼る（許可される依存方向・禁止される方向）}

## 対象
{対象スコープ}

## やること
1. 各ファイルの import / require / 参照を読み、依存元レイヤーと依存先レイヤーを特定する
2. 規約で禁止された方向の依存をすべて抽出する
3. 特にドメイン層が外部（インフラ・フレームワーク・他レイヤー）に依存していないかを重点的に見る

## 重要
- 型のみの import と値の import を区別できるなら区別する（型のみの依存は許容される場合がある — 規約に従う）
- .adr/ に記録された例外的依存は違反としない

## 出力（違反ごと）
- 場所: ファイルパス:行番号
- 依存元レイヤー → 依存先レイヤー
- 該当の import / 参照行
- 違反した規約
- 確信度（高/中/低）
```

## 3. ポート準拠・依存性逆転

**問い:** ドメインが定義するポート（インターフェース）に対し、アダプターが正しく実装しているか。ユースケースは具象ではなくポートに依存しているか。ポートの契約（メソッドシグネチャ・戻り値の型）が実装と一致しているか。

**検出方法:** ドメインのポート定義を列挙し、各ポートに対応する実装（アダプター）と利用側（ユースケース）を突き合わせる。spec/domains/ のポート定義・spec/database/ のアダプター設計があれば参照する。

```text
あなたはアーキテクチャ監査を行うエンジニアです。ポート準拠と依存性逆転を検査します。

## 前提
- まず CLAUDE.md を読み、構成・規約を把握する
- Phase 1 で確定した規約（ポート/依存性逆転の方針）:
{規約メモを貼る}
- 設計ドキュメント: spec/domains/（ポート定義）, spec/database/（アダプター）が存在すれば参照する

## やること
1. ドメインが定義するポート/インターフェースを列挙する
2. 各ポートについて、対応するアダプター実装がポートの契約（シグネチャ・型）通りか確認する
3. ユースケースがポート（抽象）に依存しているか、それとも具象アダプターを直接掴んでいるか確認する
4. ポートに定義があるのに実装がない、実装にあるのにポートにないメソッドを抽出する

## 出力（違反ごと）
- 場所: ファイルパス:行番号
- 違反の種類（具象への直接依存 / 契約不一致 / ポート未経由 等）
- 該当コードの抜粋
- あるべき構造
- 確信度（高/中/低）
```

## 4. ドメインロジックの配置（漏出）

**問い:** ビジネスルール・ドメイン知識が、本来あるべきドメイン層ではなく、ユースケース・アダプター・フロントエンドに漏れ出していないか。

典型的な漏出:

- ユースケースに計算ロジック・状態遷移ルール・不変条件の検証が直書きされている（本来はエンティティ/値オブジェクト/ドメインサービスの責務）
- アダプター（永続化層）にビジネス判断が紛れている
- フロントエンドにドメインルールが再実装されている

**検出方法:** ドメイン外のレイヤーのコードを読み、「これはビジネスルールでは？」という判断を要する。純粋なオーケストレーション（呼び出しの調整）と、ドメイン知識（業務上の意味を持つ判断）を区別する。意味解析が必要なので、機械検出ではなくサブエージェントの読解で行う。

```text
あなたはアーキテクチャ監査を行うエンジニアです。ドメインロジックの漏出を検査します。

## 前提
- まず CLAUDE.md を読み、構成・規約を把握する
- Phase 1 で確定した「各レイヤーの責務」規約:
{規約メモを貼る}
- spec/domains/ があれば、何がドメイン知識かの参考にする

## やること
1. ドメイン層以外（ユースケース・アダプター・フロントエンド）のコードを読む
2. ビジネスルール・業務上の判断・不変条件の検証・ドメイン固有の計算が、ドメイン層の外に書かれていないか探す
3. 純粋なオーケストレーション（複数の処理を順に呼ぶだけの調整）と、ドメイン知識（業務上の意味を持つ判断）を区別する — 前者は漏出ではない

## 重要
- 「設計通りユースケースが担うべき調整」と「ドメインに移すべきルール」を慎重に区別する。判断に迷うものは要確認とする
- バリデーションは規約による（値オブジェクトに置くプロジェクトもあれば、入力検証をユースケースに置くプロジェクトもある）。Phase 1 の規約に従う

## 出力（違反ごと）
- 場所: ファイルパス:行番号
- 漏出しているロジックの内容
- 本来あるべきレイヤー / 配置先
- なぜドメイン知識だと判断したかの根拠
- 確信度（高/中/低）
```

## 5. 責務の越境

**問い:** あるレイヤーが、別のレイヤーの責務を直接実行していないか（依存方向は守っていても責務を越えているケース）。

典型例:

- ユースケースが永続化の詳細（SQL・ORM 操作）を直接行っている（アダプターの責務）
- フロントエンドがバックエンドのユースケースを経由せずデータ操作している
- アダプターが複数ドメインにまたがる調整をしている（ユースケースの責務）
- ドメインが I/O（DB・HTTP・ファイル・時刻取得・乱数）を直接行い、純粋性を失っている

**検出方法:** 各レイヤーのコードが、そのレイヤーの責務範囲を超えた操作を行っていないか確認する。依存方向（観点2）と重なる場合もあるが、こちらは「何をやっているか」に注目する。

```text
あなたはアーキテクチャ監査を行うエンジニアです。責務の越境を検査します。

## 前提
- まず CLAUDE.md を読み、構成・規約を把握する
- Phase 1 で確定した「各レイヤーの責務」規約:
{規約メモを貼る}

## やること
1. 各レイヤーのコードが、そのレイヤーの責務を超えた操作をしていないか確認する
2. 特に以下を重点的に見る:
   - ドメイン層が I/O（DB・HTTP・ファイル・時刻・乱数等の副作用）を直接行っていないか（純粋性）
   - ユースケースが永続化の詳細を直接書いていないか
   - フロントエンドがユースケース/公開 API を経由せずにデータ操作していないか
3. 規約で許可された範囲（例: ユースケースがトランザクション境界を持つ等）は越境としない

## 出力（違反ごと）
- 場所: ファイルパス:行番号
- 越境の内容（どのレイヤーが何の責務を侵しているか）
- 該当コードの抜粋
- あるべき構造
- 確信度（高/中/低）
```

## 6. 命名・構造規約

**問い:** ファイル名・モジュール名・ディレクトリ構成・エラー処理の方式が、CLAUDE.md で定められた規約に従っているか。

典型例:

- 命名規約（ファイル名・型名・関数名のパターン）からの逸脱
- 規約と異なるディレクトリ構成
- エラー処理の方式が統一されていない（例: 規約が Result 型なのに例外を投げている、ドメインで throw している）
- 規約で定められたモジュール公開境界（barrel / index の方針等）の違反

**検出方法:** CLAUDE.md の規約と実装を突き合わせる。規約に明記がないものは違反としない（このスキルは「定められた規約への適合」を見るのであって、一般論のベストプラクティスを押し付けるものではない）。

```text
あなたはアーキテクチャ監査を行うエンジニアです。命名・構造規約への適合を検査します。

## 前提
- まず CLAUDE.md を読み、命名・構造・エラー処理の規約を把握する
- Phase 1 で確定した横断的規約:
{規約メモを貼る}

## やること
1. CLAUDE.md に明記された命名・ディレクトリ・エラー処理・公開境界の規約を列挙する
2. 各規約に対し、実装が従っているか確認する
3. 規約に明記がない事項は判定しない（一般論のベストプラクティスは押し付けない）

## 出力（違反ごと）
- 場所: ファイルパス:行番号
- 違反した規約（CLAUDE.md の該当箇所）
- 該当コードの抜粋
- 確信度（高/中/低）
```
