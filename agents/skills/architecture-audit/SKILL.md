# Architecture Audit — アーキテクチャ準拠の監査

実装コードがプロジェクトのアーキテクチャ・設計原則に従っているかをレイヤー横断で監査し、違反を検出・レポートし、重大な違反は Issue として起票する。

## なぜこのスキルが必要か

implement スキルファミリーは各レイヤーのレビューサイクルでアーキテクチャ準拠を確認するが、それは**実装中の各レイヤー内**のチェックにとどまる。Issue 対応・PR レビュー修正・リファクタリングが積み重なると、レイヤーをまたいだ構造的な劣化（依存方向の逆転、ドメインロジックの漏出、責務の越境）が少しずつ蓄積する。個々の変更では型も通りレビューも通るのに、全体として設計原則から逸れていく。このスキルはその逸れを**コードベース全体を横断して**洗い出す。

三兄弟の棲み分け:

| スキル | 問い | 見るもの |
|--------|------|---------|
| **implement-audit** | 完成してる？ | TODO・スタブ・仮実装・実装漏れ（**完全性**） |
| **spec-sync** | ドキュメント正しい？ | spec と実装の乖離（**ドキュメントの正確性**） |
| **architecture-audit** | 設計の構造に従ってる？ | レイヤー分離・依存方向・漏出・規約（**構造的健全性**） |

実装が「存在し」「ドキュメントと一致する」だけでは不十分で、「正しいレイヤーに正しい依存方向で置かれている」かは別の軸。このスキルがその軸を担う。

**このスキルはコードを修正しない。** 検出・レポートと、重大違反の Issue 起票までを行う。修正は implement スキルや issue-implement ワークフローに委ねる。

## Workflow

```text
Phase 1: 規約の確定
  CLAUDE.md / README.md / spec/ から、このプロジェクトの
  アーキテクチャ規約（レイヤー構成・依存方向・配置・命名）を読み取り明文化する

Phase 2: 監査（観点別・並列）
  references/audit-dimensions.md の観点ごとにサブエージェントを並列起動し、
  Phase 1 で確定した規約に照らして違反を検出する

Phase 3: レポート生成
  .audit/architecture/report.md に重要度分類つきで出力

Phase 4: Issue 起票
  Critical 違反を Issue として起票（レイヤー別に分割）

Phase 5: 結果報告
```

## Phase 1: 規約の確定

このスキルの判定基準は**プロジェクトごとに異なる**。特定のフレームワークやライブラリを前提にせず、まずこのプロジェクトの実際の規約を読み取って明文化する。ここを飛ばして一般論で判定すると偽陽性だらけになる。

CLAUDE.md / README.md / spec/ を読み、以下を確定する:

- **レイヤー構成と配置先** — どのレイヤーが存在し、それぞれどのディレクトリに置かれるか（domain / adapter / usecase / test / frontend は典型例。プロジェクトによって名称・粒度は異なる）
- **依存方向の規約** — どのレイヤーがどのレイヤーに依存してよいか。逆転してはいけない方向はどれか
- **各レイヤーの責務** — ドメインに置くべきもの、ユースケースに置くべきもの、アダプターに置くべきもの
- **横断的な規約** — エラー処理の方式（例外 / Result 型等）、バリデーションの置き場所、命名・ディレクトリ規約、ポート/依存性逆転の方針

読み取った規約を**作業メモとして明文化**し、ユーザーに提示してから Phase 2 に進む。規約が CLAUDE.md 等から読み取れない場合は、コードの実態から推定しつつ「この前提で監査する」と明示する。判断に迷う規約はユーザーに確認する。

## Phase 2: 監査（観点別・並列）

`references/audit-dimensions.md` を読み、そこで定義された観点ごとに監査する。観点は独立しているので**全観点を並列実行**する。各観点は1つのサブエージェントに担当させ、Phase 1 で確定した規約をコンテキストとして渡す。

各サブエージェントには必ず以下を渡す:

- Phase 1 で確定したアーキテクチャ規約（作業メモ）
- 担当する観点とその検出方法（`references/audit-dimensions.md` 該当節）
- 対象スコープ（指定がなければプロジェクト全体）

サブエージェントの結果を統合し、重複を排除する（同じファイル:行が複数観点で挙がることがある）。

### 対象スコープ

ユーザーが対象ディレクトリを指定していればそれに従う。指定がなければ**プロジェクト全体**を対象にする。diff や変更履歴に依存せず全体を探査することがこのスキルの価値。`node_modules`・生成ファイル・ロックファイル等は除外する。

## Phase 3: レポート生成

全観点の結果を統合し、`.audit/architecture/report.md` に出力する。各違反にレイヤーと重要度を付与する。

### 重要度の分類基準

- **Critical** — アーキテクチャの根幹を壊す違反。依存方向の逆転（例: domain → adapter）、レイヤーの越境（例: frontend が直接 DB を叩く）、ドメインロジックの重大な漏出。放置すると設計が崩壊する
- **Warning** — 設計原則からの逸脱だが局所的。ポートを経由せず具象に依存、軽微な責務の越境、命名/配置規約の違反
- **Info** — 意図的かもしれず確認が必要なもの。確信度が低い検出、判断が分かれるもの

```markdown
# Architecture Audit Report

**Date:** {日時}
**Scope:** {対象}
**規約の出典:** {CLAUDE.md / spec/ 等、Phase 1 の根拠}
**Total findings:** Critical {数} / Warning {数} / Info {数}

## サマリー（観点 × 重要度）

| 観点 | Critical | Warning | Info |
|------|----------|---------|------|
| レイヤー分離・配置 | {数} | {数} | {数} |
| 依存方向 | {数} | {数} | {数} |
| ポート準拠・依存性逆転 | {数} | {数} | {数} |
| ドメインロジックの配置 | {数} | {数} | {数} |
| 責務の越境 | {数} | {数} | {数} |
| 命名・構造規約 | {数} | {数} | {数} |

## Critical（要対応）

### [C-001] {違反の概要}

- **観点:** {観点}
- **レイヤー:** {layer}
- **場所:** `{パス}:{行番号}`
- **違反した規約:** {Phase 1 で確定したどの規約に反するか}
- **コード:**
  ```
  {該当コード}
  ```
- **なぜ問題か:** {構造的に何が崩れるか}
- **推奨対応:** {あるべき構造と、どのレイヤーをどう直すか}

## Warning（改善推奨）

### [W-001] {違反の概要}

（Critical と同じ項目構成）

## Info（要確認）

- **根拠:** {確認が必要な理由}

## 推奨アクション

| 優先度 | レイヤー | 違反の要点 | 推奨スキル |
|--------|---------|-----------|-----------|
| 1 | {layer} | Critical {数}件 | implement（Phase {N}）or issue-implement で対応 |
```

## Phase 4: Issue 起票

**Critical 違反**を `gh issue create` で起票する。Warning / Info は起票せずレポートにとどめる（ノイズを避けるため）。ユーザーが「全部 Issue にして」等と指定した場合はそれに従う。

関連する違反はまとめて1つの Issue にしてよいが、**レイヤーが異なる場合は分割**する（対応するスキル・担当が変わるため）。起票前に `gh issue list --label architecture-audit` で重複を確認する。

```bash
gh issue create \
  --title "[architecture-audit] {レイヤー}: {違反の概要}" \
  --body "$(cat <<'EOF'
## 概要
architecture-audit による監査で検出されたアーキテクチャ違反。

## 違反内容
| # | 観点 | 場所 | 違反した規約 | 内容 |
|---|------|------|------------|------|
| {番号} | {観点} | {パス:行} | {規約} | {何が問題か} |

## あるべき構造
{この違反を解消したときの正しい構造}

## 対応方針
{どのレイヤーをどう修正するか}

## 参照
- レポート: .audit/architecture/report.md
- 規約: {CLAUDE.md / spec の該当箇所}
EOF
)" \
  --label "architecture-audit"
```

`architecture-audit` ラベルが存在しない場合は事前に作成する:

```bash
gh label create architecture-audit --description "architecture-audit で検出されたアーキテクチャ違反" --color "c2e0c6"
```

## 原則

- **メインエージェントはオーケストレーションのみ** — 規約の読み取り・コード探査・違反検出はサブエージェントに委譲する
- **規約はプロジェクトから読み取る** — 特定のフレームワーク・ライブラリ・ディレクトリ名をハードコードしない。CLAUDE.md / spec/ が唯一の判定基準。これを怠ると偽陽性だらけになる
- **コードは修正しない** — このスキルの責務は検出・レポート・Issue 起票。修正は implement / issue-implement に委ねる
- **プロジェクト全体を横断する** — レイヤーをまたいだ構造劣化を捉えるのが価値。diff ではなく全体を見る
- **構造の健全性を見る（完全性・正確性ではない）** — 「実装が存在するか」は implement-audit、「ドキュメントが正しいか」は spec-sync の領分。本スキルは「正しいレイヤーに正しい依存方向で置かれているか」だけを見る
- **意図的な設計と違反を区別する** — 規約の例外（ADR に記録された設計判断等）は違反ではない。`.adr/` も確認し、判断に迷うものは Info（要確認）に分類する
- **重要度を正しく付ける** — 偽陽性は Info に落として後で除外できる。根幹を壊す違反だけ Critical にして Issue 化し、ノイズで埋もれさせない
- 委譲方式・並列実行・失敗時の扱い・委譲時のコンテキストは `../_shared/references/subagent-policy.md` に従う
