# Implement Frontend — フロントエンド実装＋レビュー

spec/pages/ の設計に基づき、フロントエンドをサブエージェントで実装する。spec/design/ が存在する場合はデザインに忠実に実装する。実装後にフロントエンドに特化したレビューサイクルを回す。

**メインエージェントの役割はオーケストレーションのみ。** コードの読み書きはすべてサブエージェントに委譲する。

## Phase 1: 準備

### spec/ の確認

以下が存在するか確認する:

- `spec/pages/` — ページ設計
- `spec/scenario/` — 利用シナリオ（UX確認用）
- `spec/design/` — UIデザイン成果物（任意）

### プロジェクトコンテキストの把握

CLAUDE.md と README.md を読み、以下を確認する:

- 使用言語・フレームワーク
- ディレクトリ構成（フロントエンドの配置先）
- typecheck・lint コマンド
- コーディング規約

### 進捗管理

`spec/progress.md` を確認し、フロントエンドの状態を把握する。バックエンド（ドメイン・アダプター・ユースケース）が実装済みであることを確認する。

### バックエンドAPI情報の把握

フロントエンドはバックエンドAPIに対してリクエストを送る必要がある。サブエージェントが正しいエンドポイントに接続できるよう、以下を事前に把握する:

1. APIルート定義・ハンドラーの配置ディレクトリを特定する（CLAUDE.md やプロジェクト構成から判断）
2. 実装済みのAPIエンドポイント一覧（パス・HTTPメソッド）を確認する

把握した情報はサブエージェントへの指示に含める（impl-guide.md のテンプレートの `{APIルート定義・ハンドラーのディレクトリ}` に設定する）。

### 実装例の確認

`docs/frontend_implementation_example.md` が存在するか確認する。存在する場合はサブエージェントに参照先として渡す。

## Phase 2: 実装

`references/impl-guide.md` を読み、その手順に従ってフロントエンドを実装する。

ページごとまたは画面グループごとにサブエージェントを起動する。

サブエージェント完了後:

1. 報告を確認する
2. typecheck + lint を実行する
3. エラーがあればサブエージェントを再起動して修正させる
4. `spec/progress.md` を更新する

## Phase 2.5: 完全性ゲート

全ページの実装が完了したら、レビューサイクルに進む前に `../_shared/references/completeness-gate.md` を読み、その手順に従って完全性ゲートを実行する。

typecheck + lint はダミーデータ表示・未接続のAPIスタブ・空のイベントハンドラーを素通りさせる。このゲートで、独立したサブエージェントが spec/pages/ の全ページ・画面要素 vs 実コードを照合し、ダミーデータで仮表示している箇所やバックエンド未接続を検出する。欠落があればその項目だけを埋めさせて再ゲートし、欠落 0 件になってから Phase 3 へ進む。

- 対象レイヤー: frontend
- spec: `spec/pages/` ＋ バックエンドAPI実装
- 入力: 実装したファイル群＋各サブエージェントのカバレッジ表

## Phase 3: レビューサイクル

完全性ゲートを通過したら、`references/review-guide.md` を読み、その手順に従ってレビューと修正を行う。

**2回連続で Blocker 0件 かつ Warning 0件** になったらレビュー完了。最大5ラウンド。5ラウンドで収束しない場合はユーザーに報告して判断を委ねる。

レビュー記録は `.review/frontend/review-{連番}.md` に保存する。

## Phase 4: 完了

1. サブエージェントから報告された設計との乖離を確認し、必要に応じて:
   - `spec/pages/` の該当ドキュメントを更新する
   - 設計判断があった場合は `.adr/{連番}-{topic}.md` に記録する
2. git commit する
3. `spec/progress.md` を更新する
4. サマリーを出す

## 原則

共通原則は `../_shared/references/implement-principles.md` に従う。本レイヤー固有:

- **spec/design/ のデザインに忠実に実装する** — HTMLデザインが存在する場合、レイアウト・色・タイポグラフィ・スペーシングをデザイントークンに合わせて再現する
- ユースケース（バックエンド API）が提供する実際のエンドポイントに合わせ、ダミーデータで代替しない
