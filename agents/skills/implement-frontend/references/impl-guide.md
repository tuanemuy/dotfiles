# フロントエンド実装ガイド

`spec/pages/` の設計に基づいて、フロントエンドを実装する。

## サブエージェントへの指示テンプレート

ページごとまたは画面グループごとにサブエージェントを起動する。

```markdown
あなたは設計ドキュメントに基づいてフロントエンドを実装するエンジニアです。

## 前提
- まずプロジェクトルートの CLAUDE.md を読み、使用言語・ディレクトリ構成・規約・コマンドを把握する

## 対象
- ページ設計: spec/pages/index.md
- 対象ページ: {page名またはグループ}
- UIデザイン（存在する場合）: spec/design/pages/{対象ページ}.html
- デザイントークン（存在する場合）: spec/design/tokens.md
- デザイン方針（存在する場合）: spec/design/index.md
- ユースケース設計: spec/usecases/
- バックエンドAPI実装: {APIルート定義・ハンドラーのディレクトリ}

## やること
1. CLAUDE.md を読み、プロジェクトの構成・規約を把握する
2. spec/pages/index.md を読み、対象ページの設計を理解する
3. **対象ページとその画面要素を漏れなく洗い出す** — 各ページ・主要コンポーネント・フォーム・テーブル・ボタン・データ取得/更新操作を一覧にする。これが実装すべき全項目であり、後でカバレッジ表になる
4. spec/design/ が存在する場合、対象ページのHTMLデザイン・デザイントークン・デザイン方針を読み、ビジュアル面の仕様を把握する
5. 関連するシナリオ（spec/scenario/）を確認する
6. **バックエンドAPIの実装コードを読み、対象ページが使うAPIエンドポイント（パス・HTTPメソッド・リクエスト/レスポンスの型）を把握する** — spec/usecases/ の設計だけでなく、実際のルート定義やハンドラーの実装を確認する。フロントエンドはバックエンドの実装済みAPIに対してリクエストを送る必要があるため、実装コードが正確な情報源になる
7. 実装例があれば参照する（{実装例パス}）
8. 既存の実装があれば Read で読んで現状を把握する
9. 手順3で洗い出した全ページ・全要素を設計とデザインに従って実装する。一部のページ・要素だけ実装して残りを省略しない
10. **フロントエンドからバックエンドへのAPI呼び出しを実装する** — 手順6で把握したエンドポイントに対して、実際のHTTPリクエスト（fetch, axios, Server Actions 等プロジェクトの方式に従う）を実装する。スタブ・ダミーデータ・仮のレスポンスで代替しない
11. CLAUDE.md に記載の typecheck・lint コマンドを実行し、エラーがなくなるまで修正する

## 重要な原則
- 適切な粒度でコンポーネントを分割する
- ビジネスロジック（計算・判定・変換）はフレームワーク非依存の純粋関数として切り出す。view は state => UI の純粋関数に保ち、レンダリング中に副作用を実行しない
- 状態は直和型（判別可能なユニオン）で組み、不正な組み合わせを型で排除する。外部データ（APIレスポンス等）は境界でデコードしてドメイン型に変換し、内部では非nullアサーションやガード節に頼らない
- 複数状態を持つ画面の遷移はステートマシンとして明示する。副作用（fetch・ストレージ・DOM）は最外殻に隔離する
- **spec/design/ のデザインに忠実に実装する** — HTMLデザインが存在する場合、レイアウト・色・タイポグラフィ・スペーシングをデザイントークンに合わせて再現する
- 既存コードのスタイル・規約に合わせる
- ディレクトリ構成は CLAUDE.md とプロジェクトの既存構造に従う
- **バックエンドAPIとの接続を完全に実装する** — ユースケース（バックエンドAPI）が提供する実際のエンドポイントに合わせてリクエストを送る。ハードコードされたダミーデータや空配列を返すスタブ関数で代替しない。APIクライアントやデータ取得関数は、実際のHTTPリクエストを行う実装にする
- 設計に問題がある場合は実装せずにその旨を報告する

## 省略を残さない

ダミーデータ/ハードコードされた配列で画面を埋める・HTTP リクエストをせず `return []`/固定値を返すデータ取得関数・`not implemented` 系や空のイベントハンドラー・フォーム送信/バリデーション/エラー表示の省略・ページや画面要素のまるごと未実装・「// 省略」等のコメントによる省略は、typecheck が通っても「未完成」。設計で要求された画面とデータ操作を本実装する。実装できない要素は黙ってダミーで埋めず、カバレッジ表に「未実装」と理由付きで明記する（メインエージェントが判断）。

## 出力
- 変更したファイルの一覧と変更内容の要約
- **カバレッジ表（必須）** — 手順3で洗い出した全ページ・主要要素を1行ずつ列挙し、実装箇所・接続したAPI・状態を示す。未実装・部分実装も必ず行に残す:

  | ページ/要素 | 実装ファイル:行 | 接続API(あれば) | 状態 |
  |------------|----------------|----------------|------|
  | {ページ名/フォーム/テーブル/データ操作} | path:line | METHOD /path | 完全 / 部分 / 未実装 |

  ダミーデータで仮表示している箇所は「部分」とし、理由を添える。
- typecheck・lint の実行結果
- **接続したバックエンドAPIエンドポイントの一覧**（パス・メソッド・用途）
- 設計との乖離（あれば: 乖離の内容・理由・実装での対応方針）
- 判断に迷った点や懸念事項（あれば）
```

## 完了確認

サブエージェント完了後、メインエージェントが typecheck + lint を実行して確認する。エラーがあればサブエージェントを再起動して修正させる。
