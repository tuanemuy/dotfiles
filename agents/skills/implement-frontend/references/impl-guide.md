# フロントエンド実装ガイド

`spec/pages/` の設計に基づいて、フロントエンドを実装する。

## サブエージェントへの指示テンプレート

ページごとまたは画面グループごとにサブエージェントを起動する。

```
あなたは設計ドキュメントに基づいてフロントエンドを実装するエンジニアです。

## 前提
- まずプロジェクトルートの CLAUDE.md を読み、使用言語・ディレクトリ構成・規約・コマンドを把握する

## 対象
- ページ設計: spec/pages/index.md
- 対象ページ: {page名またはグループ}
- UIデザイン（存在する場合）: spec/design/pages/{対象ページ}.html
- デザイントークン（存在する場合）: spec/design/tokens.md
- デザイン方針（存在する場合）: spec/design/index.md
- ユースケース設計: spec/usecases/
- バックエンドAPI実装: {APIルート定義・ハンドラーのディレクトリ}

## やること
1. CLAUDE.md を読み、プロジェクトの構成・規約を把握する
2. spec/pages/index.md を読み、対象ページの設計を理解する
3. spec/design/ が存在する場合、対象ページのHTMLデザイン・デザイントークン・デザイン方針を読み、ビジュアル面の仕様を把握する
4. 関連するシナリオ（spec/scenario/）を確認する
5. **バックエンドAPIの実装コードを読み、対象ページが使うAPIエンドポイント（パス・HTTPメソッド・リクエスト/レスポンスの型）を把握する** — spec/usecases/ の設計だけでなく、実際のルート定義やハンドラーの実装を確認する。フロントエンドはバックエンドの実装済みAPIに対してリクエストを送る必要があるため、実装コードが正確な情報源になる
6. 実装例があれば参照する（{実装例パス}）
7. 既存の実装があれば Read で読んで現状を把握する
8. 設計とデザインに従ってページを実装する
9. **フロントエンドからバックエンドへのAPI呼び出しを実装する** — 手順5で把握したエンドポイントに対して、実際のHTTPリクエスト（fetch, axios, Server Actions 等プロジェクトの方式に従う）を実装する。スタブ・ダミーデータ・仮のレスポンスで代替しない
10. CLAUDE.md に記載の typecheck・lint コマンドを実行し、エラーがなくなるまで修正する

## 重要な原則
- 適切な粒度でコンポーネントを分割する
- **TODO・FIXME・仮実装を残さない** — 設計で定義された全機能を完全に実装する。実装が困難な場合は TODO ではなくその旨を報告する
- **spec/design/ のデザインに忠実に実装する** — HTMLデザインが存在する場合、レイアウト・色・タイポグラフィ・スペーシングをデザイントークンに合わせて再現する
- 既存コードのスタイル・規約に合わせる
- ディレクトリ構成は CLAUDE.md とプロジェクトの既存構造に従う
- **バックエンドAPIとの接続を完全に実装する** — ユースケース（バックエンドAPI）が提供する実際のエンドポイントに合わせてリクエストを送る。ハードコードされたダミーデータや空配列を返すスタブ関数で代替しない。APIクライアントやデータ取得関数は、実際のHTTPリクエストを行う実装にする
- 設計に問題がある場合は実装せずにその旨を報告する

## 出力
- 変更したファイルの一覧と変更内容の要約
- typecheck・lint の実行結果
- **接続したバックエンドAPIエンドポイントの一覧**（パス・メソッド・用途）
- 設計との乖離（あれば: 乖離の内容・理由・実装での対応方針）
- 判断に迷った点や懸念事項（あれば）
```

## 完了確認

サブエージェント完了後、メインエージェントが typecheck + lint を実行して確認する。エラーがあればサブエージェントを再起動して修正させる。
