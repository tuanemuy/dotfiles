# フロントエンド層レビューガイド

フロントエンドの実装が完了した後、設計・デザイン準拠とコンポーネント品質をレビューする。レビューサイクルの共通構造（出力フォーマット・修正ループ・完了条件・レビュー記録テンプレート）は `../../_shared/references/review-loop.md` に従う。本ガイドではフロントエンド固有の観点とサブエージェントへの指示だけを定義する。

```markdown

## レビュー観点

### 設計・デザイン準拠
- 設計通りに実装されているか
- デザインに忠実か（HTMLデザイン・デザイントークンとの乖離がないか）
- 共通コンポーネントが適切に定義・再利用されているか
- ドメイン固有のコンポーネントやロジックが適切に切り出されているか
- TODO・FIXME・仮実装が残っていないか
- **バックエンドAPIとの接続が完全に実装されているか** — データ取得・作成・更新・削除の各操作が実際のエンドポイントにリクエストを送っているか。ダミーデータを返すスタブ関数やハードコードされた仮データが残っていないか

### 構造とモデル化（ビュー/ロジック分離・不正状態の排除）
- **境界でのデコード** — APIレスポンス・ストレージ・URLパラメータなど外部データが、境界でパース・検証されてドメイン型に変換されているか。`as` キャストや未検証の raw データがそのまま内部に流れていないか
- **直和型・直積型によるモデリング** — 排他的な状態は直和型（判別可能なユニオン）、共起する値は直積型で組まれているか。boolean フラグの組み合わせや optional の羅列で、あり得ない組み合わせが型上表現可能になっていないか
- **ランタイム防御への依存** — 内部状態の整合性を `if (!x) throw`・非nullアサーション・散発的なガード節で守っていないか。これらはモデルが不正状態を許しているシグナルであり、型の再設計を提案する。境界（外部データの検証）でのランタイム検証は正当であり指摘対象外
- **ビジネスロジックの純粋性** — 計算・判定・変換が React にもフレームワークにも依存しない純粋関数として書かれ、レンダリング無しでテストできるか。hooks やコンポーネント内にロジックがベタ書きされていないか
- **状態遷移の明示** — 複数状態を持つ画面（読み込み→成功/失敗、編集→送信中→完了 等）の遷移がステートマシンとして明示されているか。イベントハンドラ内のアドホックな状態更新の積み重ねで暗黙に遷移していないか
- **副作用の隔離** — fetch・ストレージ・DOM 操作などの副作用が最外殻に隔離されているか。可能なら値として記述され端で実行されているか。ロジックの中に副作用が混ざっていないか
- **view の純粋性** — view は state => UI の純粋関数に保たれているか。レンダリング中に副作用やビジネスロジックが実行されていないか

## 重要: 独立したレビュー
前回のラウンドとは独立している。設計ドキュメントと実装コードだけを見て、ゼロからレビューする。

## 出力フォーマット
#### Blockers（マージ不可の問題）
- **[B-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ問題か
  - 提案: どう直すべきか

#### Warnings（改善推奨）
- **[W-001]** 問題の説明（場所・理由・改善案）

Blocker がなければ「なし」と書く。設計からの逸脱、API未接続・ダミーデータの残存、TODO・FIXME・仮実装の残存は Blocker とする。構造とモデル化の観点では、未検証の外部データがドメインに流入している場合と、型が許す不正な状態に実際に到達し得てバグの温床になっている場合は Blocker、モデリングや分離の改善余地に留まるものは Warning とする。
```
