# Implement Test — テスト実装＋レビュー

spec/testcases/ の定義に基づき、テストをサブエージェントで実装する。実装後にテストに特化したレビューサイクルを回し、仕様カバレッジを担保する。

**メインエージェントの役割はオーケストレーションのみ。** コードの読み書きはすべてサブエージェントに委譲する。

## Phase 1: 準備

### spec/ の確認

以下が存在するか確認する:

- `spec/testcases/` — テストケース定義
- `spec/usecases/` — ユースケース設計（テスト対象の仕様参照用）

### プロジェクトコンテキストの把握

CLAUDE.md と README.md を読み、以下を確認する:

- 使用言語・フレームワーク
- テストフレームワーク・テスト配置先
- typecheck・lint・test コマンド
- コーディング規約

### 進捗管理

`spec/progress.md` を確認し、テストの状態を把握する。ドメイン層・アダプター層・ユースケース層が実装済みであることを確認する。

## Phase 2: 実装

`references/impl-guide.md` を読み、その手順に従ってテストを実装する。テストケース定義（spec/testcases/）をベースにしつつ、実装コードの分析と `../_shared/references/e2e-test-methodology.md` のE2Eテスト方法論に基づいて、仕様を網羅的にカバーする。

ドメインごとにサブエージェントを起動する。

サブエージェント完了後:

1. 報告を確認する
2. typecheck + lint を実行する
3. **テストを実行する**（CLAUDE.md のテストコマンドに従う）
4. 自動化を見送った項目がある場合、妥当性を確認する
5. テストが失敗したらサブエージェントを再起動して修正させる（最大3回）
6. `spec/progress.md` を更新する

## Phase 3: レビューサイクル

全テストの実装が完了したら、`references/review-guide.md` を読み、その手順に従ってレビューと修正を行う。

**2回連続で Blocker 0件 かつ Warning 0件** になったらレビュー完了。最大5ラウンド。5ラウンドで収束しない場合はユーザーに報告して判断を委ねる。

レビュー記録は `.review/test/review-{連番}.md` に保存する。

## Phase 4: 完了

1. サブエージェントから報告された設計との乖離を確認し、必要に応じて:
   - `spec/testcases/` の該当ドキュメントを更新する
   - 設計判断があった場合は `.adr/{連番}-{topic}.md` に記録する
2. git commit する
3. `spec/progress.md` を更新する
4. サマリーを出す

## 原則

- **メインエージェントはオーケストレーションのみ** — コードの読み書きはサブエージェントに委譲する
- **プロジェクトコンテキストに従う** — 言語・ディレクトリ構成・規約は CLAUDE.md / README.md から読み取る
- 仕様に合わせたテストを作成する — 実装に合わせない
- テストケース定義のカバレッジを網羅する
- **TODO・FIXME・仮実装を残さない**
- 品質チェック（typecheck + lint + test）はごまかさない
- **spec ドキュメントを実装と同期させる** — 設計の誤りや不足を発見したら spec を更新する
- **設計判断は `.adr/` に記録する** — 設計を変更・拡張した理由を ADR として残す
- 委譲方式・並列実行・失敗時の扱い・委譲時のコンテキストは `../_shared/references/subagent-policy.md` に従う
