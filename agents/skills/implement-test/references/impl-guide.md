# テスト実装ガイド

`spec/testcases/` の定義と `../../_shared/references/e2e-test-methodology.md` の方法論に基づいて、仕様を網羅するテストを設計・実装する。

## テスト設計のアプローチ

テストケース定義（spec/testcases/）をベースにしつつ、以下の3つの情報源を総合的に分析してテストを設計する:

1. **spec/testcases/**: テストすべきケースの定義
2. **spec/usecases/**: ユースケースの仕様（テスト対象の振る舞い）
3. **実装コード**: 実際の分岐・エッジケースの把握

spec/testcases/ の定義をそのまま機械的にテスト化するのではなく、実装を確認した上で抜け漏れがないかを検証する。E2Eテスト方法論の「テストケース設計の5つのカテゴリ」（正常系・境界値・異常系・状態遷移・既存機能への影響）を意識して、spec/testcases/ でカバーされていない観点も補完する。

## サブエージェントへの指示テンプレート

ドメインごとにサブエージェントを起動する。

```markdown
あなたは設計ドキュメントに基づいてテストを設計・実装するエンジニアです。

## 前提
- まずプロジェクトルートの CLAUDE.md を読み、使用言語・ディレクトリ構成・規約・コマンドを把握する

## 対象
- ドメイン: {domain名}
- テストケース定義: spec/testcases/{domain}/
- ユースケース設計: spec/usecases/{domain}.md

## やること
1. spec/testcases/{domain}/ のテストケース定義を読む
2. spec/usecases/{domain}.md でユースケースの仕様を把握する
3. 対応する実装コードを Read で読み、実際の振る舞い・分岐・エッジケースを理解する
4. プロジェクトの既存テスト構成を調査する
   - テストフレームワーク、ディレクトリ構成、設定ファイル、既存テストを Glob/Grep/Read で確認する
   - CLAUDE.md や README.md にテスト関連の記述があればそれも参照する
5. テストケース定義に基づきつつ、以下の観点を網羅するテストを設計する:
   - 正常系: 要件が満たされていることの検証
   - 境界値・エッジケース: 入力の境界、空値、上限・下限
   - 異常系: エラー時の振る舞い、バリデーション
   - 状態遷移: 前提条件が異なる場合の挙動
   - 既存機能への影響: 変更が既存の動作を壊していないこと
6. 既存のテストパターン・規約に合わせてテストを実装する
7. CLAUDE.md に記載の typecheck・lint コマンドを実行し、エラーがなくなるまで修正する

## 重要な原則
- 仕様に合わせたテストを作成する — 実装の内部構造ではなく振る舞いをテストする
- テストケース定義のカバレッジを網羅しつつ、実装から見えた追加観点も含める
- 既存のテストフレームワーク・パターンに合わせる — 新しいフレームワークを導入しない
- テストは安定して再現可能であること — フレイキーなテストは書かない
- テストファイル名・配置は既存の規約に従う
- UIの目視確認など自動化が困難な項目は無理に書かない
- やむを得ずリンタールールに違反する場合は、直前の行にコメントで理由を追記する

## 省略を残さない

テストの省略は「ケースを書いたフリ」になりやすい。`.skip`/`xit`/`.only` での無効化や対象外し・アサーションのない空テスト・`expect(true).toBe(true)` 等の中身のないアサーション・定義済みケースのまるごと未実装・「// 後でテスト追加」等のコメントだけで実テストがない、は残さない。自動化が困難で意図的に見送る項目（UI目視確認等）は省略と区別し、カバレッジ表に「見送り」と理由付きで明記する（黙って飛ばさない）。

## 出力
- 作成したテストファイルの一覧
- **カバレッジ表（必須）** — spec/testcases/ で定義された全ケースを1行ずつ列挙し、テスト箇所と状態を示す。未実装・見送りも必ず行に残す:

  | テストケース | テストファイル:行 | 状態 |
  |-------------|------------------|------|
  | {ケースID/名（正常系/境界値/異常系/状態遷移）} | path:line | 実装済 / 見送り / 未実装 |

  実装から見えた追加観点（spec/testcases/ にない補完テスト）も行を足して追記理由を添える。「見送り」「未実装」には理由を添える。
- typecheck・lint の実行結果
- 自動化を見送った観点とその理由（あれば）
- 設計との乖離（あれば: 乖離の内容・理由・実装での対応方針）
```

## 完了確認

サブエージェント完了後、メインエージェントが以下を実行する:
1. typecheck + lint
2. **テスト実行**（CLAUDE.md のテストコマンドに従う）
3. 自動化を見送った項目がある場合、妥当性を確認する
4. 失敗するテストがあればサブエージェントを再起動して修正させる（最大3回）
