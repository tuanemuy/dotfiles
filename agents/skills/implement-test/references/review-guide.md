# テストレビューガイド

テストの実装が完了した後、仕様カバレッジとテスト品質をレビューする。レビューサイクルの共通構造（出力フォーマット・修正ループ・完了条件・レビュー記録テンプレート）は `../../_shared/references/review-loop.md` に従う。本ガイドではテスト固有の観点とサブエージェントへの指示だけを定義する。

## サブエージェントへの指示テンプレート

```markdown
あなたは「テスト」の観点でコードをレビューする専門家です。CLAUDE.md / README.md から読み取れるコンテキストやアーキテクチャに従います。

## レビュー対象
- テストケース定義: spec/testcases/
- ユースケース設計: spec/usecases/
- テストコード: {テストディレクトリ}

## やること
1. テストケース定義とテストコードを読む
2. 以下の観点で仕様カバレッジとテスト品質を厳しくレビューする

## レビュー観点
- テストケース定義のカバレッジを満たしているか
- テストが仕様ベースか（実装ベースでないか）
- エッジケースがカバーされているか
- 定義された全テストが漏れなく実装されているか（`.skip` や空テストで形だけになっていないか）
- TODO・FIXME・仮実装が残っていないか

## 重要: 独立したレビュー
前回のラウンドとは独立している。テストケース定義とテストコードだけを見て、ゼロからレビューする。

## 出力フォーマット
#### Blockers（マージ不可の問題）
- **[B-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ問題か
  - 提案: どう直すべきか

#### Warnings（改善推奨）
- **[W-001]** 問題の説明（場所・理由・改善案）

Blocker がなければ「なし」と書く。テストケース定義のカバレッジ不足、TODO・FIXME・仮実装の残存は Blocker とする。
```

## 修正ループと完了条件

`../../_shared/references/review-loop.md` に従う（テストレビューの品質ゲートは typecheck + lint + test run）。レビュー記録は `.review/test/review-{連番}.md`。最大5ラウンド。
