# Issue Dashboard

リポジトリの全オープン Issue を横断分析し、着手の全体像を俯瞰できるダッシュボードを **GitHub Issue として** 作る。ダッシュボードもリポジトリの一部として Issue 上で共有・更新できるようにするのが狙い。

各 Issue のカテゴリー・優先度・依存先は AI が内容を読んで判断する。機械的なスコアリングやトポロジカルソートはしない。Issue を読めば、何が独立したテーマか、どれが重要か、何に依存しているかは判断できる。判断した結果を、構造そのもので意味が伝わる Markdown に落とす。あわせて、オープン PR が紐づく Issue は作業中として印を付け、誰かが着手済みであることが一目で分かるようにする。

既にダッシュボード Issue があれば、その後に増えた新規 Issue と close 済みの Issue を反映して整理・更新する。なければ新規作成する。

## Workflow

1. 既存のダッシュボード Issue を探す（Step 1）
2. 全オープン Issue とオープン PR を取得する（ダッシュボード Issue 自身は除く）（Step 2）
3. 各 Issue を分析する。解析範囲は既存ダッシュボードの有無で決める（下記「解析の範囲」）。カテゴリー・優先度・依存先を判断し、Issue が多ければサブエージェントで分担（Step 3）
4. ダッシュボード本文を組み立てる（Step 4）
5. 既存があれば更新、なければ新規作成する（Step 5）
6. 完了報告する（Step 6）

### 解析の範囲（全件 / 差分）

ダッシュボードは見るたびに同じ構造であってほしい。全 Issue を毎回ゼロから判断し直すと、構造に変化がなくてもカテゴリー分けや並び順が揺れて差分が無駄に荒れるし、Issue が多いほど重い。そこで既存ダッシュボードの有無で解析範囲を分ける。

- **新規作成のとき（既存ダッシュボードなし）**、または **ユーザーが明示的に「整理し直して」「並べ替えて」と言ったとき** は、全件をフル解析する（Step 3〜5 をそのまま全 Issue に適用）。
- **既存ダッシュボードを更新するとき** は差分だけ扱う。既存本文のカテゴリー分け・並び順・依存サブリストはそのまま引き継ぎ、(a) 新しく増えた Issue だけを分析して適切なセクションに挿入し、(b) close 済みの Issue を本文から外し、(c) 作業中マーカーをオープン PR から全件付け直す。構造の作り直しはしない。

作業中マーカーの付け直し（c）はオープン PR を見るだけで AI 判断を要さないので、差分更新でも毎回全 Issue に対して行う。これが Issue 実装ワークフロー（issue-implement の Phase 6）からの「1 件を作業中にするだけ」の呼び出しを軽く保つ要。

## Step 1: 既存ダッシュボード Issue の確認

open な Issue から、タイトルが `Issue Dashboard` のものを探す。

```bash
gh issue list --state open --search "Issue Dashboard in:title" --json number,title,body
```

本文の先頭に `<!-- issue-dashboard -->` マーカーがあるかで最終確認する（同名タイトルの誤検出を避けるため）。見つかればその Issue 番号と現在の本文を控える。見つからなければ新規作成扱いにする。

## Step 2: 全オープン Issue とオープン PR の取得

```bash
gh issue list --state open --limit 200 --json number,title,body,labels,milestone,assignees,url
```

ダッシュボード Issue 自身は分析対象から除く。本文は依存・カテゴリーの判断に使うので含める。取得できない場合（リポジトリ未指定・権限不足）はユーザーに確認する。推測で進めない。件数が多い（目安 30 件超）場合は Step 3 をサブエージェントで分担する（委譲方式は `_shared/references/subagent-policy.md`。各サブエージェントには全 Issue の番号・タイトル一覧を渡し、依存先を正しく参照できるようにする）。

続けてオープンな PR も取得する。誰かが既に着手して PR を出している Issue を「作業中」と示すための材料で、ダッシュボードを見た人が二重着手を避けられるようにするのが狙い。

```bash
gh pr list --state open --limit 200 --json number,title,headRefName,closingIssuesReferences
```

PR がどの Issue に紐づくかは 2 つの手がかりで判断する。`closingIssuesReferences` に並ぶ Issue 番号（PR が close 予定として明示的にリンクしている Issue）と、`headRefName` が `issue/{番号}/...` 形式のときのその番号。両者を合わせて「オープン PR が存在する Issue 番号」の集合を作る。

## Step 3: 各 Issue の分析

カテゴリー・優先度・依存先のカテゴリー判断（下記 3 つ）は重い解析なので、上の「解析の範囲」に従って対象を絞る。フル解析なら全 Issue が対象。差分更新なら新しく増えた Issue だけが対象で、既存 Issue は本文の判断を引き継ぐ。**作業中かどうか**は軽い判定なので、どちらの場合も全 Issue に対して付け直す。

各 Issue について次の 3 つを判断する。

**カテゴリー。** 独立したテーマで Issue をグループ分けし、内容を表す適切な見出しを付ける（例: 認証 / プロジェクト管理 / タスク管理 / リリース基盤 / ドキュメント など）。ラベルがテーマをよく表していれば参考にしてよいが、ラベルに縛られず内容で判断する。重要度の高いカテゴリーを上に置く。

**優先度。** ビジネス価値・緊急度（マイルストーンの期日、ブロッカー性、バグの深刻度）と、「これを終わらせると多くの Issue が動き出す」という波及度から、各カテゴリー内での並び順を決める。重要なものほど上。

**依存先。** その Issue を着手する前に先に完了している必要がある Issue を特定する。本文の `#123` 参照・`- [ ] #123` のタスクリスト・GitHub の sub-issue といった構造的な手がかりをまず拾い、さらに内容から読み取れる論理的な依存（例: 認証基盤がないとこの機能は作れない）も拾う。ただし保守的に判断する。確信が持てない弱い関連は依存にしない（過剰な依存は全体を不必要に直列に見せてしまう）。sub-issue は次で取得できる。

```bash
gh api repos/{owner}/{repo}/issues/{番号}/sub_issues --jq '.[].number' 2>/dev/null
```

**作業中かどうか。** Step 2 で作った「オープン PR が存在する Issue 番号」の集合に入っていれば、その Issue は誰かが着手中なので作業中として扱う。優先度・依存とは別軸の状態で、依存サブリストは通常どおり付けたうえで、行末に作業中の印を添える（次の Step 4 参照）。

## Step 4: ダッシュボード本文の組み立て

Step 3 の判断をそのまま Markdown 構造に落とす。フォーマットの規則は次のとおり。

- 本文の先頭に `<!-- issue-dashboard -->` マーカーを置く（次回の検出用）。
- 独立したカテゴリーごとに `## 見出し` セクションに分け、重要度の高い順に並べる。
- 見出しの直後は空行を 1 行空けてからリストを書く。
- セクション内は Issue を優先度の高い順に `- #番号 タイトル` で箇条書きにする。
- その Issue が作業中（オープン PR が存在する）なら、タイトルの直後に半角スペースを空けて `(In Progress)` を付ける。例: `- #102 ユーザー認証API (In Progress)`。作業中でなければ付けない。
- その Issue の依存先（先に終わらせる必要がある Issue）を、番号だけのサブリストで列挙する。複数あれば全部並べる。依存先がなければサブリストは書かない。
- テーブル・太字・絵文字・罫線・説明文などの装飾は使わない。構造だけで伝える。
- 循環依存・競合（同じ領域を触り並行着手でコンフリクトしやすい Issue）など順序に乗らない例外があれば、末尾に空行を空けて `Cycle #6 #7` `Conflict #111 #112` のような最小限の 1 行で添える。なければ何も書かない。
- 段落やリスト項目の途中で改行を入れない。1 項目は 1 行に収める。

Issue のタイトルが各 Issue を表すので、本文に `# リポジトリ名` のような H1 は付けない。

### 本文の例

```markdown
<!-- issue-dashboard -->

## 認証

- #102 ユーザー認証API（サインアップ・ログイン） (In Progress)
  - #101
- #105 ログイン画面のUI
  - #102

## プロジェクト・タスク管理

- #107 タスクボード画面（カンバン）
  - #104
  - #106
- #103 プロジェクトCRUD API
  - #102
- #104 タスクCRUD API
  - #103
- #106 プロジェクト一覧・詳細画面
  - #103
  - #105
- #111 タスクへのコメント機能
  - #104
  - #107
- #112 タスク詳細モーダルのデザイン改善
  - #107

## 基盤・リリース

- #101 DBスキーマとマイグレーション基盤の整備
- #110 本番環境へのCI/CDパイプライン構築
  - #101
  - #107

## その他

- #108 READMEのセットアップ手順を更新
- #109 アプリのロゴとファビコンを差し替え

Conflict #111 #112
```

依存先がない Issue（#101, #108, #109 など）はサブリストがないので、すぐ着手できると一目でわかる。サブリストに並ぶ番号が、その Issue の前に片付けるべきもの。同じカテゴリー内で依存先のない Issue 同士は並行して進められる。`(In Progress)` が付いた Issue（#102）は既に PR が出ていて着手中なので、重ねて手を出さないよう避ける。

## Step 5: Issue の作成 または 更新

本文を一時ファイルに書き出してから反映する。

**既存のダッシュボード Issue があった場合（差分更新）。** 旧本文をベースに、変わった分だけ手を入れる。構造を作り直さないことで、見るたびにカテゴリー分けや並び順がブレるのを防ぐ。

1. 旧本文に載っている Issue 番号の集合と、今回のオープン集合を比べ、新規（オープンだが旧本文にない）と close 済み（旧本文にあるが今はオープンでない）を把握する。
2. 旧本文のカテゴリー見出し・並び順・各 Issue の行・依存サブリストはそのまま引き継ぐ。
3. close 済みの Issue の行を本文から削る。その番号を依存サブリストに持っていた Issue があれば、そのサブリスト項目も消す。
4. 新規 Issue は Step 3 で分析した結果（カテゴリー・優先度・依存先）に従って、該当セクションの適切な位置に挿入する。新しいテーマなら新セクションを足す。
5. 全 Issue の行末の `(In Progress)` を、今回のオープン PR 集合に合わせて付け直す（PR が消えた Issue からは外し、新たに PR が付いた Issue には足す）。
6. 末尾の `Cycle` / `Conflict` 行も、現状に合わなくなっていれば直す。

ユーザーが明示的に「整理し直して」と言った場合だけ、引き継がず全件フル解析で作り直す。

```bash
gh issue edit <ダッシュボード番号> --body-file <一時ファイル>
```

**なかった場合。** 新規作成する。

```bash
gh issue create --title "Issue Dashboard" --body-file <一時ファイル>
```

## Step 6: 完了報告

ダッシュボードのハイライトと、更新なら差分を伝える。

```
Issue ダッシュボードを {作成 / 更新} しました！ → {Issue URL}

- 対象: オープン Issue {N} 件 / {C} カテゴリー
- すぐ着手できる（依存先なし）: #{a}, #{b}, #{c}
- 作業中（PR あり）: {番号一覧 / なければ省略}
- 多くをブロックしている要注意 Issue: #{x}
- 競合注意: {あれば番号ペア / なければ省略}
- 更新時の差分: 新規 {追加された番号} / close 済みで除外 {外れた番号}
```
