# Web Spec Crawler

既存のWebサービスを agent-browser で**しらみつぶしに**操作・調査し、到達可能な全ページと各ページの全UI操作分岐を漏れなく spec/ に構造化する。API や実装の詳細は対象外で、ユーザーから見える振る舞い・画面・分岐に集中する点は web-spec-explorer と同じ。

違いは網羅の保証方法にある。web-spec-explorer は「重要箇所に集中、低優先度は名前だけ」だが、本スキルは**カバレッジ台帳で未探索を1件残らず管理し、台帳が空になるまで止まらない**。網羅は注意深さでは達成できない。frontier（未探索キュー）を機械的にドレインし切ることで達成する。

## このスキルの3本柱

網羅と完走を成立させるのは次の3つ。順に効く。

1. **カバレッジ台帳（frontier）** — 発見した全ページ・全操作要素・全分岐を状態つきで記録する単一の真実。メインエージェントの記憶ではなくファイルが真実。台帳に未探索が1件でもあれば完了ではない。→ `references/coverage-ledger.md`
2. **クロールループ** — 台帳の未探索エントリをサブエージェント（explorer）でドレインし、見つかった新ページ・新分岐を台帳に積み増す。frontier が空になるまで反復する。
3. **完全性ゲート（auditor 再確認）** — explorer の「調べ終えた」を鵜呑みにしない。独立した auditor が同じページを開き直し、snapshot 上の全要素が報告に含まれているか裏取りする。抜けが見つかれば台帳に差し戻す。`../_shared/references/completeness-gate.md` と同じ find → adversarially verify の考え方を、ページ網羅に適用する。

## 前提条件

`agent-browser` CLI が必要。セットアップ・セッション管理・認証パターン・headed/headless の使い分け・レートリミット対策・頻出コマンドリファレンスは web-spec-explorer と共通。**重複を避けるため、ブラウザ操作のメカニクスは `../web-spec-explorer/SKILL.md` を参照すること。**本スキルはその上に「網羅エンジン」を載せる。

委譲方式（サブエージェント / OSプロセス / 親内シミュレート）と並列化の原則は `../_shared/references/subagent-policy.md` に従う。メインエージェントがオーケストレーションに徹する原則は `../_shared/references/implement-principles.md` の精神を踏襲する。

## アーキテクチャ

```
メインエージェント（オーケストレーション・台帳管理）
  ├── Phase 0: 戦略策定（スコープ・ロール・分岐定義・爆発抑制ルール・停止条件）
  ├── サブエージェント: scout（初期偵察 → 台帳 seed）
  ├── クロールループ（frontier が空になるまで反復）
  │     ├── explorer-{page}（1ページの全要素列挙＋分岐探索）× 最大3並列
  │     ├── 台帳更新（新ページ・新分岐を dedup しつつ enqueue）
  │     └── auditor-{page}（カバレッジ再確認 → 抜けを差し戻し）× 最大3並列
  └── 構造化出力（spec/）→ レビュー → クリーンアップ
```

メインエージェントはブラウザを直接操作しない。台帳を管理し、未探索をサブエージェントに割り当て、返ってきた結果を台帳に反映し、auditor で裏取りし、frontier が空になるまで回し続けるのが仕事。

## 探索フロー

### Phase 0: 戦略策定

クロールを始める前に「何を1分岐と数えるか」「組み合わせ爆発をどう抑えるか」「どのロールで巡るか」「どこで止まるか」を決める。ここが曖昧だと網羅は無限に発散するか、逆に漏れる。判断基準・値の等価クラス・dedup（状態シグネチャ）・停止条件は `references/strategy.md` に従う。

戦略はユーザーと合意する。特に次はユーザー確認必須:

- **スコープ**: 対象ドメイン、除外セクション（管理画面・課金の実行・破壊的操作など）
- **巡回ロール**: 未認証 / 一般ユーザー / 管理者など。ロールごとに見えるページ・分岐が変わるため、網羅対象のロールを確定する（ロールは網羅の乗数。決めないと漏れる）
- **安全境界**: 実データの作成・更新・削除をどこまで許すか（原則しない。`references/strategy.md` 参照）

合意した戦略は `spec/_coverage/strategy.md` に書き出し、以降の判断基準にする。

### Phase 1: 準備

メインエージェントが行う:

1. agent-browser の利用可能性を確認（`../web-spec-explorer/SKILL.md` 参照）
2. `/agent-browser-cleanup` を Skill ツールで呼び、残存プロセスをクリーンアップ
3. 認証が必要なら web-spec-explorer の認証パターンに従ってセットアップ（認証情報は spec/ に書かない）
4. `spec/` と `spec/_coverage/` ディレクトリを作成
5. カバレッジ台帳を初期化する（`references/coverage-ledger.md` のフォーマット）

### Phase 2: 初期偵察（scout サブエージェント）

1つのサブエージェントでサイト全体の入り口を把握し、台帳の初期 frontier を作る。ここはまだ網羅ではなく「種まき」。トップ・ナビ・主要セクションから到達できる URL を広く拾い、発見した URL を全て台帳に `discovered`（未調査）として登録する。

scout への指示は web-spec-explorer の scout テンプレートを流用しつつ、**「発見した全リンク先 URL を漏れなく列挙して返す」**ことを強調する（網羅の起点なので取りこぼし厳禁）。

### Phase 3: クロールループ — explorer でドレイン

台帳の `discovered` ページを explorer サブエージェントに割り当て、1ページずつ全要素を列挙させる。並列は最大3。`discovered` が4件以上あれば複数ラウンドに分ける。

explorer は1ページについて、snapshot 上の**全インタラクティブ要素**（ボタン・リンク・入力・セレクト・チェック・ラジオ・トグル・タブ・アコーディオン・メニュー・モーダルトリガー等）を1つ残らず列挙し、各要素が何をするか（遷移 / 状態変化 / 送信）を分類して返す。全要素列挙・全フォーム状態・全エラーケースの徹底手順とプロンプトテンプレートは `references/enumeration.md` に従う。

explorer の結果から、メインエージェントは台帳を更新する:

- 新たに発見したページ URL → `discovered` で enqueue（既知・dedup 済みは除く）
- 新たに発見した分岐（モーダル・状態切替・条件表示）→ 未探索分岐として enqueue
- 調べ終えたページ → `explored` に更新

dedup（同一 URL テンプレート・同一状態シグネチャの再登録防止）は `references/strategy.md` に従う。これがないとループが終わらない。

### Phase 4: 完全性ゲート — auditor で再確認

explorer の「全部調べた」を信用しない。`explored` ページを auditor サブエージェントに渡し、**同じページを開き直して** snapshot 上の全要素が explorer の報告に含まれているか裏取りさせる。これは `../_shared/references/completeness-gate.md` の find → adversarially verify をページに適用したもの。

- auditor が抜けを発見 → 該当要素・分岐を台帳に差し戻し（`discovered` / 未探索分岐に戻す）→ Phase 3 で再調査
- 抜けゼロ → ページを `verified` に更新

auditor の独立性が肝。explorer 本人は自分の省略を「些末」と正当化しがちなので、別サブエージェントにゼロベースで探させる。auditor のプロンプトは `references/enumeration.md` 参照。

### Phase 5: ループ判定

台帳を見て、`discovered` ページ・未探索分岐・未 `verified` ページが**1件でも残っていれば** Phase 3 に戻る。全ページが `verified` かつ未探索分岐ゼロになって初めて frontier ドレイン完了。

「主要ページは押さえた」で打ち切らず、台帳が空になるまで回す。安全上・スコープ上どうしても探索不能なエントリは `excluded` に明示し、理由を台帳に記録する（暗黙のスキップはしない）。停止条件の詳細は `references/strategy.md`。

### Phase 6: 構造化出力

frontier ドレイン完了後、全結果を spec/ に構造化する。網羅版は web-spec-explorer の出力に加えて、各ページの全分岐テーブル・完全な画面遷移図・カバレッジサマリ（探索済み/除外の集計）を含める。出力フォーマットは `references/output-format.md` に従う。

### Phase 7: レビューとクリーンアップ

1. spec/ とカバレッジサマリをユーザーに提示し、抜け・認識違い・追加調査箇所のフィードバックを受ける。フィードバックがあれば該当ページ・分岐を台帳に差し戻し（`discovered` へ）、Phase 3-5 を部分再実行する。
2. 全完了後、agent-browser の残存プロセスを片付ける:

   ```bash
   agent-browser close --all 2>/dev/null || true
   ```

   続けて Skill ツールで `/agent-browser-cleanup` を呼ぶ。

## メインエージェントの原則

このスキルの価値は完走にある。次を守る。

- **台帳が真実、記憶は信用しない** — 各ラウンドの開始時に台帳を読み直し、未探索エントリの実数で進捗を判断する。「だいたい調べた」という主観で判断しない。
- **explorer の完了報告を鵜呑みにしない** — 必ず auditor で裏取りしてから `verified` にする。完全性は列挙した本人の自己申告では担保されない。
- **frontier が空になるまで継続する** — 残件が多く長くなる場合は、打ち切らずユーザーに進捗（探索済み N / 残り M）を報告して継続意思を確認しながら進める。
- **探索不能は明示する** — スコープ外・安全上不可・到達不能なエントリは `excluded` にして理由を残す。暗黙のスキップは網羅の虚偽になる。
- **ブラウザは触らない** — 全ブラウザ操作はサブエージェントに委譲。メインは台帳・割り当て・裏取り・集約に徹する。
- **新発見は即 enqueue** — explorer/auditor が新ページ・新分岐を返したら、その場で台帳に積む。

## 注意事項

- **本番データに触れない**: フォーム送信は原則しない。バリデーション確認のための空送信・不正値入力は可。実データの作成・変更・削除は絶対にしない。破壊的操作の分岐は「操作はせず存在と遷移先だけ記録」する。
- **認証情報を spec/ に書かない**: auth.md には認証方式の仕様のみ。パスワード・トークンは書かない。
- **外部リンクを踏まない**: `--allowed-domains` で対象ドメインに限定する。
- **レート制限を守る**: ページ遷移ごとに sleep を入れ、並列は最大3。429/CAPTCHA が出たら間隔を広げる。
- **screenshot の保存先**: `/tmp/spec-screenshots/` 配下に整理。spec/ からは相対パスで参照。
- **網羅と爆発のバランス**: 「全分岐」は等価クラス（代表値＋境界値）と状態シグネチャ dedup で有限化する。全データ組み合わせの列挙ではない。詳細は `references/strategy.md`。
