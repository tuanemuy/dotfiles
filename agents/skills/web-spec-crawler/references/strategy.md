# 戦略策定と爆発抑制

「全UI操作分岐を漏れなく」は、何も決めずに始めると2つの失敗をする。組み合わせ爆発で無限に発散するか、定義が曖昧で取りこぼすか。この2つを同時に防ぐのがこのドキュメント。クロール開始前にここで方針を確定する。

## 何を1分岐と数えるか

網羅対象の「分岐」を明確に定義する。曖昧だと explorer ごとに粒度がブレて漏れる。本スキルでは次を**全て**1分岐として台帳に記録する。

| 分岐の種類 | 例 | 記録すること |
|-----------|-----|------------|
| ナビゲーション | リンク・遷移ボタン | 遷移先 URL（新規なら frontier に enqueue） |
| 状態変化（同一ページ内） | モーダル開閉・タブ切替・アコーディオン・ドロップダウン・トースト | トリガー要素と、開いた後に現れる新しい要素/分岐 |
| フォーム送信の結果 | 成功 / 各バリデーションエラー / サーバエラー | 入力条件ごとの遷移先・表示変化 |
| 条件付き表示 | 空状態 / データあり / ローディング / 権限による出し分け | 条件と、それぞれで現れる要素/分岐 |
| 認証・認可の境界 | 未ログインでアクセス → リダイレクト | アクセス制御の結果 |

**状態変化を開いたら、その中身も探索対象**。モーダルを開いたらモーダル内の全要素を列挙し、さらに分岐があれば台帳に積む。「モーダルがある」で終わらせない。これが漏れの最大の原因。

## 爆発を抑える3つの等価化

全分岐を有限にするための等価クラス化。これをしないと組み合わせが指数的に増える。

### 1. 入力値は等価クラスで代表させる

各入力項目について、全値を試すのではなく代表値で分岐を代表させる:

- **代表値（正常系）**: 妥当な値を1つ
- **境界値**: 最小長・最大長・0・上限など、振る舞いが変わる境目
- **不正値**: 形式違反・空・型違反など、バリデーションを発火させる代表

フィールドが3つあっても「3フィールド × 3クラス」の組み合わせ9通りを全部試すのではない。**1フィールドずつ不正にして他は正常**（One-At-a-Time）でバリデーション分岐を網羅し、全正常で成功パスを1本通す。フィールド間に依存（A を選ぶと B が必須化など）がある場合のみ、その依存が生む分岐を追加で記録する。

### 2. 繰り返し構造は代表1件＋パターン

リスト・グリッド・テーブルの行など、同型要素が N 個並ぶものは、**代表1件を探索し、件数とパターンを記録**する。全 N 件は探索しない。ただし「種類が違う行」（例: 通常アイテムと sponsored アイテム）は別パターンとして各1件探索する。

### 3. パラメータ URL はテンプレート化

`/items/1`, `/items/2`, ... は1つのテンプレート `/items/:id` に畳む。台帳には URL テンプレート単位で登録し、代表インスタンス1件を探索する。クエリパラメータによる表示違い（`?tab=...`, `?sort=...`）は表示が変わるものだけ別分岐として記録する。

## 状態シグネチャと dedup

ループが終わるための最重要ルール。**既に見た状態に戻る分岐を再 enqueue しない。**

各到達状態に**シグネチャ**を与える: `URLテンプレート + 有意な状態差分`。例: `/dashboard`（通常）と `/dashboard`（設定モーダル開）は別シグネチャ。`/items/1` と `/items/2` は同一シグネチャ（テンプレート同じ・状態差なし）。

分岐を辿った結果が**既知シグネチャ**なら、台帳に「back-edge（既知へ戻る）」として遷移だけ記録し、探索対象としては enqueue しない。新規シグネチャなら未探索として enqueue する。

これにより、相互リンクするページ群や開閉トグルでループが無限化するのを防ぐ。

## ロールという乗数

認証・認可があるサービスでは、**ロールごとに見えるページ・分岐が変わる**。ロールは網羅の乗数なので、Phase 0 で巡回対象ロールを確定する。

- 典型: 未認証 / 一般ユーザー / 管理者 / （あれば）ゲスト・有料プラン等
- 台帳のページ・分岐エントリには「どのロールで確認したか」を持たせる
- 同一 URL でもロールによって出る要素が違えば、ロールごとに別分岐として記録する
- 全ロールを巡るとコストが乗数で増えるため、ユーザーと「どのロールまで網羅するか」を合意する

ロールを決めずに1ロールだけ巡ると「網羅した」が嘘になる。必ず明示的に範囲を決める。

## 安全境界

網羅は安全より優先しない。本番データを壊してまで全分岐を踏まない。

- 実データの**作成・更新・削除は原則しない**。バリデーション確認の空送信・不正値入力は可
- 破壊的操作（削除・送金・公開など）の分岐は、**操作を実行せず**「トリガー要素の存在・確認ダイアログの内容・遷移先の想定」だけを記録する。「未実行のため遷移先は推定」と台帳に明示する
- 課金・外部送信・メール送信など副作用のある操作も同様に「存在記録のみ」
- 安全上踏めない分岐は台帳で `excluded`（理由: 破壊的/副作用あり）にする。黙ってスキップしない

## 停止条件（完走の定義）

frontier ドレイン完了 = 次が**全て**満たされた状態:

1. 台帳に `discovered`（未調査ページ）が0件
2. 台帳に未探索の分岐が0件
3. 全ページが `verified`（auditor 裏取り済み）
4. 探索しなかったエントリは全て `excluded` で理由が記録されている

「主要部分を押さえた」「だいたい網羅した」は停止条件ではない。上記4つを台帳の実数で確認する。残件が多くて長時間化する場合は、打ち切るのではなく、進捗（探索済み/残り）をユーザーに報告して継続意思を確認しながら進める。スコープを狭める判断はユーザーがする。
