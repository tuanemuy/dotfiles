# 全要素列挙と再確認のサブエージェント

explorer（全要素を列挙する）と auditor（列挙の抜けを裏取りする）のプロンプトテンプレート。この2つの組が `../_shared/references/completeness-gate.md` の find → adversarially verify を「ページの網羅」に適用したもの。explorer の自己申告を auditor が独立に検証することで、「調べたつもり」の漏れを潰す。

委譲方式・並列化・委譲時のコンテキストは `../_shared/references/subagent-policy.md` に従う。各サブエージェントには独立したセッション名を渡す（`explorer-{page}`, `auditor-{page}`）。

## explorer の原則

1ページについて、snapshot 上の**全インタラクティブ要素を1つ残らず**列挙する。「主要な」「目立つ」要素ではなく全部。網羅はここの取りこぼしで崩れる。

各要素を次のいずれかに分類して報告する:

- **ナビゲーション** → 遷移先 URL を記録（クリックして遷移先 URL を確認）
- **状態変化** → クリックして開いた後に現れる新要素・新分岐を記録（モーダル・タブ・アコーディオン等）
- **フォーム送信** → 入力条件ごとの結果を記録（`strategy.md` の等価クラスに従う）

状態変化を開いたら、その中身も列挙する。「モーダルがある」で止めず、モーダル内の全要素まで降りる。

## やること

1. ページを開き、snapshot を取得（--max-output 8000）。大きければセクション別に分割取得する
2. snapshot 上の【全インタラクティブ要素】を列挙する。取りこぼし厳禁:
   ボタン / リンク / テキスト入力 / テキストエリア / セレクト / チェックボックス /
   ラジオ / トグル・スイッチ / スライダー / 日付ピッカー / ファイル選択 /
   タブ / アコーディオン / ドロップダウンメニュー / モーダル等を開くトリガー /
   ページネーション / 並び替え・フィルタ操作 / コンテキストメニュー
3. 各要素を分類し、振る舞いを確認する:
   - ナビゲーション系: クリックして遷移先 URL を取得（戻る で復帰）
   - 状態変化系: クリックして開き、現れた新要素・新分岐を列挙。開いた先にさらに
     操作要素があれば、それも列挙する（モーダル内のボタン等まで降りる）
   - フォーム系: 下記「フォームの分岐網羅」に従う
4. 条件付き表示を確認する: 空状態 / データあり / ローディング / 権限による出し分けなど、
   表示が変わる条件と、それぞれで現れる要素を記録
5. 破壊的操作（削除・送信・公開・課金など）は【実行しない】。トリガー要素の存在・
   確認ダイアログの内容・想定遷移先のみ記録し「未実行・推定」と明記する
6. screenshot を /tmp/spec-screenshots/{page_id}/ に保存

ページ遷移ごとに sleep 1 を入れること。実データの作成・更新・削除はしないこと。

## フォームの分岐網羅（strategy.md の等価クラス）

- 各フィールドを1つずつ不正値にし他は正常にして送信し、出るバリデーションを記録（One-At-a-Time）
- 境界値（最小/最大長・0・上限）で振る舞いが変わるものを記録
- 全フィールド正常での送信結果（成功遷移先）を記録。ただし実データを作る送信は避け、
  避けられない場合は「未送信・推定」と明記
- フィールド間依存（A選択で B が必須化など）があれば、その依存が生む分岐も記録

## auditor の原則

explorer の「全部調べた」を信用しない。**同じページを開き直し**、snapshot 上の全要素が explorer の報告に含まれているか、ゼロベースで照合する。explorer 本人は自分の省略を「些末」と正当化しがちなので、別サブエージェントに独立して探させる。

### auditor プロンプトテンプレート

```
あなたはページ調査の網羅性を検証する独立した監査エージェントです。
調査した本人ではないので、忖度せず「列挙から漏れた操作要素・分岐」を探してください。

対象ページ: {url}（確認ロール: {role}）
セッション名: auditor-{page_id}
認証: {auth_info}
allowed-domains: {domains}
explorer の報告: {explorer の全操作要素一覧・分岐一覧を貼る}

## やること

1. ページを開き、snapshot を【自分で】取得する（explorer の報告を見る前提にしない）
2. snapshot 上の全インタラクティブ要素を自分で数え上げる
3. explorer の報告と突き合わせ、次を検出する:
   - 報告に【無い】操作要素（列挙漏れ）
   - 状態変化要素なのに「開いた中身」が未確認のもの（モーダルがある、で止まっている等）
   - フォームでバリデーション分岐が一部しか確認されていないもの
   - 条件付き表示（空状態・権限差など）の確認漏れ
4. 破壊的操作は実行しないこと。存在記録の妥当性だけ確認する

## 重要

- snapshot に映る操作要素が報告に全て対応づくか、要素単位で確認する
- 「だいたい網羅されている」では合格にしない。1件でも漏れがあれば不合格として列挙する
- 判断に迷うもの（動的生成・条件依存で出ない要素など）は「要確認」として挙げる

## 出力

- 漏れが1件もなければ「網羅 OK」と明記する
- あれば各項目について: 漏れた要素/分岐, snapshot 上の位置（ref やラベル）, 種別
  （列挙漏れ / 中身未確認 / フォーム分岐不足 / 条件表示漏れ / 要確認）, 何を追加調査すべきか
```

## メインエージェントの差し戻し

auditor が漏れを返したら、メインエージェントは該当ページ/分岐を台帳で `discovered`（または未探索分岐）に戻し、**漏れリストを具体的に明示して** explorer に再割り当てする（「全部見直して」ではなく「この要素のこの中身が未確認、ここを調べよ」）。再調査後、再度 auditor に渡す。漏れゼロになって初めて `verified`。3 ラウンドで収束しない場合は残りをユーザーに報告して判断を仰ぐ。
