# カバレッジ台帳（frontier）

網羅・完走の心臓部。発見した全ページ・全分岐を状態つきで記録する単一の真実。メインエージェントの記憶ではなく、この台帳ファイルが進捗の判断基準になる。台帳に未探索が1件でもあれば「完了」ではない、という不変条件をここで担保する。

## なぜファイルなのか

網羅は長丁場になる。会話のコンテキストは流れて忘れる。記憶を頼ると「だいたい調べた」で止まる。ファイルに未探索エントリを実数で残すことで、メインエージェントは毎ラウンド客観的に「あと何件残っているか」を判断できる。これが途中放棄を防ぐ唯一の仕組み。

各ラウンドの開始時に必ず台帳を Read し直し、`discovered`/未探索分岐/未 `verified` の実数で継続判断する。台帳を見ずに進めない。

## 配置

```
spec/_coverage/
├── strategy.md      # Phase 0 で合意した戦略（スコープ・ロール・安全境界・停止条件）
├── pages.md         # ページ frontier（全ページの状態台帳）
└── branches.md      # 分岐 frontier（ページ内の状態変化・条件表示・フォーム分岐の台帳）
```

ページ数が多い場合 `pages.md` が肥大化するため、セクション別に `spec/_coverage/pages/{section}.md` に分割してよい。形式は同じ。

## ページ frontier（pages.md）

到達可能な全ページ（URL テンプレート単位）を1行ずつ管理する。

```markdown
# ページ frontier

| ID | URLテンプレート | 状態 | ロール | 発見元 | シグネチャ | 備考 |
|----|---------------|------|--------|--------|-----------|------|
| P01 | / | verified | 未認証 | (起点) | top | - |
| P02 | /login | verified | 未認証 | P01 ヘッダー | login | - |
| P03 | /dashboard | explored | 一般 | P02 送信後 | dashboard | auditor 待ち |
| P04 | /items/:id | discovered | 一般 | P03 一覧リンク | - | 代表 /items/42 |
| P05 | /admin/* | excluded | - | P03 メニュー | - | スコープ外（管理画面） |
```

### 状態（ページ）

| 状態 | 意味 | 次のアクション |
|------|------|--------------|
| `discovered` | 発見したが未調査 | explorer に割り当てる（Phase 3） |
| `exploring` | explorer 調査中 | 完了待ち |
| `explored` | 全要素列挙が一旦完了 | auditor で裏取り（Phase 4） |
| `verified` | auditor が抜けゼロを確認 | 完了。何もしない |
| `excluded` | スコープ外・安全上不可・到達不能 | 理由を備考に明記。探索しない |

`verified` 以外が残っていればループ継続。`excluded` には必ず理由を書く（黙殺禁止）。

## 分岐 frontier（branches.md）

ページ内の状態変化・条件表示・フォーム分岐を管理する。ナビゲーション（別ページへ）は pages.md 側で管理するので、ここは**同一ページ内に閉じる分岐**と**フォーム送信結果の分岐**を担当する。

```markdown
# 分岐 frontier

| ID | 所属ページ | 分岐の種類 | トリガー | 状態 | 到達シグネチャ | 結果 |
|----|-----------|----------|---------|------|--------------|------|
| B01 | P03 | モーダル | 「設定」ボタン | verified | dashboard+settings-modal | 設定モーダル展開（B05-B07へ） |
| B02 | P03 | タブ切替 | 「アーカイブ」タブ | explored | dashboard?tab=archive | アーカイブ一覧表示 |
| B03 | P02 | フォーム結果 | 空送信 | verified | login+error | 「必須です」表示 |
| B04 | P02 | フォーム結果 | 不正メール | discovered | - | （未確認） |
| B05 | P03 | モーダル内 | 設定モーダルの「保存」 | discovered | - | 破壊的=存在記録のみ予定 |
```

### 状態（分岐）

ページと同じく `discovered → exploring → explored → verified`、および `excluded`。**分岐を開いた結果さらに分岐があれば、それを新しい行として enqueue する**（B01 → B05-B07 のように）。ここで連鎖を止めると「モーダルはあるが中身は不明」という漏れになる。

## dedup（再 enqueue 防止）— ループを終わらせる規則

explorer/auditor が新ページ・新分岐を報告したら、台帳に積む前に必ず重複判定する。これがないと相互リンクや開閉トグルで frontier が無限に増えて完走しない。

判定手順:

1. **URL テンプレート化**: `/items/1` → `/items/:id`。既存テンプレートと一致すれば新規ページではない
2. **シグネチャ照合**: 到達状態のシグネチャ（`URLテンプレート + 有意な状態差分`）が台帳の既存エントリと一致すれば、新規探索対象にしない
3. **一致した場合**: 遷移だけを記録（back-edge）し、状態は変えない。新しい行は作らない
4. **不一致の場合**: 新規エントリとして `discovered` で enqueue

シグネチャの作り方は `strategy.md` の「状態シグネチャと dedup」に従う。

## ループ運用（メインエージェントの毎ラウンド手順）

1. 台帳を Read する
2. `discovered` ページを最大3件取り出し、explorer に割り当てる（Phase 3）
3. explorer の結果で台帳更新: 対象を `explored` に、新発見を dedup して enqueue
4. `explored` ページを auditor に渡す（Phase 4）
5. auditor の結果で台帳更新: 抜けゼロなら `verified`、抜けありなら該当を `discovered`/未探索分岐に差し戻し
6. 台帳に `verified` 以外（`discovered`/`exploring`/`explored`/未探索分岐）が残っていれば 1 に戻る
7. 残りが `verified` と `excluded` だけになったら frontier ドレイン完了 → Phase 6（構造化出力）

## カバレッジサマリ

完了時、台帳から集計を作り `spec/_coverage/summary.md`（または overview に節を設ける）に出す。網羅したことを数字で示す成果物。

```markdown
# カバレッジサマリ

- 調査日: {YYYY-MM-DD}
- 巡回ロール: 未認証 / 一般ユーザー / 管理者
- ページ: 全 42 / verified 39 / excluded 3（管理画面2, 破壊的1）
- 分岐: 全 156 / verified 151 / excluded 5（破壊的操作 5）
- 未探索（discovered 残り）: 0  ← 0 でなければ未完走

## excluded 一覧（探索しなかった箇所と理由）

| ID | 対象 | 理由 |
|----|------|------|
| P05 | /admin/* | スコープ外（ユーザー合意） |
| B12 | アカウント削除の実行 | 破壊的操作・存在のみ記録 |
```

`discovered` 残りが 0 であることが完走の証明。0 でなければ完了と報告してはいけない。
