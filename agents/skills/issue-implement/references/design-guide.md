# Phase 1.5: デザイン作成ガイド（条件付き）

このフェーズは、Issue がユーザー向けの UI 画面を新設・変更するときだけ実行する。実装に入る前に、新しい画面の見た目を HTML モックとして固めておくことで、Phase 2 の実装サブエージェントが「何を作るか」を視覚的に把握でき、手戻りを減らせる。

design-flow スキルの後半（トークン → デザイン作成 → レビュー）を踏襲するが、**ドラフト提案フェーズは行わない**。issue-implement は既存実装のあるプロジェクトで動くので、デザインの方向性はすでにアプリが体現している。ゼロから方向性を提案するのではなく、**今出荷されている見た目に馴染ませる**のがこのフェーズのゴール。

HTML 作成・レビューの細かい手順（レスポンシブ条件、ビューポート、レビューパイプライン等）は design-flow の参照ファイルと共通なので、そちらを正とする。この guide では issue-implement 固有の差分（gate 判定・既存コードからの抽出・実装への引き渡し）だけを書く。

## Step 0: gate 判定（デザインが必要か）

`.issue/{Issue番号}/plan.md` を読み、このIssueが**ユーザーに見える UI 画面を新設・変更するか**を判断する。

デザインフェーズに入るのは次のような場合:

- 新しい画面・ページ・ルートを追加する
- 既存画面のレイアウトや構成を実質的に変える
- 視覚的なボリュームのある新規コンポーネント（カード一覧、フォーム、ダッシュボード的なもの）を作る

入らない（スキップする）のは次のような場合:

- バックエンド・API・ロジックのみの変更
- 文言修正、軽微なバグ修正、リファクタで見た目が変わらないもの
- 既存コンポーネントの内部修正で、レイアウト・配置・スタイルが変わらないもの

判断に迷うグレーな変更（小さなUI調整など）は、**モックを作るほどではない** と考えてスキップ側に倒す。デザインフェーズは「新しい画面の絵を描いて初めて意味がある」もの。既存の見た目をなぞるだけの小変更に重い工程をかけない。

判定結果は必ず一言ログに残す（例: 「plan.md に新規画面〇〇の追加あり → デザインフェーズ実行」「API 変更のみ・UI 画面の新設なし → デザインフェーズをスキップ」）。スキップする場合はそのまま Phase 2 へ進む。

実行する場合は、対象となる画面・コンポーネントの一覧を plan.md から拾い出してから Step 1 へ進む。

## Step 1: デザイン言語の確立（既存コードから抽出）

design-flow ではユーザーが選んだドラフトからトークンを起こすが、ここでは**既存の実装から現行のデザイン言語を抽出する**。新規画面が「後から付け足した別物」に見えないよう、出荷済みの見た目に揃えるのが目的。

### 抽出元（優先順）

1. **`spec/design/tokens.md` と `spec/design/index.md`** — 過去に design-flow を回していれば、確立されたトークン・方針がある。あれば読み込んで土台にする。
2. **実際のフロントエンド実装** — これが最終的な正。spec ドキュメントが古くて実装とズレている場合は、**実装側を優先**する。Issue が触る画面の近くにある既存コードを読み、現行の見た目を拾う:
   - デザイントークン・テーマ定義ファイル（Tailwind config、CSS 変数、theme.ts、design tokens など）
   - 共通 UI コンポーネント（Button, Card, Input, Nav など）の実装とスタイル
   - 既存画面のレイアウト・余白・タイポグラフィの取り方

抽出するもの: カラー、タイポグラフィ（フォント・サイズスケール・ウェイト）、スペーシング、border-radius、シャドウ、そして **Button / Card / Form / Nav といった既存コンポーネントの見た目のパターン**。新規画面はこれらを再利用した姿で描く。

### tokens.md の扱い

- `spec/design/tokens.md` が既にあれば、それを正としつつ、実装とズレている箇所だけ実装に合わせて補正する。
- 無ければ、抽出した内容を `spec/design/tokens.md` として書き起こす（モックと今後の作業が同じ参照を共有できるように）。トークンの粒度・命名規則は design-flow の `../../design-flow/references/phase2-tokens.md` の「CSS カスタムプロパティの命名規則」に従う。**新しい方向性を発明しない** — あくまで既存コードの言語化。

ドラフト提案・ユーザーへの方向性確認は行わない。方向性は「既存に馴染ませる」で確定している。

## Step 2: デザイン作成

design-flow の `../../design-flow/references/phase3-production.md` の手順に従い、Step 0 で特定した各画面の HTML デザインを `/frontend-design` で作成する。レスポンシブ条件・ビューポート・トークン適用・自己チェックはすべてそちらを正とする。

出力先: `spec/design/pages/{画面名}.html`

issue-implement 固有の差分として、`/frontend-design` に渡すコンテキストへ次を加える:

- **このIssueの文脈**: `.issue/{Issue番号}/plan.md` から、この画面で実現したいこと・表示する情報・操作
- **既存コンポーネントへの準拠**: Step 1 で読んだ既存の Button / Card / Form / Nav 等のパターンを再利用し、**新規画面が既存アプリの一部として自然に見える**ようにすること。独自の新しいUIパターンを持ち込まない
- **デザイントークン**: `spec/design/tokens.md`（または Step 1 で抽出した内容）

既存画面のスクリーンショットがあると `/frontend-design` の精度が上がる。Issue が変更する近接画面が既に動くなら、agent-browser で開いて参考スクリーンショットを撮り、コンテキストに添えてもよい（任意）。

## Step 3: レビュー

design-flow の `../../design-flow/references/phase4-review.md` のレビューパイプライン（agent-browser での視覚確認 → `/critique` → `/polish` → `/audit`）に従う。

レビュー記録は `spec/design/review/${連番}.md` に残す。観点に加えて issue-implement 固有のチェックを一つ足す:

- **既存との一貫性**: 新規画面が既存画面・既存コンポーネントと並べて違和感がないか。色・余白・コンポーネントの見た目が出荷済みのものと揃っているか

問題が収束したら完了。design-flow の終了条件（/audit まで通過、または軽微な指摘のみ）と同じ。

## 完了後の片付けと引き渡し

1. agent-browser の残存プロセスを片付ける。各セッションは使用後に `close` し、最後に取りこぼし対策で一括クローズ:

   ```bash
   agent-browser close --all 2>/dev/null || true
   ```

   その後 Skill ツールで `/agent-browser-cleanup` を呼び出す。

2. 作成した成果物を一言サマリーしてから Phase 2 へ進む。

### Phase 2 への引き渡し

このフェーズの成果物 `spec/design/pages/*.html` は、Phase 2 の実装サブエージェントにとっての**視覚的・構造的な仕様**になる。実装ガイド（`implementation-guide.md` Step 2）の手順で、実装サブエージェントにこのモックを参照させ、既存コンポーネントを使った実フレームワークのコードへ落とし込ませる。モックはあくまで完成イメージであり、そのまま貼り付けるものではない点を実装側に伝える。

成果物（`spec/design/` 配下のファイル）は Phase 2 のコミットに含めて、実装と一緒に PR へ載せる。
