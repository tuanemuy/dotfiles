# Phase 3: レビュー・修正ガイド

Phase 2 で作成した PR に対して、セルフレビュー・修正を繰り返し品質を担保する。レビューループの共通構造（出力フォーマット・修正ループ）は `../../_shared/references/review-loop.md` を参照。ただし**完了条件は本ガイドの Step 7 で上書き**する（共通側の「2連続クリーン」ではなく「直すべき指摘ゼロのラウンドで完了」を採用）。

トークン効率の原則: レビューの本文はサブエージェントが**直接ファイルに書き**、メインには件数と指摘の一行リストだけを返させる。メインのコンテキストにレビュー全文を往復させない。

## 出力先

すべての成果物は `.issue/{Issue番号}/` 配下に保存する。

- レビューファイル: `.issue/{Issue番号}/review/review-{NNN}-{レイヤー}.md`（各レビュアーが直接書く）
- ラウンドサマリー: `.issue/{Issue番号}/review/review-NNN.md`（メインが件数・Verdict だけ書く）
- ADR: `.issue/{Issue番号}/adr.md`（既存ファイルに追記）

## Step 1: PR の取得と分析

```bash
gh pr view <PR番号> --json title,body,baseRefName,headRefName,files
gh pr diff <PR番号>
```

差分の内容から、レビューすべきレイヤーを判定する。

## Step 2: レイヤーの判定

まず `.issue/{Issue番号}/plan.md` の **複雑度** メタデータを読む。

- **複雑度: 小規模** の場合 → レイヤー分割せず **「General Review」1本**で実施する（変更が1〜2ファイル程度に閉じているため、多角レビューのコストが見合わない）。Step 3 のサブエージェントは1つだけ起動し、変更全体を一気に見させる。
- **複雑度: 中〜大規模** の場合 → 以下の通りレイヤーを判定する。

変更されたファイルのパスと内容から、レビューすべきレイヤーを自動判定する。以下はよくあるレイヤーの例だが、プロジェクトの構造に合わせて柔軟に決める。

**クリーンアーキテクチャ系:**
- **Domain** — エンティティ、値オブジェクト、ドメインルール
- **Use Case** — アプリケーションロジック、ユースケースの責務
- **Interface Adapter** — コントローラー、プレゼンター、ゲートウェイ
- **Infrastructure** — DB、外部API、フレームワーク固有コード

**横断的関心事:**
- **Test** — テストの網羅性、テスト設計、モック戦略
- **Frontend** — コンポーネント設計、状態管理、UX
- **Security** — 認証・認可、入力バリデーション、機密情報
- **Performance** — N+1、不要な計算、メモリ効率

変更がないレイヤーはスキップしてOK。**変更規模に応じて1〜5レイヤー**に絞る（小さい変更は1レイヤーで十分、複数レイヤーをまたぐ変更は最大5つまで）。

## Step 3: サブエージェントでレビュー

判定したレイヤーごとに、サブエージェントを起動する（委譲方式は `../../_shared/references/subagent-policy.md` に従う）。

- レイヤーが**1つの場合**（小規模 Issue 等）→ サブエージェント1つを単発実行する。
- レイヤーが**2つ以上の場合** → サブエージェントを並列に起動する。

各サブエージェントには以下を渡す:

```text
あなたは「{レイヤー名}」の観点でPRをレビューする専門家です。

## レビュー対象
- PR: {PR番号}
- リポジトリ: {リポジトリパス}
- 実装計画: .issue/{Issue番号}/plan.md

## やること
1. `gh pr diff {PR番号}` で差分を取得
2. `.issue/{Issue番号}/plan.md` を読み、計画との整合性を確認する。特に「受け入れ基準」の表を見て、担当レイヤーに関わる基準が実装で満たされているかを検証する
3. 必要に応じて関連ファイルを読む
4. 「{レイヤー名}」の観点で厳しくレビュー
5. レビュー結果を `.issue/{Issue番号}/review/review-{NNN}-{レイヤー}.md` に**直接書き込む**

## レビューファイルのフォーマット
以下のフォーマットでファイルに書くこと:

### {レイヤー名}

#### Blockers
- **[B-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜこれが問題か
  - 提案: どう直すべきか

#### Warnings
- **[W-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ気になるか
  - 提案: 改善案

#### Notes
- **[N-001]** 良い点や参考情報

BlockerがなければBlockersセクションに「なし」と書く。
厳しく見ること。妥協しない。でも的外れな指摘はしない。

このフォーマットは下限であって上限ではない。必須なのは ID 付きの指摘（B-/W-/N-、場所・理由・提案）だけ。
分析の過程・前提・コード引用・代替案の比較など、伝えるべきことがあれば自由にセクションを足して深く書いてよい。

## 返答（レビュー全文は返さない）
ファイルへの書き込みが済んだら、返答には**サマリーだけ**を返す:
- Blockers: {数} / Warnings: {数} / Notes: {数}
- 各指摘の一行リスト: `[B-001] {タイトル} — {ファイルパス:行番号}`
```

## Step 4: ラウンドサマリーの作成

すべてのサブエージェントが完了したら、返ってきたサマリーから連番のラウンドサマリーを作る。レビュー全文を読み直したり転記したりしない。

保存先: `.issue/{Issue番号}/review/review-NNN.md`（NNNは001から始まる3桁の連番。レイヤー別ファイル `review-NNN-{レイヤー}.md` と同じ連番を使う）

```markdown
# PR Review #NNN — {PRタイトル}

**PR:** #{PR番号}
**Date:** {日付}
**Round:** {N}回目

## Summary

- Blockers: {数}
- Warnings: {数}
- Notes: {数}
- Verdict: **BLOCKED** / **APPROVED**

## レイヤー別ファイル

- {レイヤー名}: review-NNN-{レイヤー}.md（B: {数} / W: {数}）

## 指摘一覧

- [B-001] {タイトル} — `ファイルパス:行番号`（{レイヤー}）
- [W-001] {タイトル} — `ファイルパス:行番号`（{レイヤー}）
```

## Step 5: 指摘の修正

基本方針: **Blocker・Warning問わず、できる限りすべて修正する。** 後回しは最終手段。

メインエージェントはここでもオーケストレーションに徹する。**「どの指摘を直すか／後回しにするか」という判断はメインが担い、実際のコード編集はサブエージェントに委譲する**（委譲方式は `../../_shared/references/subagent-policy.md` に従う）。

ラウンドサマリーに指摘がある場合:

1. メインが**指摘一覧（一行リスト）**を見て、すべての Blocker・Warning を「このPRで直す」「後回し」に仕分ける。詳細が必要な指摘だけレイヤー別ファイルの該当箇所を確認する
   - 後回し判断の前に `../SKILL.md` Phase 4 のスコープ基準を確認する。同じファイル・機能の中で完結する指摘や軽微な指摘は、ほぼ確実にこのPRで直す（軽微なほど修正コストが低い）
   - それでもこのPRで対応できないと判断した場合のみ後回しにする（メインが処理する）:
     - `.issue/{Issue番号}/adr.md` に判断理由を記録する
     - `gh issue create` で別Issueを起票し、ADRへのリンクを含める
     - レビューファイルの該当指摘に `→ 別Issue #{新Issue番号} で対応` と追記する
2. 「このPRで直す」と仕分けた指摘を、サブエージェントに委譲して修正させる。**Blocker・Warning とも残さず潰す**
   - 指摘がレビューレイヤー（Domain / Use Case / Test 等）をまたぐ場合は、レイヤーごとに分けて**並列**で起動してよい。1〜2ファイルに閉じるなら1つにまとめて単発でよい
   - 各サブエージェントには次を渡す:
     - 対象のレイヤー別レビューファイル `.issue/{Issue番号}/review/review-NNN-{レイヤー}.md`（該当する指摘ID）。指摘の詳細はサブエージェントがファイルから読む — メインが転記しない
     - `.issue/{Issue番号}/plan.md`
     - 「指摘の提案に従って修正し、CLAUDE.md 記載の typecheck・lint を通すこと。修正対象外へスコープを広げないこと」
3. サブエージェント完了後、**メインが typecheck + lint を実行して整合性を確認する**。エラーが残っていればサブエージェントを再起動して修正させる
4. 修正が終わったら、Step 3に戻って再レビュー（次の連番ファイルを作る）

## Step 6: ADR への追記

レビュー中に重要な設計判断が見つかった場合、`.issue/{Issue番号}/adr.md` に追記する。

Phase 1（計画時）で既に adr.md が存在する場合は、既存の連番の続きから追記する。なければ新規作成する。

各決定ごとに以下の形式で記載:

```markdown
## ADR-{NNN}: {タイトル}

### Status
Proposed

### Context
{なぜこの判断が必要になったか}

### Decision
{何を決めたか}

### Consequences
{この判断の結果として何が起きるか}
```

ADRは本当に重要な設計判断のみ。些細な実装詳細はADRにしない。

## Step 7: 完了判定

完了条件は「**そのラウンドで『このPRで直す』と仕分けた指摘がゼロ**」（= Blocker 0件 かつ 修正対象の Warning 0件）。

- 各ラウンドで Blocker・Warning を全件仕分けする（Step 5 の基準）。**Blocker は必ず直す。Warning も、的外れ・スコープ外として見送り記録したもの以外はすべて直す**
- 修正した指摘が1件でもあれば、Step 3 に戻ってフル再レビュー（次の連番）。各ラウンドは毎回ゼロベースのフルレビューでよい — 同じ箇所に複数回・複数の目を通すこと自体に検出価値がある
- 「Warning 0件」を文字通り待つのではなく、**見送り判断（ADR・別Issue・レビューファイルへの記録）が済んだ Warning は完了を妨げない**。フレッシュなレビュアーが毎回新しく出す主観的な Warning は、仕分けで吸収して収束させる

**最大ラウンド数: 10回。** 10ラウンドに達しても収束しない場合は強制終了し、残っている指摘を最終ラウンドサマリーにまとめてユーザーに判断を委ねる。

## Step 8: PR を Ready for review に切り替え

APPROVED になったら Ready for review に切り替える。

```bash
gh pr ready <PR番号>
```

APPROVED に至らず強制終了した場合は Draft のまま残し、ユーザーに判断を委ねる。

## 重要な注意点

共通原則（厳しくレビュー・確信あるものだけ修正・毎ラウンド新規作成）は `../../_shared/references/review-loop.md` に従う。固有の点:

- メインはオーケストレーションに徹する。レビュー（Step 3）も修正（Step 5）も実作業はサブエージェントに委譲し、メインは仕分け・品質ゲート・ADR/起票の判断に集中する
- レビュー全文・修正内容はメインのコンテキストに乗せない。サブエージェントがファイルに直接書き、メインはサマリー（件数と一行リスト）だけ扱う
- 完了条件は共通の「2連続クリーン」ではなく Step 7 の「直すべき指摘ゼロのラウンドで完了（見送り記録済みの Warning は妨げない）」を採用する
- plan.md との整合性もレビュー観点に含める
