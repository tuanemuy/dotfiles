# Spec Sync — 設計ドキュメントと実装の同期

spec/ の設計ドキュメントと実装コードを照合し、乖離を検出して解消する。

## なぜこのスキルが必要か

implement スキルファミリーは「spec に忠実に実装する」を原則としているが、実際の開発では複数回の Issue 対応やレビュー修正が積み重なり、実装が設計から徐々に逸れていく。個々の変更は小さくても、蓄積すると spec と実装の乖離が大きくなる。この乖離を放置すると、次に spec を参照したときに誤った情報をもとに作業してしまう。

implement-audit は「コードに TODO や未実装がないか」を検証するスキル。spec-sync は逆方向で、「spec が実装の現状を正しく反映しているか」を検証する。両者は補完関係にある。

## 乖離の3分類

照合で検出された乖離は以下の3つに分類する。分類によって対応が異なる。

| 分類 | 意味 | 対応 |
|------|------|------|
| **設計判断** | 実装中に意図的に設計を変更した（より良い方法が見つかった、制約が判明した等） | spec を更新 + ADR 記録 |
| **実装誤り** | コードが spec と異なるが、spec 側が正しい | Issue 起票 |
| **実装漏れ** | spec に定義されているが実装されていない | Issue 起票 |

分類に迷う場合はユーザーに確認する。このスキルは spec の更新が主目的であり、コードの修正は行わない。実装誤り・漏れは Issue として起票し、issue-implement ワークフローでの対応につなげる。

## Workflow

```
Phase 1: 準備
  プロジェクトコンテキスト把握 → spec/ 全体の照合計画作成

Phase 2: プロジェクト全体の照合
  レイヤー×ドメインごとにサブエージェントで並列スキャン → 乖離の検出と分類

Phase 3: spec 更新 + Issue 起票
  設計判断 → spec 反映 + ADR 記録
  実装誤り・漏れ → Issue 起票

Phase 4: 結果報告
```

---

## Phase 1: 準備

### プロジェクトコンテキストの把握

CLAUDE.md と README.md を読み、以下を確認する:
- ディレクトリ構成（各レイヤーの配置先）
- 命名規約（spec のドメイン名と実装のモジュール名の対応）

### 照合計画の作成

spec/ 内の全ファイルを走査し、レイヤー×ドメインの照合対象リストを作成する。存在するレイヤーのみ対象にする（例: フロントエンドがないプロジェクトでは spec/pages/ をスキップ）。

```
## 照合計画

### ドメイン層
- [ ] {domain名}: spec/domains/{name}.md ↔ {実装パス}

### アダプター層
- [ ] {adapter名}: spec/database/{name}.md ↔ {実装パス}

### ユースケース層
- [ ] {usecase名}: spec/usecases/{name}.md ↔ {実装パス}

### フロントエンド層
- [ ] {page名}: spec/pages/{name}.md ↔ {実装パス}

### テスト層
- [ ] {testcase名}: spec/testcases/{name}.md ↔ {実装パス}
```

---

## Phase 2: プロジェクト全体の照合

`references/comparison-guide.md` を読み、その手順に従ってレイヤーごとに照合する。

照合計画の全項目について、サブエージェントを起動して並列に照合する。レイヤーごとに1つのサブエージェントを起動し、そのレイヤーの全ドメイン/全ファイルをまとめて照合させる。レイヤー間は独立なので全レイヤーを並列実行する。

各サブエージェントは以下を出力する:
- 乖離の一覧（場所・内容・分類）
- 乖離がない場合は「一致」と報告

サブエージェントの結果を統合し、分類ごとに整理する。分類の判断基準は `references/comparison-guide.md` の「推定分類の判断基準」に従う。

---

## Phase 3: spec 更新 + Issue 起票

照合結果に基づき、spec 更新と Issue 起票を行う。

### 設計判断 → spec 更新 + ADR 記録

spec ファイルを実装に合わせて更新する。更新はサブエージェントに委譲する。

```
あなたは設計ドキュメントを更新するテクニカルライターです。

## 前提
- まずプロジェクトルートの CLAUDE.md を読み、プロジェクトの構成・規約を把握する

## やること
1. 対象の spec ファイルを読む
2. 対象の実装コードを読む
3. 以下の乖離について、spec を実装に合わせて更新する

## 更新対象
{乖離レポートから該当項目を転記}

## 更新の原則
- 実装の現状を正確に反映する — spec は「あるべき姿」ではなく「現在の設計」を表す
- 既存の記述スタイル・フォーマットに合わせる
- 変更箇所の周辺も整合性を確認する（型の変更が他のセクションに影響していないか等）
- 不要になった記述は削除する（コメントアウトで残さない）

## 出力
- 更新したファイルの一覧と変更内容の要約
```

設計判断には ADR を記録する。既存の `.adr/` ディレクトリの連番に続けて作成する。

```markdown
# ADR-{連番}: {判断の概要}

## ステータス
承認済み

## コンテキスト
{なぜこの変更が必要になったか — Issue 番号、レビューコメント、技術的制約など}

## 決定
{何をどう変えたか}

## 結果
{spec のどのファイルを更新したか}
```

### 実装誤り・漏れ → Issue 起票

`gh issue create` で Issue を起票する。関連する乖離はまとめて1つの Issue にしてよいが、レイヤーが異なる場合は分割する。

Issue のフォーマット:

```
gh issue create \
  --title "[spec-sync] {レイヤー}: {乖離の概要}" \
  --body "$(cat <<'EOF'
## 概要
spec-sync による照合で検出された実装の乖離。

## 乖離内容
{乖離レポートから該当項目を転記}

| # | 分類 | spec | 実装 | 内容 |
|---|------|------|------|------|
| {番号} | {誤り/漏れ} | {spec パス} | {実装パス} | {何が異なるか} |

## 対応方針
{spec 側が正しいので、実装を spec に合わせて修正する}

## 参照
- spec: {関連する spec ファイルパス}
EOF
)" \
  --label "spec-sync"
```

`spec-sync` ラベルが存在しない場合は事前に `gh label create spec-sync --description "spec-sync で検出された乖離" --color "d4c5f9"` で作成する。

起票前に `gh issue list --label spec-sync` で既存の重複 Issue がないか確認する。

## Phase 4: 結果報告

```
spec 同期が完了しました！

## 照合結果
- 照合対象: {レイヤー数}レイヤー / {ドメイン数}ドメイン
- 乖離検出: {総数}件
  - 設計判断: {数}件 → spec 更新済み
  - 実装誤り: {数}件 → Issue 起票済み
  - 実装漏れ: {数}件 → Issue 起票済み

## 更新ファイル
- spec: {更新したファイルの一覧}
- ADR: {作成した ADR の一覧}

## 起票した Issue
{実装誤り・漏れがある場合}
- #{Issue番号}: {タイトル}
- #{Issue番号}: {タイトル}

issue-implement スキルで対応できます。
```

## 原則

- **メインエージェントはオーケストレーションのみ** — 照合・spec 更新はサブエージェントに委譲する
- **コードは修正しない** — このスキルの責務は spec の同期。コード修正は implement-audit や implement-* に委ねる
- **プロジェクト全体をスキャンする** — 複数回の Issue 対応やレビュー修正で蓄積した乖離をまとめて洗い出すため、spec/ の全ファイルを照合対象にする
- **自走する** — 照合から spec 更新・Issue 起票まで、ユーザー確認を挟まずに一気通貫で実行する
- **既存の記述スタイルを維持する** — spec の更新は内容の正確性が目的。フォーマットの改善は含めない
- **ADR で判断を追跡可能にする** — 設計判断による spec 変更は必ず ADR を残す。「なぜ変わったか」を追えるようにする
- 委譲方式・並列実行・失敗時の扱い・委譲時のコンテキストは `../_shared/references/subagent-policy.md` に従う
