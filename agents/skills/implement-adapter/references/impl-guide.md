# アダプター実装ガイド

`spec/database/` の設計とドメインのポートインターフェースに基づいて、アダプターを実装する。

## サブエージェントへの指示テンプレート

```markdown
あなたは設計ドキュメントに基づいてアダプターを実装するエンジニアです。

## 前提
- まずプロジェクトルートの CLAUDE.md を読み、使用言語・ディレクトリ構成・規約・コマンドを把握する

## 対象
- 設計ファイル: spec/database/
- ドメインのポート定義: CLAUDE.md に記載のドメイン層ディレクトリ内のポート定義

## やること
1. CLAUDE.md を読み、プロジェクトの構成・規約を把握する
2. spec/database/ を読み、DB設計を理解する
3. ドメインのポートインターフェースを読む
4. **実装すべきメソッドを漏れなく洗い出す** — 各ポートインターフェースの全メソッドを一覧にする。これが実装すべき全項目であり、後でカバレッジ表になる
5. 実装例があれば参照する（{実装例パス}）
6. 既存の実装があれば Read で読んで現状を把握する
7. 手順4で洗い出した全メソッドを実装する。インターフェースを満たすためだけの空実装・固定値返しで埋めない
8. CLAUDE.md に記載の typecheck・lint コマンドを実行し、エラーがなくなるまで修正する

## 重要な原則
- ポートインターフェースに準拠する — シグネチャだけ合わせて中身を省略しない
- 外部ライブラリの使い方は既存の実装パターンに合わせる
- ディレクトリ構成は CLAUDE.md とプロジェクトの既存構造に従う
- 設計に問題がある場合は実装せずにその旨を報告する

## 省略を残さない

実際の DB アクセスを伴わないスタブ（`return []`/固定値返し）・ダミーデータ・`not implemented` 系シグナル・空ボディやエラー握り潰し・一部のメソッドだけ実装して残りを放置・「// 省略」等のコメントによる省略は、typecheck が通っても「未完成」。ポートが要求するクエリ・永続化を実際に行う。実装できないメソッドは黙ってスタブ化せず、カバレッジ表に「未実装」と理由付きで明記する（メインエージェントが判断）。

## 出力
- 変更したファイルの一覧と変更内容の要約
- **カバレッジ表（必須）** — ポートの全メソッドを1行ずつ列挙し、実装箇所と状態を示す。未実装・スタブのまま残っているものも必ず行に残す:

  | ポート.メソッド | 実装ファイル:行 | 状態 |
  |----------------|----------------|------|
  | {RepositoryName.method} | path:line | 完全 / 部分 / 未実装 |

  「部分」「未実装」がある行には理由を添える。
- typecheck・lint の実行結果
- 設計との乖離（あれば: 乖離の内容・理由・実装での対応方針）
- 判断に迷った点や懸念事項（あれば）
```

## 完了確認

サブエージェント完了後、メインエージェントが typecheck + lint を実行して確認する。エラーがあればサブエージェントを再起動して修正させる。
