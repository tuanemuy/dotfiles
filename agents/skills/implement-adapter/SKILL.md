# Implement Adapter — アダプター実装＋レビュー

spec/database/ の設計とドメインのポートインターフェースに基づき、アダプターをサブエージェントで実装する。実装後にアダプター層に特化したレビューサイクルを回す。

**メインエージェントの役割はオーケストレーションのみ。** コードの読み書きはすべてサブエージェントに委譲する。

## Phase 1: 準備

### spec/ の確認

以下が存在するか確認する:

- `spec/database/` — DB設計
- `spec/domains/` — ドメイン設計（ポートインターフェースの参照元）

### プロジェクトコンテキストの把握

CLAUDE.md と README.md を読み、以下を確認する:

- 使用言語・フレームワーク
- ディレクトリ構成（アダプター層の配置先、ドメイン層のポート定義場所）
- typecheck・lint コマンド
- コーディング規約

### 進捗管理

`spec/progress.md` を確認し、アダプター層の状態を把握する。ドメイン層が実装済みであることを確認する。

### 実装例の確認

`docs/backend_implementation_example.md` が存在するか確認する。存在する場合はサブエージェントに参照先として渡す。

## Phase 2: 実装

`references/impl-guide.md` を読み、その手順に従ってアダプターを実装する。

サブエージェント完了後:

1. 報告を確認する
2. typecheck + lint を実行する
3. エラーがあればサブエージェントを再起動して修正させる
4. `spec/progress.md` を更新する

## Phase 3: レビューサイクル

全アダプターの実装が完了したら、`references/review-guide.md` を読み、その手順に従ってレビューと修正を行う。

**2回連続で Blocker 0件 かつ Warning 0件** になったらレビュー完了。最大5ラウンド。5ラウンドで収束しない場合はユーザーに報告して判断を委ねる。

レビュー記録は `.review/adapter/review-{連番}.md` に保存する。

## Phase 4: 完了

1. サブエージェントから報告された設計との乖離を確認し、必要に応じて:
   - `spec/database/` の該当ドキュメントを更新する
   - 設計判断があった場合は `.adr/{連番}-{topic}.md` に記録する
2. git commit する
3. `spec/progress.md` を更新する
4. サマリーを出す

## 原則

- **メインエージェントはオーケストレーションのみ** — コードの読み書きはサブエージェントに委譲する
- **プロジェクトコンテキストに従う** — 言語・ディレクトリ構成・規約は CLAUDE.md / README.md から読み取る
- 設計ドキュメントに忠実に実装する — スコープ外の改善を含めない
- ポートインターフェースに準拠する
- **TODO・FIXME・仮実装を残さない**
- 品質チェック（typecheck + lint）はごまかさない
- **spec ドキュメントを実装と同期させる** — 設計の誤りや不足を発見したら spec を更新する
- **設計判断は `.adr/` に記録する** — 設計を変更・拡張した理由を ADR として残す
- 委譲方式・並列実行・失敗時の扱い・委譲時のコンテキストは `../_shared/references/subagent-policy.md` に従う
