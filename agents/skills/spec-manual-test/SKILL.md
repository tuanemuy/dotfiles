# Manual Test

`spec/` の設計ドキュメントをもとに、人間が実行できるマニュアルテストドキュメントを生成する。
シナリオ（ユーザーフロー）単位で構成し、正常系・異常系の両方をカバーする。

## 設計思想

### テストは「ユーザーの目的達成」を検証する

自動テスト（`spec/testcases/`）がコードレベルの正しさを検証するのに対し、
マニュアルテストは「ユーザーが実際の画面を操作して目的を達成できるか」を検証する。
シナリオに沿って画面を操作し、期待どおりの結果が得られるかを確認する手順書になる。

### 誰が読んでも迷わず実行できる

開発者もQAも同じドキュメントを使う。前提知識を仮定せず、環境構築からテストデータの準備、
操作手順、期待結果まで、順番に追えば実行できる粒度で書く。

## 前提条件

以下の spec/ 成果物が存在すること（最低限シナリオとページ設計があれば生成可能）:

- `spec/scenario/` — シナリオ設計（必須）
- `spec/pages/` — ページ設計（必須）
- `spec/usecases/` — ユースケース設計（あれば活用）
- `spec/testcases/` — テストケース定義（あれば活用）

## 成果物

```
spec/manual-tests/
├── index.md                          # テスト一覧・実行管理表
└── {scenario-category}.md            # シナリオ単位のテストドキュメント
```

## Workflow

```
Phase 1: 準備
  spec/ の成果物を読み込み、テスト対象のシナリオを特定する

Phase 2: テストドキュメント生成
  シナリオ単位で手順書を生成する
  → references/test-doc-guide.md の手順に従う

Phase 3: レビュー
  生成したドキュメントをレビューし、漏れや曖昧さを修正する
```

---

## Phase 1: 準備

以下を読み込む:

1. `spec/scenario/index.md` でシナリオ一覧を把握する
2. `spec/scenario/{category}.md` で各シナリオの詳細フローを確認する
3. `spec/pages/index.md` で画面構成を把握する
4. `spec/usecases/` があれば、操作に対応するユースケースの入出力・エラーケースを確認する
5. `spec/testcases/` があれば、テスト条件・期待結果を取り込む

ユーザーが特定のシナリオだけを指定している場合は、対象を絞って進める。

---

## Phase 2: テストドキュメント生成

`references/test-doc-guide.md` を読み、その手順に従ってドキュメントを生成する。

成果物:
- `spec/manual-tests/index.md` — テスト一覧と実行管理表
- `spec/manual-tests/{scenario-category}.md` — シナリオ別テストドキュメント

---

## Phase 3: レビュー

出力先: `spec/manual-tests/review/{連番}.md`

レビュー観点:
- すべてのシナリオ（正常系・異常系）がテストケースとしてカバーされているか
- 手順が曖昧でなく、初見の人でも迷わず実行できるか
- 期待結果が具体的で検証可能か（「正しく表示される」ではなく「○○が△△と表示される」）
- テストデータや前提条件が明記されているか
- 画面遷移がページ設計と整合しているか
- ユースケースのエラーケースがテスト手順に反映されているか

問題があれば修正し、再レビュー。再レビュー時は前回のレビューファイルを参照せず、
チェックリストに沿って一から評価する。クリーンになるまで繰り返す。

---

## 完了報告

すべてのドキュメント生成とレビューが完了したら、サマリーを出す。

```
マニュアルテストドキュメントの生成が完了しました！

## 成果物

### テスト一覧
- spec/manual-tests/index.md

### テストドキュメント
- spec/manual-tests/{category}.md (x{数})

### テストケース総数
- 正常系: {数} 件
- 異常系: {数} 件
- 合計: {数} 件
```

---

## 原則

- シナリオ設計を起点にする — 自動テストケースだけでなく、ユーザーフローに沿った手順にする
- 具体的に書く — 「適切な値を入力」ではなく「メールアドレス欄に `test@example.com` と入力」
- 期待結果は検証可能にする — 目で見て判断できる具体的な状態を書く
- 1つのテストケースは1つの検証目的に絞る
- spec/ の成果物と整合性を保つ — 画面名・操作名は spec/ の用語と一致させる
