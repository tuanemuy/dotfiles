# テストドキュメント生成ガイド

## Step 1: テスト一覧の作成

すべてのシナリオカテゴリーを一覧化し、実行管理表を作成する。

出力先: `spec/manual-tests/index.md`

フォーマット:

```markdown
# マニュアルテスト一覧

## 概要

| 項目 | 値 |
|---|---|
| 対象プロジェクト | {プロジェクト名} |
| 作成日 | {日付} |
| spec バージョン | {最終更新のコミットハッシュまたは日付} |

## テストドキュメント一覧

| カテゴリー | ファイル | テストケース数 | 正常系 | 異常系 | 境界値 |
|---|---|---|---|---|---|
| {カテゴリー名} | [{category}.md](./{category}.md) | {数} | {数} | {数} | {数} |
| ... | ... | ... | ... | ... | ... |
| **合計** | | **{総数}** | **{総数}** | **{総数}** | **{総数}** |

## 実行記録

テスト実行時に以下の表をコピーして記録に使う。

| 実行日 | 実行者 | 環境 | 結果 | 備考 |
|---|---|---|---|---|
| | | | /件 PASS | |
```

## Step 2: シナリオ別テストドキュメントの生成

`spec/scenario/{category}.md` のシナリオごとにテストドキュメントを生成する。

出力先: `spec/manual-tests/{scenario-category}.md`

各ドキュメントは以下の構成にする:

### 2-1: ヘッダー

```markdown
# {カテゴリー名} テスト

## 概要

このドキュメントは {カテゴリーの説明} に関するマニュアルテストの手順書です。

対応シナリオ: `spec/scenario/{category}.md`
```

### 2-2: 前提条件

テスト実行に必要な環境・データ・状態を明記する。

```markdown
## 前提条件

### 環境
- {必要な環境の説明}

### テストデータ
- {テストに必要なデータの説明}

### 事前準備
1. {準備手順1}
2. {準備手順2}
```

前提条件は以下のソースから導出する:
- シナリオの開始状態（ログイン済み、特定のページにいる等）
- ユースケースの入力条件
- テストケース定義の前提条件列

### 2-3: テストケース

シナリオの各フローをテストケースに変換する。正常系を先に、異常系を後に配置する。

各テストケースは以下のテーブル形式で記述する:

```markdown
## TC-{連番}: {テスト名}

**種別**: 正常系 / 異常系 / 境界値
**目的**: {このテストで何を検証するか}

| # | 操作 | 期待結果 |
|---|---|---|
| 1 | {具体的な操作手順} | {目で見て確認できる具体的な結果} |
| 2 | {次の操作} | {期待結果} |
| ... | ... | ... |

**確認ポイント**:
- {テスト完了時に追加で確認すべき事項があれば記載}
```

テストケース作成のルール:

- **操作は具体的に書く**: 「情報を入力する」ではなく「名前欄に `テスト太郎` と入力する」
- **期待結果は検証可能にする**: 「正しく表示される」ではなく「画面上部に `登録が完了しました` と表示される」
- **1ケース1目的**: 複数の検証目的を1つのケースに詰め込まない
- **画面名は spec/pages/ の用語と一致させる**: 独自の呼び方をしない
- **操作の起点を明記する**: 各テストケースの最初のステップで、どの画面から開始するか書く

### 2-4: テストケースの情報源と変換ルール

テストケースは以下の優先順位で情報を統合して作成する:

1. **シナリオ（最優先）**: ユーザーフローの正常系・異常系がテストの骨格になる
2. **ページ設計**: 操作対象の画面名・機能・入力項目をシナリオと紐づける
3. **ユースケース（あれば）**: エラーケースや入力バリデーションの条件を補完する
4. **テストケース定義（あれば）**: 前提条件・期待結果の具体的な値を取り込む

変換の流れ:
- シナリオの正常フロー → 正常系テストケース（フローの各ステップを操作手順に分解）
- シナリオの異常フロー → 異常系テストケース
- ユースケースのエラーケース → 追加の異常系テストケース（シナリオにない観点を補完）
- テストケース定義の境界値 → 境界値テストケース

### 2-5: 異常系・エッジケースの体系的導出（必須）

シナリオはハッピーパス中心に書かれるため、2-4 の変換だけではテストが正常系に偏る。
以下の観点チェックリストをカテゴリーごとに一巡し、該当する操作・画面があれば
テストケースを追加する。該当しない場合も「検討した」ことをカバレッジ表（2-6）に残す。

| 観点 | 導出するテストケースの例 |
|---|---|
| **入力バリデーション** | 必須項目の未入力、形式不正（メール・URL・日付等）、型不一致（数値欄に文字列） |
| **境界値** | 文字数の上限ちょうど／超過、数値の最小・最大・ゼロ・負数、0件・1件・最大件数 |
| **認証・権限** | 未ログインでのアクセス、他ユーザーのリソースへのアクセス、権限のないロールでの操作 |
| **空状態・初期状態** | データが0件のときの一覧・検索結果・ダッシュボードの表示 |
| **重複・競合** | 同一データの二重登録、一意制約違反（メールアドレス重複等）、二重送信（連打） |
| **削除・変更の影響** | 参照されているデータの削除、編集中に他者が更新したデータの保存 |
| **操作の中断・逸脱** | 入力途中でのページ離脱・戻るボタン、確認ダイアログでのキャンセル |
| **特殊入力** | 絵文字・機種依存文字・HTMLタグ・前後の空白・極端に長い文字列 |
| **UIの状態** | 送信中の再操作、エラー後のリトライ、必須項目エラー表示後の修正 → 再送信 |

導出のルール:

- **入力フォームのある画面**: 必須フィールドごとに未入力ケース、制約のあるフィールドごとに境界値ケースを最低1件ずつ作る
- **ユースケースのエラーケース**: 全件をテストケースに変換する。UIから再現不能なもの（インフラ障害等）は対象外として理由を記録する
- **CRUD操作**: 削除・更新には「対象が存在しない／他者に変更された」ケースを検討する
- **期待結果は回復可能性まで書く**: エラーが出ることだけでなく、「入力値が保持されている」「エラーメッセージが `○○` と表示され、修正して再送信できる」まで検証する

### 2-6: カバレッジ表（必須）

各テストドキュメントの末尾に、異常系の網羅を証明するカバレッジ表を付ける。

```markdown
## カバレッジ

### ユースケースエラーケース対応表

| ユースケース | エラーケース | 対応TC | 備考 |
|---|---|---|---|
| {ユースケース名} | {エラーケース} | TC-{番号} | |
| {ユースケース名} | {エラーケース} | 対象外 | {UIから再現不能な理由} |

### 観点チェックリスト

| 観点 | 対応TC | 対象外の理由 |
|---|---|---|
| 入力バリデーション | TC-{番号}, TC-{番号} | |
| 境界値 | TC-{番号} | |
| 認証・権限 | 対象外 | {このカテゴリーに認証が絡む操作がない等} |
| ... | ... | ... |
```

「対応TC」が空欄のまま（対象外の理由もない）行があれば、テストケースの導出漏れ。
Step 2-5 に戻って追加する。

## Step 3: レビュー

生成したドキュメントを SKILL.md のレビュー観点に沿ってレビューする。

出力先: `spec/manual-tests/review/{連番}.md`
