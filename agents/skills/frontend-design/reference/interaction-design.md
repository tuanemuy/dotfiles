# インタラクションデザイン

## 8つのインタラクティブ状態

すべてのインタラクティブ要素にこれらの状態のデザインが必要：

| 状態 | いつ | ビジュアル処理 |
|-------|------|------------------|
| **デフォルト** | 休止時 | ベーススタイリング |
| **ホバー** | ポインターが上に（タッチではない） | 微妙な浮き上がり、色の変化 |
| **フォーカス** | キーボード/プログラムフォーカス | 目に見えるリング（下記参照） |
| **アクティブ** | 押されている | 押し込み、暗く |
| **無効** | インタラクティブでない | 不透明度低下、ポインターなし |
| **ローディング** | 処理中 | スピナー、スケルトン |
| **エラー** | 無効な状態 | 赤いボーダー、アイコン、メッセージ |
| **成功** | 完了 | 緑のチェック、確認 |

**よくある見落とし**: フォーカスなしのホバーデザイン、またはその逆。これらは異なるものです。キーボードユーザーはホバー状態を見ません。

## フォーカスリング：正しく実装する

**代替なしの`outline: none`は絶対に使わない。** アクセシビリティ違反です。代わりに`:focus-visible`を使用してキーボードユーザーにのみフォーカスを表示：

```css
/* マウス/タッチのフォーカスリングを非表示 */
button:focus {
  outline: none;
}

/* キーボードのフォーカスリングを表示 */
button:focus-visible {
  outline: 2px solid var(--color-accent);
  outline-offset: 2px;
}
```

**フォーカスリングデザイン**：
- ハイコントラスト（隣接色に対して最低3:1）
- 2-3pxの太さ
- 要素からのオフセット（内側ではなく）
- すべてのインタラクティブ要素で一貫

## フォームデザイン：非自明なポイント

**プレースホルダーはラベルではない**—入力時に消えます。常に目に見える`<label>`要素を使用してください。**blurでバリデート**し、キーストロークごとではなく（例外：パスワード強度）。エラーはフィールドの**下**に`aria-describedby`で接続して配置。

## ローディング状態

**楽観的更新**: 即座に成功を表示し、失敗時にロールバック。低リスクアクション（いいね、フォロー）に使用し、支払いや破壊的アクションには使用しない。**スケルトンスクリーン > スピナー**—コンテンツの形をプレビューし、汎用的なスピナーよりも速く感じます。

## モーダル：inertアプローチ

モーダルでのフォーカストラップには以前は複雑なJavaScriptが必要でした。今は`inert`属性を使用：

```html
<!-- モーダルが開いているとき -->
<main inert>
  <!-- モーダルの背後のコンテンツはフォーカスもクリックもできない -->
</main>
<dialog open>
  <h2>モーダルタイトル</h2>
  <!-- フォーカスはモーダル内に留まる -->
</dialog>
```

またはネイティブの`<dialog>`要素を使用：

```javascript
const dialog = document.querySelector('dialog');
dialog.showModal();  // フォーカストラップ付きで開き、Escapeで閉じる
```

## Popover API

ツールチップ、ドロップダウン、非モーダルオーバーレイにはネイティブのポップオーバーを使用：

```html
<button popovertarget="menu">メニューを開く</button>
<div id="menu" popover>
  <button>オプション1</button>
  <button>オプション2</button>
</div>
```

**メリット**: ライトディスミス（外側クリックで閉じる）、適切なスタッキング、z-indexの戦争なし、デフォルトでアクセシブル。

## 破壊的アクション：確認よりも取り消し

**取り消しは確認ダイアログよりも優れている**—ユーザーは確認を無意識にクリックします。UIから即座に削除し、取り消しトーストを表示し、トースト期限後に実際に削除。確認を使用するのは、本当に不可逆なアクション（アカウント削除）、高コストアクション、またはバッチ操作のみ。

## キーボードナビゲーションパターン

### ローヴィングTabindex

コンポーネントグループ（タブ、メニュー項目、ラジオグループ）では、1つのアイテムがタブ可能で、矢印キーでグループ内を移動：

```html
<div role="tablist">
  <button role="tab" tabindex="0">タブ1</button>
  <button role="tab" tabindex="-1">タブ2</button>
  <button role="tab" tabindex="-1">タブ3</button>
</div>
```

矢印キーが`tabindex="0"`をアイテム間で移動。Tabキーは次のコンポーネント全体に移動。

### スキップリンク

キーボードユーザーがナビゲーションを飛ばすためのスキップリンク（`<a href="#main-content">メインコンテンツへスキップ</a>`）を提供。画面外に非表示にし、フォーカス時に表示。

## ジェスチャーの発見可能性

スワイプで削除などのジェスチャーは見えません。その存在をヒントする：

- **部分的に表示**: 端から削除ボタンを覗かせる
- **オンボーディング**: 初回使用時のコーチマーク
- **代替手段**: 常に目に見えるフォールバックを提供（「削除」のあるメニュー）

ジェスチャーをアクション実行の唯一の方法にしないでください。

---

**避けるべきこと**: 代替なしのフォーカスインジケーター削除。ラベルとしてのプレースホルダーテキスト使用。44x44px未満のタッチターゲット。汎用的なエラーメッセージ。ARIA/キーボードサポートなしのカスタムコントロール。
