# モーションデザイン

## 持続時間：100/300/500ルール

タイミングはイージングより重要です。これらの持続時間はほとんどのUIに適切：

| 持続時間 | ユースケース | 例 |
|----------|----------|----------|
| **100-150ms** | 即時フィードバック | ボタン押下、トグル、色変更 |
| **200-300ms** | 状態変化 | メニューオープン、ツールチップ、ホバー状態 |
| **300-500ms** | レイアウト変更 | アコーディオン、モーダル、ドロワー |
| **500-800ms** | エントランスアニメーション | ページ読み込み、ヒーローの表示 |

**退出アニメーションは登場より速く**—登場時間の約75%を使用。

## イージング：正しいカーブを選ぶ

**`ease`を使用しないでください。** 最適になることが稀な妥協です。代わりに：

| カーブ | 用途 | CSS |
|-------|---------|-----|
| **ease-out** | 要素の登場 | `cubic-bezier(0.16, 1, 0.3, 1)` |
| **ease-in** | 要素の退出 | `cubic-bezier(0.7, 0, 0.84, 0)` |
| **ease-in-out** | 状態の切り替え（行って戻る） | `cubic-bezier(0.65, 0, 0.35, 1)` |

**マイクロインタラクションには指数関数的カーブを使用**—実際の物理（摩擦、減速）を模倣するため自然に感じます：

```css
/* Quart out - スムーズで洗練（推奨デフォルト） */
--ease-out-quart: cubic-bezier(0.25, 1, 0.5, 1);

/* Quint out - やや劇的 */
--ease-out-quint: cubic-bezier(0.22, 1, 0.36, 1);

/* Expo out - キビキビ、自信のある */
--ease-out-expo: cubic-bezier(0.16, 1, 0.3, 1);
```

**bounceとelasticカーブは避ける。** 2015年にはトレンドでしたが、今ではダサくアマチュアっぽい。実際の物体は止まるときにバウンドしない—スムーズに減速する。オーバーシュートエフェクトはコンテンツではなくアニメーション自体に注意を引きます。

## アニメーションすべきプロパティは2つだけ

**transform**と**opacity**のみ—それ以外はすべてレイアウトの再計算を引き起こします。高さアニメーション（アコーディオン）には、`height`を直接アニメーションする代わりに`grid-template-rows: 0fr → 1fr`を使用。

## スタッガードアニメーション

よりクリーンなスタッガーにはCSSカスタムプロパティを使用：`animation-delay: calc(var(--i, 0) * 50ms)`と各アイテムに`style="--i: 0"`。**合計スタッガー時間に上限を設ける**—50msで10アイテム = 合計500ms。多くのアイテムの場合、アイテムごとの遅延を減らすか、スタッガー数に上限を。

## モーション軽減

これはオプションではありません。前庭障害は40歳以上の成人の約35%に影響します。

```css
/* 通常通りアニメーションを定義 */
.card {
  animation: slide-up 500ms ease-out;
}

/* モーション軽減の代替を提供 */
@media (prefers-reduced-motion: reduce) {
  .card {
    animation: fade-in 200ms ease-out;  /* モーションの代わりにクロスフェード */
  }
}

/* または完全に無効化 */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}
```

**保持すべきもの**: プログレスバー、ローディングスピナー（速度を落とす）、フォーカスインジケーターなどの機能的なアニメーションは引き続き動作すべき—空間的な移動なしで。

## 知覚されるパフォーマンス

**サイトの実際の速度は誰も気にしない—速く*感じる*かどうかだけ。** 知覚は実際のパフォーマンスと同じくらい効果的です。

**80msの閾値**: 私たちの脳は知覚を同期するために約80msの感覚入力をバッファします。80ms未満のものは瞬間的で同時に感じます。これがマイクロインタラクションの目標です。

**能動的 vs 受動的時間**: 受動的な待ち（スピナーを見つめる）は能動的な関与よりも長く感じます。バランスを変える戦略：

- **先行開始**: ローディング中にすぐにトランジションを開始する（iOSアプリのズーム、スケルトンUI）。ユーザーは作業が行われていると感じる。
- **早期完了**: コンテンツを段階的に表示—すべてを待たない。ビデオのバッファリング、プログレッシブ画像、ストリーミングHTML。
- **楽観的UI**: インターフェースを即座に更新し、失敗をグレースフルに処理。Instagramのいいねはオフラインでも機能—UIが即座に更新され、後で同期。低リスクアクションに使用；支払いや破壊的操作には避ける。

**イージングは知覚される持続時間に影響する**: Ease-in（完了に向けて加速）はタスクを短く感じさせます。ピークエンド効果が最後の瞬間を重く評価するためです。Ease-outは登場に満足感がありますが、タスクの終わりに向けてのease-inは知覚される時間を圧縮します。

**注意**: 速すぎる応答は知覚される価値を下げる場合があります。ユーザーは複雑な操作（検索、分析）の即座の結果を信用しないことがあります。短い遅延が「実際の作業」が行われていることを示す場合もあります。

## パフォーマンス

`will-change`を予防的に使用しない—アニメーションが差し迫っているときのみ（`:hover`、`.animating`）。スクロールトリガーアニメーションにはスクロールイベントではなくIntersection Observerを使用；1回アニメーション後にunobserve。一貫性のためにモーショントークンを作成（持続時間、イージング、一般的なトランジション）。

---

**避けるべきこと**: すべてをアニメーション化する（アニメーション疲れは本物）。UIフィードバックに500ms以上を使用する。`prefers-reduced-motion`を無視する。遅いローディングを隠すためにアニメーションを使用する。
