# UXライティング

## ボタンラベルの問題

**「OK」「送信」「はい/いいえ」は絶対に使わない。** これらは安易で曖昧です。具体的な動詞＋目的語パターンを使用：

| 悪い例 | 良い例 | 理由 |
|-----|------|-----|
| OK | 変更を保存 | 何が起きるかを伝える |
| 送信 | アカウントを作成 | 結果に焦点 |
| はい | メッセージを削除 | アクションを確認する |
| キャンセル | 編集を続ける | 「キャンセル」の意味を明確にする |
| ここをクリック | PDFをダウンロード | 遷移先を説明する |

**破壊的アクション**には破壊を名前にする：
- 「取り消す」ではなく「削除」（削除は永久、取り消しは復元可能を暗示）
- 「選択を削除」ではなく「5件を削除」（数を表示）

## エラーメッセージ：フォーミュラ

すべてのエラーメッセージは以下に答えるべき：(1) 何が起きたか？ (2) なぜ？ (3) どう修正するか？ 例：「メールアドレスが無効です。@を含めてください。」であり「無効な入力」ではない。

### エラーメッセージテンプレート

| 状況 | テンプレート |
|-----------|----------|
| **フォーマットエラー** | 「[フィールド]は[形式]である必要があります。例：[例]」 |
| **必須項目の欠如** | 「[欠けているもの]を入力してください」 |
| **権限拒否** | 「[対象]へのアクセス権がありません。[代替手段]」 |
| **ネットワークエラー** | 「[対象]に接続できませんでした。接続を確認して[アクション]。」 |
| **サーバーエラー** | 「こちら側で問題が発生しました。調査中です。[代替アクション]」 |

### ユーザーを責めない

エラーを言い換える：「日付をMM/DD/YYYY形式で入力してください」であり「無効な日付を入力しました」ではない。

## 空の状態はチャンス

空の状態はオンボーディングの瞬間：(1) 簡潔に認識し、(2) 埋めることの価値を説明し、(3) 明確なアクションを提供。「まだプロジェクトがありません。最初のプロジェクトを作成して始めましょう。」であり「アイテムなし」だけではない。

## ボイス vs トーン

**ボイス**はブランドの個性—どこでも一貫。
**トーン**は瞬間に合わせて適応。

| 瞬間 | トーンの変化 |
|--------|------------|
| 成功 | 祝福的、簡潔：「完了！変更が公開されました。」 |
| エラー | 共感的、助けになる：「うまくいきませんでした。試してみてください...」 |
| ローディング | 安心させる：「作業を保存中...」 |
| 破壊的な確認 | 真面目、明確：「このプロジェクトを削除しますか？取り消しできません。」 |

**エラーにユーモアを使用しない。** ユーザーはすでにフラストレーションを感じています。可愛いのではなく、助けになりましょう。

## アクセシビリティのためのライティング

**リンクテキスト**は単独で意味を持つ必要がある—「ここをクリック」ではなく「料金プランを見る」。**altテキスト**は画像ではなく情報を説明する—「チャート」ではなく「Q4に売上が40%増加」。装飾画像には`alt=""`。**アイコンボタン**にはスクリーンリーダーのコンテキストのために`aria-label`が必要。

## 翻訳のためのライティング

### 拡張を計画する

ドイツ語テキストは英語より約30%長い。スペースを確保：

| 言語 | 拡張率 |
|----------|-----------|
| ドイツ語 | +30% |
| フランス語 | +20% |
| フィンランド語 | +30-40% |
| 中国語 | -30%（文字数は少ないが同じ幅） |

### 翻訳に適したパターン

数値を分離する（「あなたに3件の新しいメッセージがあります」ではなく「新しいメッセージ：3」）。完全な文を単一の文字列として（語順は言語によって異なる）。略語を避ける（「5 mins ago」ではなく「5 minutes ago」）。翻訳者に文字列がどこに表示されるかのコンテキストを提供。

## 一貫性：用語の問題

一つの用語を選んで使い続ける：

| 不一貫 | 一貫 |
|--------------|------------|
| 削除 / 取り消す / ゴミ箱 | 削除 |
| 設定 / 環境設定 / オプション | 設定 |
| サインイン / ログイン / 入る | サインイン |
| 作成 / 追加 / 新規 | 作成 |

用語集を作成し、それを守る。バラエティは混乱を生む。

## 冗長なコピーを避ける

見出しで説明されているなら、導入文は冗長。ボタンが明確なら、再度説明しない。一度言い、うまく言う。

## ローディング状態

具体的にする：「読み込み中...」ではなく「下書きを保存中...」。長い待ち時間には期待を設定する（「通常30秒かかります」）か進捗を表示。

## 確認ダイアログ：控えめに使う

ほとんどの確認ダイアログはデザインの失敗—代わりに取り消しを検討。確認が必要な場合：アクションを名前にし、結果を説明し、具体的なボタンラベルを使用する（「はい」/「いいえ」ではなく「プロジェクトを削除」/「プロジェクトを残す」）。

## フォームの説明

フォーマットは説明ではなくプレースホルダーで表示。自明でないフィールドにはなぜ聞いているかを説明。

---

**避けるべきこと**: 説明なしの専門用語。ユーザーを責める（「エラーが発生しました」→「この項目は必須です」）。曖昧なエラー（「問題が発生しました」）。バラエティのための用語の変更。エラーへのユーモア。
