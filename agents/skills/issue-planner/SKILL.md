# Issue Planner

GitHub Issueに対する実装計画を、1人のプランナーが初稿を作成し、2視点並列レビュー → 修正のループ（最大3周）で磨き込むスキル。レビューの観点を明確に分けることで、要件の見落としと実現可能性の問題を早期に検出し、堅牢な計画を生み出す。

## Workflow Overview

```
Issue取得 → 複雑度判定
  → [小規模] 直接プランニング → plan.md作成 → testing.md作成 → 完了
  → [中〜大規模] 1プランナーで初稿作成 → plan.md作成
    → 設計判断があれば adr.md に記録
    → レビューループ（最大3周）:
       2視点並列レビュー（要件カバレッジ / アーキ・リスク）
       → 問題点を plan.md に反映
       → 全レビュアーが「問題なし」なら終了、未解決なら再レビュー
    → 動作確認計画を testing.md に作成 → 完了
```

## Step 1: Issue の取得と分析

Issue番号またはURLを受け取ったら、まず内容を把握する。

```bash
gh issue view <Issue番号> --json title,body,labels,assignees,milestone,comments
```

Issueが取得できない場合（番号違い、権限不足、リポジトリ違いなど）はユーザーに確認して正しい情報を得る。推測で進めない。

Issueの内容を読み、以下を整理する:
- **目的**: このIssueは何を達成したいのか
- **要件**: 具体的に何が求められているか
- **制約**: 明示的・暗黙的な制約はあるか
- **スコープ**: Issueの範囲に含まれるもの・含まれないもの
- **コメントの要点**: コメント欄での議論・方針変更・合意事項があれば要約する（コメントには最新の合意や却下されたアプローチなど重要な情報が含まれることが多い）

### 既存計画の確認

`.issue/{Issue番号}/plan.md` が既に存在するかチェックする。存在する場合はユーザーに以下を確認する:
- 既存の計画を上書きして新しく作り直すか
- 既存の計画をベースに更新するか

ユーザーの指示に従って進める。

### 複雑度の判定

Issueの内容から複雑度を判定する:

- **小規模**: 変更対象が1-2ファイル程度で、修正内容が明確（typo修正、設定値変更、小さなバグ修正など）
- **中〜大規模**: 複数ファイルにまたがる変更、設計判断が必要、新機能追加、リファクタリングなど

**小規模の場合**: サブエージェントの委譲は行わず、自分で直接コードを調査して計画を作成し、Step 2 の plan.md テンプレートに沿って出力する（**複雑度メタデータには「小規模」と記録する** — 後段の issue-implement Phase 3 がこれを読んでレビューを単発化する）。Step 3（ADR）は不要、Step 4（レビューループ）もスキップし、Step 5（testing.md）→ Step 6（完了報告）に進む。

**中〜大規模の場合**: Step 2 に進み、1プランナーで初稿を作成する。**複雑度メタデータには「中〜大規模」と記録する**。

この情報は後続のエージェントに渡すため、正確に把握する。

## Step 2: 実装計画の作成

サブエージェントを1つ起動して初稿を作成する（委譲方式は `_shared/references/subagent-policy.md` に従う）。後続のレビューループで品質を担保するため、プランナーには特定の視点を強く与えず、Issueの要件を満たすバランスの取れた計画を立てさせる。

### エージェントへのプロンプト

```
あなたはIssueの実装計画を立てるエンジニアです。
Issueの要件を満たし、**プロジェクトとしてあるべきアーキテクチャ**に沿った実装計画を立ててください。

## 対象Issue
- Issue番号: #{Issue番号}
- タイトル: {タイトル}
- 内容:
{Issueの本文}
- コメント・議論の要点:
{関連するコメントの要約 — 方針変更や合意事項があれば必ず含める}

## やること
1. `gh issue view {Issue番号}` でIssueの詳細とコメントを確認する
2. **プロジェクトのあるべきアーキテクチャを把握する** — `spec/` 配下の設計ドキュメント、`CLAUDE.md`、`README.md`、アーキテクチャ関連ドキュメントを読み、プロジェクトが目指している設計・レイヤー構造・規約を理解する
3. Issueに関連する既存のコードを調査する（検索ツールやファイル参照ツールを使って徹底的に）
4. 既存コードが「あるべき姿」と一致しているか、それとも乖離しているかを判断する
5. 実装計画を立てる

## 重要な原則
- **Issueの要件を満たすことに集中する** — スコープ外の改善やリファクタリングは含めない
- **既存コードを盲目的に模倣しない** — 既存コードはあくまで現状の参考。プロジェクトのあるべきアーキテクチャ・設計ドキュメントが正であり、既存実装がそれと乖離している場合は、Issue範囲内で理想形に寄せる
- 既存コードと理想形が一致している箇所では、確立されたパターンを尊重する
- 変更の影響範囲を明確にする

## 出力フォーマット
以下のフォーマットで計画を返して:

### 概要
{1-2文で実装方針を要約}

### 調査結果
- 関連ファイル: {調査で見つけた関連ファイルとその役割}
- あるべきアーキテクチャ: {`spec/` や `CLAUDE.md` 等から読み取った設計方針・レイヤー構造・規約}
- 既存実装の状態: {あるべき姿と一致している箇所 / 乖離している箇所 / 乖離があれば本Issueでどう扱うか}
- 依存関係: {この変更が影響する他のコンポーネント}

### 実装ステップ
1. {ステップ1}
   - 対象ファイル: `パス`
   - 変更内容: {具体的に何をするか}
   - 理由: {なぜこの変更が必要か}
2. {ステップ2}
   ...

### 設計判断
- {判断が必要なポイントとその推奨案}

### リスクと注意点
- {潜在的なリスクや注意すべき点}

### テスト方針
- {どうやって動作確認するか}
```

エージェントが失敗・タイムアウトした場合は、自分で同じ手順をなぞって初稿を作成する。

### plan.md への保存

エージェントの返した内容を `.issue/{Issue番号}/plan.md` に保存する。ディレクトリがなければ作成する。

```markdown
# 実装計画 — Issue #{Issue番号}: {タイトル}

**Issue:** #{Issue番号}
**作成日:** {日付}
**複雑度:** {小規模 / 中〜大規模}

---

## 目的

{Issueが達成したいこと}

## スコープ

### 含まれるもの
- {スコープ内の項目}

### 含まれないもの
- {明示的にスコープ外とするもの}

## 実装ステップ

### 1. {ステップタイトル}

- **対象ファイル:** `パス`
- **変更内容:** {具体的な変更}
- **理由:** {この変更が必要な理由}

### 2. {ステップタイトル}
...

## 設計判断

{重要な設計判断のサマリー — 詳細はadr.mdを参照}

## リスクと注意点

- {リスク1}
- {リスク2}

## テスト方針

- {テスト項目1}
- {テスト項目2}
```

## Step 3: ADR の作成

計画策定の過程で**技術的な設計判断**（アーキテクチャ・データモデル・ライブラリ選定・抽象化粒度など、トレードオフのある技術選択）があった場合のみ `.issue/{Issue番号}/adr.md` に記録する。

要件・スコープの取捨選択（何をやる/やらない、優先順位）は ADR ではなく plan.md の「スコープ」セクションで扱う。些細な実装詳細も対象外。


```markdown
# ADR — Issue #{Issue番号}: {タイトル}

## ADR-001: {判断のタイトル}

### Status
Proposed

### Context
{なぜこの判断が必要になったか — どのような選択肢があったか}

### Decision
{何を選んだか、なぜそれを選んだか}

### Consequences
- 良い点: {メリット}
- トレードオフ: {あれば}

---
```

レビューループ中に新たな設計判断が生じた場合は、その時点で adr.md に追記する。

## Step 4: レビューループ（最大3周）

plan.md が完成したら、**2つのサブエージェントを並列に起動してレビュー → 反映** を最大3周繰り返す（委譲方式は `_shared/references/subagent-policy.md` に従う）。各エージェントには異なる視点を与えることで、同一プロンプトで収束する問題を回避する。

レビューの観点は「要件カバレッジ」と「アーキテクチャ整合性・リスク」の2つに絞る — 計画段階で最も致命的になりやすい2点を集中的に検証する。Phase 3（実装後の PR レビュー）でレイヤー別の多角レビューが入る前提で、ここは要件と実現可能性の確認に振り切る。

### 終了条件

以下のいずれか先に到達したら終了:

- **全レビュアーが「問題点ゼロ」を報告** — 計画が両視点で問題なしと判断された
- **3周完了** — 残課題があれば plan.md の「未解決事項」セクションに明記して終了し、実装フェーズに引き継ぐ

### 各周の流れ

1. 2エージェント並列でレビュー実施
2. 両方の結果を統合（同じ問題を指摘していたら1件にまとめる）
3. 問題点（要修正）を plan.md に反映、改善提案はスコープ内で妥当なもののみ取り込む
4. 重要な設計判断が生じたら adr.md に追記
5. plan.md 末尾の「レビュー履歴」セクションに反映内容を追記
6. 両レビュアーとも問題点ゼロなら終了、未解決の問題が残れば次の周に進む

いずれかのレビュアーが失敗・タイムアウトした場合は、成功した結果だけで進める。両方失敗した場合はその周をスキップして次の周に進む（連続2周失敗ならレビューループを打ち切り、未解決事項として記録）。

### 共通テンプレート

各エージェントには以下のテンプレートを渡す（**視点** の部分だけ差し替える）:

```
あなたはIssueの実装計画をレビューするエンジニアです。
あなたの視点: **{視点}**

## レビュー対象
- Issue番号: #{Issue番号}
- 計画ファイル: .issue/{Issue番号}/plan.md
- 設計判断ファイル: .issue/{Issue番号}/adr.md（存在する場合）

## やること
1. `gh issue view {Issue番号}` でIssueの要件とコメントでの議論を確認する
2. `.issue/{Issue番号}/plan.md` を読む
3. `.issue/{Issue番号}/adr.md` が存在すれば読み、設計判断の妥当性も確認する
4. 計画に関連する既存コードを調査する
5. 上記の視点で計画をレビューする

## 重要な原則
- **Issueの要件を満たすかどうかが最重要** — スコープ外の提案はしない
- Issueのコメント欄での議論・合意事項と計画が矛盾していないか確認する
- 具体的で実行可能なフィードバックを出す
- 良い点も明確に伝える
- 問題点が見当たらない場合は「問題点ゼロ」と明示する（無理に粗探しをしない）

## 出力フォーマット

#### 問題点（要修正）
- **[P-001]** {問題の説明}
  - 理由: {なぜ問題か}
  - 提案: {どう修正すべきか}

（問題点がなければ「問題点ゼロ」と明記）

#### 改善提案（検討推奨）
- **[S-001]** {提案の説明}
  - 理由: {なぜ改善になるか}

#### 良い点
- {計画の優れている点}
```

### 視点1: 要件カバレッジ・スコープ整合性

```
**Issueの要件カバレッジ・スコープ整合性** を重視してレビューしてください。Issue本文・コメントで合意された要件をすべて満たしているか、スコープ外の作業が紛れ込んでいないか、見落とされた受け入れ基準がないかを徹底的に確認してください。
```

### 視点2: アーキテクチャ整合性・リスク

```
**プロジェクトのあるべきアーキテクチャとの整合性・実現可能性・リスク** を重視してレビューしてください。`spec/` 配下の設計ドキュメント・`CLAUDE.md`・アーキテクチャ関連ドキュメントを読み、プロジェクトが目指している設計・レイヤー構造・規約に計画が沿っているかを確認してください。既存コードに合わせるだけで「あるべき姿」から外れていないか、逆にIssue範囲を超えて理想形を追求しすぎていないかを両方向で見てください。加えて、提案された手順が現実的に動くか、見落とされた依存関係や副作用はないか、エッジケースは考慮されているか、設計判断のトレードオフは適切かを徹底的に確認してください。
```

### レビュー履歴セクション

各周で反映した内容は plan.md の末尾に追記する:

```markdown
## レビュー履歴

### 1周目
**修正した点**:
- {P-001への対応: 何をどう修正したか}
- {P-002への対応}

**取り込んだ改善提案**:
- {S-001: 何を取り込んだか}

**見送った提案とその理由**:
- {S-002: 見送り理由 — スコープ外、等}

### 2周目
...

### 3周目（実施した場合）
...
```

レビュアーが「問題点ゼロ」を返した周も「{N}周目: 両視点とも問題点ゼロで終了」と記録する。

### 未解決事項セクション（3周完了しても残った場合のみ）

3周終わっても解消できない問題があれば、plan.md の末尾に追記する:

```markdown
## 未解決事項

- **[P-XXX]** {未解決の問題}
  - 経緯: {なぜ3周で解消できなかったか}
  - 引き継ぎ: {実装フェーズでどう扱うべきか}
```

## Step 5: 動作確認計画の作成

plan.md の修正が完了したら、実装後に実機で動作確認すべき観点とその方法を `.issue/{Issue番号}/testing.md` にまとめる。

自動テスト（ユニットテスト等）で担保できる範囲ではなく、実際にアプリケーションを動かして人間が確認すべき項目に焦点を当てる。plan.md の実装ステップと要件から、何をどう確認すれば「この実装が正しく動いている」と言えるかを具体的に記述する。

### Step 5a: 確認に必要な実行コマンドの特定（testing.md を書く前に必ず実行）

testing.md の「確認環境」セクションには、このIssueの変更を確認するために必要な実行コマンドだけを書く。プロジェクト全体のセットアップ手順（依存パッケージのインストール等）は不要。

まず plan.md の実装ステップから「この変更を確認するには何を動かす必要があるか」を特定する。次に、そのコマンドを以下のソースから実際にファイルを読んで確認する:

1. **CLAUDE.md** — プロジェクトルートと変更対象パッケージの CLAUDE.md
2. **README.md**
3. **package.json** — `scripts` セクションの**全項目**に目を通す。ローカル実行系のキーワード（dev, start, build）だけで止めない
4. **リポジトリルートの設定ファイル** — Makefile やデプロイ設定ファイルが存在すれば読む
5. **既存の `.issue/*/testing.md`** — Glob で探し、過去の記載を参考にする

testing.md に書く前に、各コマンドが上記ソースに実在するかを実際に確認すること。記憶や推測でコマンドを書いてはいけない。

**変更確認の流れの全体像を考えること**: 変更箇所を検証環境で確認するために必要なステップを、抜けなく洗い出す。検証環境のサーバー起動方法と、デプロイ方法（ステージングや本番環境への反映手順）は別セクションとして分けて記載する。マイグレーションやシードが必要かもプロジェクトの実態から判断する。

**禁止事項**:

- 「手動でデプロイしてください」「環境に応じて適宜実行してください」のような曖昧な逃げ文句は書かない。該当するコマンドがドキュメントに存在するなら、確認した上でそのコマンドを記載する。
- 上記ソースで確認できないコマンドは testing.md に書かない。
- どうしても該当手順が見つからなかった場合のみ「要確認: {具体的に何の手順が見つからなかったか}」と明記する。

### Step 5b: testing.md の作成

Step 5a で確認した情報を使って testing.md を作成する。

```markdown
# 動作確認計画 — Issue #{Issue番号}: {タイトル}

**Issue:** #{Issue番号}
**作成日:** {日付}

---

## 確認環境

このIssueの変更を確認するために必要な手順のみ記載（プロジェクト全体のセットアップは省略）。

### 検証環境の起動
{検証用サーバーを起動するコマンド。Step 5a で確認した実在するコマンドのみ記載。}

### デプロイ方法
{ステージング・本番環境への反映手順。不要（検証環境のみで確認できる）場合は「なし」と記載。Step 5a で確認した実在するコマンドのみ記載。}

## 確認項目

### 1. {確認項目のタイトル}

- **目的:** {何が正しく動くことを確認するか}
- **手順:**
  1. {具体的な操作手順}
  2. {次の操作}
  3. ...
- **期待結果:** {どうなれば正しいか}
- **確認ポイント:** {特に注意して見るべき箇所}

### 2. {確認項目のタイトル}
...

## エッジケース・異常系

### 1. {確認項目のタイトル}

- **目的:** {どんな異常系を確認するか}
- **手順:**
  1. {操作手順}
- **期待結果:** {エラー時にどう振る舞うべきか}

## 既存機能への影響確認

- {変更によって影響を受ける可能性のある既存機能と、その確認方法}

## 確認チェックリスト

- [ ] {確認項目1}
- [ ] {確認項目2}
- [ ] {既存機能の動作確認}
- [ ] ...
```

## Step 6: 完了報告

計画が完成したら、サマリーを出す。小規模Issueの場合は該当しない項目を省略する。

```
Issue #{Issue番号} の実装計画が完成しました！

- 計画ファイル: .issue/{Issue番号}/plan.md
- 設計判断: .issue/{Issue番号}/adr.md（{あり/なし}）
```
