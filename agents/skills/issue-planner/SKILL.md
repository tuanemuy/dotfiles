# Issue Planner

GitHub Issueに対する実装計画を、複数の独立したエージェントによる並列プランニング → 統合 → レビュー → 修正のサイクルで作成する。各エージェントに異なる視点を与えることで、多角的な検討が可能になり、見落としを減らし、より堅牢な計画を生み出す。

## Workflow Overview

```
Issue取得 → 複雑度判定
  → [小規模] 直接プランニング → plan.md作成 → 完了
  → [中〜大規模] 3エージェント並列プランニング → 計画比較・統合 → plan.md作成
    → 設計判断があれば adr.md に記録
    → 3エージェント並列レビュー → レビュー結果を反映して plan.md 修正
    → 動作確認計画を testing.md に作成 → 完了
```

## Step 1: Issue の取得と分析

Issue番号またはURLを受け取ったら、まず内容を把握する。

```bash
gh issue view <Issue番号> --json title,body,labels,assignees,milestone,comments
```

Issueが取得できない場合（番号違い、権限不足、リポジトリ違いなど）はユーザーに確認して正しい情報を得る。推測で進めない。

Issueの内容を読み、以下を整理する:
- **目的**: このIssueは何を達成したいのか
- **要件**: 具体的に何が求められているか
- **制約**: 明示的・暗黙的な制約はあるか
- **スコープ**: Issueの範囲に含まれるもの・含まれないもの
- **コメントの要点**: コメント欄での議論・方針変更・合意事項があれば要約する（コメントには最新の合意や却下されたアプローチなど重要な情報が含まれることが多い）

### 既存計画の確認

`.issue/{Issue番号}/plan.md` が既に存在するかチェックする。存在する場合はユーザーに以下を確認する:
- 既存の計画を上書きして新しく作り直すか
- 既存の計画をベースに更新するか

ユーザーの指示に従って進める。

### 複雑度の判定

Issueの内容から複雑度を判定する:

- **小規模**: 変更対象が1-2ファイル程度で、修正内容が明確（typo修正、設定値変更、小さなバグ修正など）
- **中〜大規模**: 複数ファイルにまたがる変更、設計判断が必要、新機能追加、リファクタリングなど

**小規模の場合**: エージェント並列は行わず、自分で直接コードを調査して計画を作成し、Step 3 の plan.md テンプレートに沿って出力する。Step 4（ADR）は不要、Step 5-6（レビュー）もスキップし、Step 7（testing.md）→ Step 8（完了報告）に進む。

**中〜大規模の場合**: Step 2 に進み、3エージェント並列プランニングを実行する。

この情報は後続のエージェントに渡すため、正確に把握する。

## Step 2: 3エージェントで並列プランニング

3つのサブエージェントを **並列に** 起動する（委譲方式は `_shared/references/subagent-policy.md` に従う）。各エージェントは独立して Issue を読み、既存実装を調査し、実装計画を立てる。

各エージェントには異なる視点を与える。同じプロンプトだとLLMは似た結論に収束しやすいため、視点を分けることで本当に異なるアプローチが生まれ、統合時により良い計画が得られる。

### エージェント1: アーキテクチャ整合性重視

```
あなたはIssueの実装計画を立てるエンジニアです。
あなたの視点: **既存アーキテクチャとの整合性・パフォーマンス** を特に重視して計画を立ててください。既存の設計パターンにどう合わせるか、パフォーマンスへの影響はないかを深く考えてください。

## 対象Issue
- Issue番号: #{Issue番号}
- タイトル: {タイトル}
- 内容:
{Issueの本文}
- コメント・議論の要点:
{関連するコメントの要約 — 方針変更や合意事項があれば必ず含める}

## やること
1. `gh issue view {Issue番号}` でIssueの詳細とコメントを確認する
2. Issueに関連する既存のコードを調査する（検索ツールやファイル参照ツールを使って徹底的に）
3. 実装計画を立てる

## 重要な原則
- **Issueの要件を満たすことに集中する** — スコープ外の改善やリファクタリングは含めない
- 既存コードのパターンや規約を尊重する
- 変更の影響範囲を明確にする

## 出力フォーマット
以下のフォーマットで計画を返して:

### 概要
{1-2文で実装方針を要約}

### 調査結果
- 関連ファイル: {調査で見つけた関連ファイルとその役割}
- 既存パターン: {プロジェクトで使われている関連パターンや規約}
- 依存関係: {この変更が影響する他のコンポーネント}

### 実装ステップ
1. {ステップ1}
   - 対象ファイル: `パス`
   - 変更内容: {具体的に何をするか}
   - 理由: {なぜこの変更が必要か}
2. {ステップ2}
   ...

### 設計判断
- {判断が必要なポイントとその推奨案}

### リスクと注意点
- {潜在的なリスクや注意すべき点}

### テスト方針
- {どうやって動作確認するか}
```

### エージェント2: テスタビリティ・保守性重視

エージェント1と同じテンプレートを使うが、視点の部分を以下に変更する:

```
あなたの視点: **テスタビリティ・保守性・可読性** を特に重視して計画を立ててください。将来のメンテナンスがしやすい設計か、テストが書きやすい構造か、コードの意図が読み取りやすいかを深く考えてください。
```

### エージェント3: シンプルさ・最小変更重視

エージェント1と同じテンプレートを使うが、視点の部分を以下に変更する:

```
あなたの視点: **シンプルさ・最小限の変更** を特に重視して計画を立ててください。要件を満たす最もシンプルな方法は何か、変更量を最小限に抑えられないかを深く考えてください。YAGNI原則を意識してください。
```

## Step 3: 計画の比較と統合

3つのエージェントがすべて完了したら、それぞれの計画を比較する。いずれかのエージェントが失敗・タイムアウトした場合は、成功したエージェントの結果だけで進める（2つあれば十分に比較可能、1つでもそれをベースに計画を作成する）。

比較の観点:
- **要件カバレッジ**: Issueの要件をどれだけ漏れなく満たしているか
- **実現可能性**: 既存コードとの整合性、実装の具体性
- **影響範囲**: 変更の範囲が適切か（広すぎないか）
- **リスク**: 潜在的な問題の洗い出しができているか

最も優れた計画をベースに、他の計画の良い点を取り込んで統合する。

統合した計画を `.issue/{Issue番号}/plan.md` に保存する。ディレクトリがなければ作成する。

```markdown
# 実装計画 — Issue #{Issue番号}: {タイトル}

**Issue:** #{Issue番号}
**作成日:** {日付}

---

## 目的

{Issueが達成したいこと}

## スコープ

### 含まれるもの
- {スコープ内の項目}

### 含まれないもの
- {明示的にスコープ外とするもの}

## 実装ステップ

### 1. {ステップタイトル}

- **対象ファイル:** `パス`
- **変更内容:** {具体的な変更}
- **理由:** {この変更が必要な理由}

### 2. {ステップタイトル}
...

## 設計判断

{重要な設計判断のサマリー — 詳細はadr.mdを参照}

## リスクと注意点

- {リスク1}
- {リスク2}

## テスト方針

- {テスト項目1}
- {テスト項目2}

## 参考: エージェント比較

| 観点 | エージェント1 (アーキテクチャ) | エージェント2 (保守性) | エージェント3 (シンプルさ) |
|------|-------------------------------|------------------------|---------------------------|
| ベース採用 | {○/×} | {○/×} | {○/×} |
| 取り込んだ点 | {あれば} | {あれば} | {あれば} |
```

## Step 4: ADR の作成

計画策定の過程で設計判断があった場合、`.issue/{Issue番号}/adr.md` に記録する。

設計判断とは、複数の選択肢がある中で特定のアプローチを選んだケース。エージェント間で意見が分かれた点は特に記録する価値がある。些細な実装詳細はADRにしない。

```markdown
# ADR — Issue #{Issue番号}: {タイトル}

## ADR-001: {判断のタイトル}

### Status
Proposed

### Context
{なぜこの判断が必要になったか — どのような選択肢があったか}

### Decision
{何を選んだか、なぜそれを選んだか}

### Consequences
- 良い点: {メリット}
- トレードオフ: {あれば}

---
```

設計判断がない場合はADRファイルを作成しなくてよい。

## Step 5: 3エージェントで並列レビュー

plan.mdが完成したら、3つのサブエージェントを **並列に** 起動してレビューさせる（委譲方式は `_shared/references/subagent-policy.md` に従う）。各エージェントには同じ指示を与え、独立してレビューさせる。いずれかのエージェントが失敗・タイムアウトした場合は、成功したエージェントの結果だけでレビューを進める。

各サブエージェントには以下を渡す:

```
あなたはIssueの実装計画をレビューするエンジニアです。

## レビュー対象
- Issue番号: #{Issue番号}
- 計画ファイル: .issue/{Issue番号}/plan.md
- 設計判断ファイル: .issue/{Issue番号}/adr.md（存在する場合）

## やること
1. `gh issue view {Issue番号}` でIssueの要件とコメントでの議論を確認する
2. `.issue/{Issue番号}/plan.md` を読む
3. `.issue/{Issue番号}/adr.md` が存在すれば読み、設計判断の妥当性も確認する
4. 計画に関連する既存コードを調査する
5. 計画をレビューする

## 重要な原則
- **Issueの要件を満たすかどうかが最重要** — スコープ外の提案はしない
- Issueのコメント欄での議論・合意事項と計画が矛盾していないか確認する
- 具体的で実行可能なフィードバックを出す
- 良い点も明確に伝える

## 出力フォーマット

#### 問題点（要修正）
- **[P-001]** {問題の説明}
  - 理由: {なぜ問題か}
  - 提案: {どう修正すべきか}

#### 改善提案（検討推奨）
- **[S-001]** {提案の説明}
  - 理由: {なぜ改善になるか}

#### 良い点
- {計画の優れている点}
```

## Step 6: レビュー結果の反映

3つのレビュー結果を受け取ったら:

1. **問題点（要修正）** を最優先で計画に反映する
2. **改善提案（検討推奨）** はIssueのスコープ内で妥当なもののみ取り込む
3. 反映内容を plan.md に直接適用する（新しいファイルは作らない）
4. 重要な設計判断が生じた場合は adr.md に追記する
5. plan.md の末尾に「レビュー反映」セクションを追加する:

```markdown
## レビュー反映

### 修正した点
- {P-001への対応: 何をどう修正したか}
- {P-002への対応}

### 取り込んだ改善提案
- {S-001: 何を取り込んだか}

### 見送った提案とその理由
- {S-002: 見送り理由 — スコープ外、等}
```

## Step 7: 動作確認計画の作成

plan.md の修正が完了したら、実装後に実機で動作確認すべき観点とその方法を `.issue/{Issue番号}/testing.md` にまとめる。

自動テスト（ユニットテスト等）で担保できる範囲ではなく、実際にアプリケーションを動かして人間が確認すべき項目に焦点を当てる。plan.md の実装ステップと要件から、何をどう確認すれば「この実装が正しく動いている」と言えるかを具体的に記述する。

### Step 7a: 確認に必要な実行コマンドの特定（testing.md を書く前に必ず実行）

testing.md の「確認環境」セクションには、このIssueの変更を確認するために必要な実行コマンドだけを書く。プロジェクト全体のセットアップ手順（依存パッケージのインストール等）は不要。

まず plan.md の実装ステップから「この変更を確認するには何を動かす必要があるか」を特定する。次に、そのコマンドを以下のソースから実際にファイルを読んで確認する:

1. **CLAUDE.md** — プロジェクトルートと変更対象パッケージの CLAUDE.md
2. **README.md**
3. **package.json** — `scripts` セクションの**全項目**に目を通す。ローカル実行系のキーワード（dev, start, build）だけで止めない
4. **リポジトリルートの設定ファイル** — Makefile やデプロイ設定ファイルが存在すれば読む
5. **既存の `.issue/*/testing.md`** — Glob で探し、過去の記載を参考にする

testing.md に書く前に、各コマンドが上記ソースに実在するかを実際に確認すること。記憶や推測でコマンドを書いてはいけない。

**変更確認の流れの全体像を考えること**: 変更箇所を検証環境で確認するために必要なステップを、抜けなく洗い出す。検証環境のサーバー起動方法と、デプロイ方法（ステージングや本番環境への反映手順）は別セクションとして分けて記載する。マイグレーションやシードが必要かもプロジェクトの実態から判断する。

**禁止事項**:

- 「手動でデプロイしてください」「環境に応じて適宜実行してください」のような曖昧な逃げ文句は書かない。該当するコマンドがドキュメントに存在するなら、確認した上でそのコマンドを記載する。
- 上記ソースで確認できないコマンドは testing.md に書かない。
- どうしても該当手順が見つからなかった場合のみ「要確認: {具体的に何の手順が見つからなかったか}」と明記する。

### Step 7b: testing.md の作成

Step 7a で確認した情報を使って testing.md を作成する。

```markdown
# 動作確認計画 — Issue #{Issue番号}: {タイトル}

**Issue:** #{Issue番号}
**作成日:** {日付}

---

## 確認環境

このIssueの変更を確認するために必要な手順のみ記載（プロジェクト全体のセットアップは省略）。

### 検証環境の起動
{検証用サーバーを起動するコマンド。Step 7a で確認した実在するコマンドのみ記載。}

### デプロイ方法
{ステージング・本番環境への反映手順。不要（検証環境のみで確認できる）場合は「なし」と記載。Step 7a で確認した実在するコマンドのみ記載。}

## 確認項目

### 1. {確認項目のタイトル}

- **目的:** {何が正しく動くことを確認するか}
- **手順:**
  1. {具体的な操作手順}
  2. {次の操作}
  3. ...
- **期待結果:** {どうなれば正しいか}
- **確認ポイント:** {特に注意して見るべき箇所}

### 2. {確認項目のタイトル}
...

## エッジケース・異常系

### 1. {確認項目のタイトル}

- **目的:** {どんな異常系を確認するか}
- **手順:**
  1. {操作手順}
- **期待結果:** {エラー時にどう振る舞うべきか}

## 既存機能への影響確認

- {変更によって影響を受ける可能性のある既存機能と、その確認方法}

## 確認チェックリスト

- [ ] {確認項目1}
- [ ] {確認項目2}
- [ ] {既存機能の動作確認}
- [ ] ...
```

## Step 8: 完了報告

計画が完成したら、サマリーを出す。小規模Issueの場合は該当しない項目を省略する。

```
Issue #{Issue番号} の実装計画が完成しました！

- 計画ファイル: .issue/{Issue番号}/plan.md
- 設計判断: .issue/{Issue番号}/adr.md（{あり/なし}）
- 動作確認計画: .issue/{Issue番号}/testing.md
- 実装ステップ数: {数}
- ベース採用エージェント: {1(アーキテクチャ)/2(保守性)/3(シンプルさ)}
- レビューで修正した点: {数}件
- レビューで取り込んだ改善提案: {数}件
- 動作確認項目数: {数}件
```

## 重要な注意点

- 委譲方式（サブエージェント / OSプロセス / 親内シミュレート）と並列実行は `_shared/references/subagent-policy.md` に従う — プランニングもレビューも必ず並列
- **Issueの要件を満たすことに集中する** — スコープ外の改善やリファクタリングを計画に含めない。これが最も重要な原則
- 各エージェントには既存コードを**実際に読ませる** — 推測ではなく、検索とファイル参照で調査させる
- エージェント間の意見の相違は設計判断として adr.md に記録する価値がある
- plan.md は実装者が見てすぐ作業に取りかかれる具体性を持たせる
- Issueのコメント欄の議論を無視しない — コメントには方針変更や却下されたアプローチなど重要な文脈が含まれる
- エージェントが失敗・タイムアウトした場合は、成功した結果だけで柔軟に進める
