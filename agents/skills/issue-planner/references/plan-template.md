# plan.md / adr.md テンプレート

プランナー（および計画修正のサブエージェント）はこのテンプレートに従って `.issue/{Issue番号}/` 配下にファイルを直接書き込む。

**このテンプレートは下限であって上限ではない。** 後続フェーズが機械的に参照する項目（メタデータの **複雑度**、見出し「受け入れ基準」「実装ステップ」「スコープ」「テスト方針」、レビュー履歴・未解決事項の指摘ID）は必ず含めること。それ以外は、Issueの性質に応じてセクションを追加したり、分析を自由に深く書いてよい — テンプレートに収めるために思考を削らない。

**小規模 Issue では該当しないセクションを省略してよい**（例: 設計判断・リスクが無ければセクションごと書かない）。埋めるための作文はしない。

## plan.md

```markdown
# 実装計画 — Issue #{Issue番号}: {タイトル}

**Issue:** #{Issue番号}
**作成日:** {日付}
**複雑度:** {小規模 / 中〜大規模}

---

## 目的

{Issueが達成したいことを1-2文で。受け入れ基準と重複する内容をここで繰り返さない}

## 受け入れ基準

Issue 本文・コメントの要件を「何ができたら満たしたと言えるか」の検証可能な基準に分解し、対応する実装ステップを紐づける。レビュー（要件カバレッジ視点）・PR レビュー・testing.md はこの表を基準にする。

| # | 基準（検証可能な形で） | 由来 | 対応ステップ |
|---|---|---|---|
| AC-1 | {例: 一覧画面で◯◯がソートできる} | {Issue本文 / コメントの合意} | {ステップ番号} |

## スコープ

### 含まれないもの
- {明示的にスコープ外とするもの — 受け入れ基準に含めない理由があれば一言添える}

（「含まれるもの」は受け入れ基準が兼ねるので書かない）

## 調査結果

- 関連ファイル: {調査で見つけた関連ファイルとその役割}
- あるべきアーキテクチャ: {`spec/` や `CLAUDE.md` 等から読み取った設計方針・レイヤー構造・規約}
- 既存実装の状態: {あるべき姿と一致している箇所 / 乖離している箇所 / 乖離があれば本Issueでどう扱うか}
- 依存関係: {この変更が影響する他のコンポーネント}

## 設計

実装ステップを書く前に、変更を**レイヤーの内側から外側へ**向かって設計する（プロジェクトのアーキテクチャ — spec/・CLAUDE.md — を正とする）。バックエンドに触れる変更なら、ファイル変更の列挙から入らず、まずドメインモデルへの影響から考えること。該当しないレイヤーは「なし」と一言書くか省略する。

### ドメインモデルへの影響
{エンティティ・値オブジェクト・不変条件・ドメインルール・ポート（インターフェース）に何を追加/変更するか。新しい概念が増えるのか、既存の概念の責務が変わるのか。影響がなければ「なし」と理由を一言}

### ユースケース / アプリケーションロジック
{どのユースケースを追加/変更するか。ドメインに置くべきロジックが紛れ込んでいないか}

### アダプター / 永続化 / 外部連携
{リポジトリ実装・スキーマ変更・マイグレーション・外部API}

### UI / プレゼンテーション
{画面・コンポーネント・ルーティングへの影響}
```

## adr.md

計画策定の過程で**技術的な設計判断**（アーキテクチャ・データモデル・ライブラリ選定・抽象化粒度など、トレードオフのある技術選択）があった場合のみ作成する。要件・スコープの取捨選択は ADR ではなく plan.md の「スコープ」で扱う。些細な実装詳細も対象外。

```markdown
# ADR — Issue #{Issue番号}: {タイトル}

## ADR-001: {判断のタイトル}

### Status
Proposed

### Context
{なぜこの判断が必要になったか — どのような選択肢があったか}

### Decision
{何を選んだか、なぜそれを選んだか}

### Consequences
- 良い点: {メリット}
- トレードオフ: {あれば}

---
```

既に adr.md がある場合は既存の連番の続きに追記する。

## レビュー履歴セクション（plan.md 末尾に追記）

```markdown
## レビュー履歴

### {N}周目
**修正した点**:
- {P-001への対応: 何をどう修正したか}

**取り込んだ改善提案**:
- {S-001: 何を取り込んだか}

**見送った提案とその理由**:
- {S-002: 見送り理由 — スコープ外、等}
```

レビュアーが「問題点ゼロ」を返した周は「{N}周目: 両視点とも問題点ゼロで終了」と記録する。

## 未解決事項セクション（3周完了しても残った場合のみ、plan.md 末尾に追記）

```markdown
## 未解決事項

- **[P-XXX]** {未解決の問題}
  - 経緯: {なぜ3周で解消できなかったか}
  - 引き継ぎ: {実装フェーズでどう扱うべきか}
```
