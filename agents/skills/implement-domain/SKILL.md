# Implement Domain — ドメインモデル実装＋レビュー

spec/domains/ の設計に基づき、ドメインモデルをサブエージェントで実装する。実装後にドメイン層に特化したレビューサイクルを回し、設計との整合性を担保する。

**メインエージェントの役割はオーケストレーションのみ。** コードの読み書きはすべてサブエージェントに委譲する。

## Phase 1: 準備

### spec/ の確認

以下が存在するか確認する:

- `spec/domains/` — ドメイン設計

存在しない場合はユーザーに確認する。

### プロジェクトコンテキストの把握

CLAUDE.md と README.md を読み、以下を確認する:

- 使用言語・フレームワーク
- ディレクトリ構成（ドメイン層の配置先）
- typecheck・lint コマンド
- コーディング規約

### 進捗管理

`spec/progress.md` を確認し、ドメイン層の状態を把握する。なければ新規作成する。

### 実装例の確認

`docs/backend_implementation_example.md` が存在するか確認する。存在する場合はサブエージェントに参照先として渡す。

## Phase 2: 実装

`references/impl-guide.md` を読み、その手順に従ってドメインモデルを実装する。

ドメインごとにサブエージェントを起動する。独立したドメインは並列で起動してよい。ドメイン間に依存関係がある場合は、依存先を先に実装してから依存元を起動する（spec/domains/ の設計から依存関係を読み取る）。

各サブエージェント完了後:

1. 報告を確認する
2. typecheck + lint を実行する
3. エラーがあればサブエージェントを再起動して修正させる
4. `spec/progress.md` を更新する

## Phase 2.5: 完全性ゲート

全ドメインの実装が完了したら、レビューサイクルに進む前に `../_shared/references/completeness-gate.md` を読み、その手順に従って完全性ゲートを実行する。

typecheck + lint はスタブ・ダミーデータ・部分実装・spec 要素のまるごと未実装を素通りさせる（型が通ってしまう）。このゲートで、独立したサブエージェントが spec/domains/ の全要素 vs 実コードを照合し、省略を検出する。欠落があればその項目だけを埋めさせて再ゲートし、欠落 0 件になってから Phase 3 へ進む。

- 対象レイヤー: domain
- spec: `spec/domains/`
- 入力: 実装したファイル群＋各サブエージェントのカバレッジ表

## Phase 3: レビューサイクル

完全性ゲートを通過したら、`references/review-guide.md` を読み、その手順に従ってレビューと修正を行う。

**2回連続で Blocker 0件 かつ Warning 0件** になったらレビュー完了。最大5ラウンド。5ラウンドで収束しない場合はユーザーに報告して判断を委ねる。

レビュー記録は `.review/domain/review-{連番}.md` に保存する。

## Phase 4: 完了

1. サブエージェントから報告された設計との乖離を確認し、必要に応じて:
   - `spec/domains/` の該当ドキュメントを更新する
   - 設計判断があった場合は `.adr/{連番}-{topic}.md` に記録する
2. git commit する
3. `spec/progress.md` を更新する
4. サマリーを出す

## 原則

共通原則は `../_shared/references/implement-principles.md` に従う。本レイヤー固有:

- 独立したドメインは並列で実装し、依存関係がある場合は依存先を先に実装する
