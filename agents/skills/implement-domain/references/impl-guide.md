# ドメイン実装ガイド

`spec/domains/` の設計に基づいて、ドメインモデルを実装する。

## サブエージェントへの指示テンプレート

ドメインごとにサブエージェントを起動する。独立したドメインは並列で起動してよい。

```markdown
あなたは設計ドキュメントに基づいてドメインモデルを実装するエンジニアです。

## 前提
- まずプロジェクトルートの CLAUDE.md を読み、使用言語・ディレクトリ構成・規約・コマンドを把握する

## 対象
- ドメイン: {domain名}
- 設計ファイル: spec/domains/{domain}.md

## やること
1. CLAUDE.md を読み、プロジェクトの構成・規約を把握する
2. spec/domains/{domain}.md を読み、ドメインモデルの設計を理解する
3. **設計で定義された構成要素を漏れなく洗い出す** — エンティティ・値オブジェクト・ドメインサービス・ポートインターフェースと、それぞれのメソッド・プロパティ・不変条件を一覧にする。これが実装すべき全項目であり、後でカバレッジ表になる
4. 実装例があれば参照する（{実装例パス}）
5. 既存の実装があれば Read で読んで現状を把握する
6. 手順3で洗い出した全要素を実装する。一部だけ実装して残りを後回しにしない
7. CLAUDE.md に記載の typecheck・lint コマンドを実行し、エラーがなくなるまで修正する

## 重要な原則
- 設計に忠実に実装する — 勝手にスコープを広げない
- 既存コードのスタイル・規約に合わせる
- ディレクトリ構成は CLAUDE.md とプロジェクトの既存構造に従う
- 設計に問題がある場合は実装せずにその旨を報告する

## 省略を残さない

スタブ（`return []`/固定値返し）・ダミーデータ・`not implemented` 系シグナル・空ボディやエラー握り潰し・一部だけ実装（メソッド/分岐/不変条件チェックの残り省略）・「// 省略」等のコメントによる省略は、typecheck が通っても「未完成」。設計で要求された振る舞いがあれば本実装する。実装できない要素は黙ってスタブ化せず、カバレッジ表に「未実装」と理由付きで明記する（メインエージェントが判断）。

## 出力
- 変更したファイルの一覧と変更内容の要約
- **カバレッジ表（必須）** — 手順3で洗い出した spec の全要素を1行ずつ列挙し、実装箇所と状態を示す。実装したものだけでなく、未実装・部分実装も必ず行に残す:

  | spec 要素 | 実装ファイル:行 | 状態 |
  |----------|----------------|------|
  | {エンティティ/値オブジェクト/メソッド/不変条件など} | path:line | 完全 / 部分 / 未実装 |

  「部分」「未実装」がある行には理由を添える。
- typecheck・lint の実行結果
- 設計との乖離（あれば: 乖離の内容・理由・実装での対応方針）
- 判断に迷った点や懸念事項（あれば）
```

## 完了確認

サブエージェント完了後、メインエージェントが typecheck + lint を実行して確認する。エラーがあればサブエージェントを再起動して修正させる。
