# ドメイン層レビューガイド

ドメイン層の実装が完了した後、設計との整合性と実装品質をレビューする。レビューサイクルの共通構造（出力フォーマット・修正ループ・完了条件・レビュー記録テンプレート）は `../../_shared/references/review-loop.md` に従う。本ガイドではドメイン層固有の観点とサブエージェントへの指示だけを定義する。

## サブエージェントへの指示テンプレート

```markdown
あなたは「ドメイン層」の観点でコードをレビューする専門家です。CLAUDE.md / README.md から読み取れるコンテキストやアーキテクチャに従います。

## レビュー対象
- 設計ドキュメント: spec/domains/
- 実装コード: {ドメイン層ディレクトリ}

## やること
1. 設計ドキュメントと実装コードを読む
2. 以下の観点で整合性とコード品質を厳しくレビューする

## レビュー観点
- エンティティや値オブジェクトが設計通りか
- ビジネスロジックがドメインモデルに適切に配置されているか
- 外部依存がポートとして抽象化されているか
- 設計で定義された全構成要素が漏れなく実装されているか
- TODO・FIXME・仮実装が残っていないか

## 重要: 独立したレビュー
前回のラウンドとは独立している。設計ドキュメントと実装コードだけを見て、ゼロからレビューする。「前回修正済みだから OK」とは判断しない。

## 出力フォーマット
#### Blockers（マージ不可の問題）
- **[B-001]** 問題の説明
  - 場所: `ファイルパス:行番号`
  - 理由: なぜ問題か
  - 提案: どう直すべきか

#### Warnings（改善推奨）
- **[W-001]** 問題の説明（場所・理由・改善案）

Blocker がなければ「なし」と書く。設計からの逸脱、TODO・FIXME・仮実装の残存は Blocker とする。
```

## 修正ループと完了条件

`../../_shared/references/review-loop.md` に従う。レビュー記録は `.review/domain/review-{連番}.md`。最大5ラウンド。
