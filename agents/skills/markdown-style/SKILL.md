Markdownドキュメントを、読み手がすぐ構造を把握でき、レンダラーが意図どおり表示できる状態に整える。新規に書くときも、既存のドキュメントを整形するときも、同じ原則に従う。

## なぜシンプルさが要るのか

Markdownは「軽い記法で構造を表す」ための言語であって、見た目を作り込むための言語ではない。装飾や独自ルールを足すほど、記法とノイズの区別がつかなくなり、レンダリング環境ごとに表示が割れ、編集時の差分も読みにくくなる。だから足すのではなく、構造を正しく表すために必要なものだけを残す。

判断に迷ったら「これは構造を表しているか、それとも見た目を飾っているだけか」で考える。飾りなら削る。

## ルール

### 見出し

見出しは文書の構造そのものなので、構造を正しく反映させる。

- h1（`#`）は1ドキュメントに1つだけ。文書のタイトルに使う。
- レベルを飛ばさない。`##` の下はいきなり `####` ではなく `###` に進む。
- 強調（`**太字**`）を見出しの代わりに使わない。見出しは必ず `#` で表す。強調はアウトラインに乗らず、目次にも拾われず、構造を壊す。

悪い例:

```markdown
**設定方法**
まず依存関係をインストールする。
```

良い例:

```markdown
### 設定方法

まず依存関係をインストールする。
```

### 強調・装飾

強調は本文の語句を際立たせるためのものに限る。装飾そのものを目的にしない。

- 区切り線（`---` や `***`）を入れない。セクションの区切りは見出しが担う。水平線は構造を表さず、視覚的な飾りでしかない。
- 絵文字や記号の連打で飾らない。意味のある箇所に控えめに使うのは構わないが、装飾目的の乱用はしない。
- 強調は `**太字**` か `*斜体*` を、本当に強調したい語にだけ使う。文全体を太字にしない。

### 空行とマーカーの統一

ブロック要素の境界を空行で明示すると、レンダラーが正しく解釈し、差分も読みやすくなる。

- 見出し・リスト・コードブロック・テーブルの前後には空行を入れる。
- 箇条書きのマーカーは `-` に統一する（`*` や `+` を混在させない）。
- 行末に余分な空白を残さない。
- 番号付きリストは順序に意味があるとき（手順など）だけ使う。順不同なら `-` を使う。

### 改行と段落

段落内・リスト項目内で勝手に改行しない。1つの段落は論理的に1行（ソフトラップに任せる）として書く。

- 文の途中で改行を入れると、ソース上の見た目を整えているだけで、レンダリング結果や意図とは無関係なノイズになる。
- 段落を分けたいときは空行を1つ入れる。
- どうしても明示的な改行が要る稀なケースだけ、行末の半角スペース2つではなく意味が伝わる構造（リストやテーブル）に置き換えられないか先に検討する。

悪い例:

```markdown
このツールは設定ファイルを読み込み、
内容を検証してから、
処理を実行する。
```

良い例:

```markdown
このツールは設定ファイルを読み込み、内容を検証してから、処理を実行する。
```

### コードとリンク

コードとリンクは専用の記法で表す。プレーンテキストに混ぜない。

- コードブロックには言語を指定する（` ```bash `, ` ```ts ` など）。シンタックスハイライトが効き、何のコードか伝わる。
- コマンド・ファイル名・識別子など、本文に登場するコード片はインラインコード（`` `getUser()` ``）で囲む。
- リンクテキストはリンク先の内容を表す。「こちら」「ここをクリック」のような中身のないテキストにしない。

悪い例:

```markdown
詳しくは[こちら](https://example.com/docs)を参照。
```

良い例:

```markdown
詳しくは[セットアップガイド](https://example.com/docs)を参照。
```

### 文章

ドキュメントの文章は、構造に語らせて言葉を減らす。

- 自明な前置きやまとめの定型文を書かない。「以下では〜について説明します」「以上が〜でした」のような、見出しを読めば分かる文は削る。
- 一文を短くする。1文に詰め込みすぎず、論点ごとに区切る。
- 同じことを言い換えて繰り返さない。重複は片方に寄せる。

## 整形の進め方

既存のドキュメントを整えるとき、または書いたMarkdownを仕上げるときは次の順で見る。

1. 対象を決める。引数 `target` があればそれ、なければ git diff の変更分や会話中で扱っているMarkdownを対象にする。
2. まず見出し構造を確認する。h1が1つか、レベル飛びがないか、強調で見出しを代用していないか。
3. 装飾を削る。区切り線・過剰な絵文字・飾りの強調を取り除く。
4. 記法を整える。空行・リストマーカー・コードフェンスの言語指定・リンクテキスト・末尾空白。
5. 段落内の不自然な改行をつなげ、1段落1行にする。
6. 文章を削る。自明な前置き・まとめ・重複を落とす。

意味を変えずに、構造と記法だけを整えるのが原則。内容の追加や大幅な書き換えが必要だと感じたら、勝手に進めずユーザーに確認する。
