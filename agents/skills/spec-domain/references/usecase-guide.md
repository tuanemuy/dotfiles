# ユースケース設計ガイド

ドメインごとにユースケースを詳細に定義する。
ユースケースはドメインモデルの操作を組み合わせて要件を実現するオーケストレーター。
ビジネスロジック自体はドメイン層に置き、ユースケースには含めない。

出力先: `spec/usecases/${domain}.md`

## 命名と粒度

- ユースケースは画面ではなくドメインの言葉で命名する（〜Page 等の画面語彙を使わない）
- 複数の読み取りを1つに合成してよいのは、合成をまたぐ規則があるときだけ。その合成にはドメインの名詞が付くはず — 名詞が「〜ページ」しかないなら分割し、束ねるのは画面の責務
- 1画面のデータ一式 ≠ 1ユースケース

## ユースケースの構成

各ユースケースについて以下を記載する:

### ユースケース名

### 概要

このユースケースが実現する操作の概要。

### 入力DTO

フィールドごとに:
- フィールド名、型、必須/任意、バリデーションルール

### 出力DTO

フィールドごとに:
- フィールド名、型

### 処理フロー

主要なステップを番号付きで記述する。
各ステップで呼び出すドメインモデルの振る舞いやポートを明記する。

### エラーケース

発生しうるエラーを列挙する:
- エラーの条件
- エラーの種類（バリデーションエラー、ビジネスルール違反、外部サービスエラー等）

## レビュー

出力先: `spec/usecases/review/${連番}.md`

レビュー観点:

- **トレーサビリティ**: 上流の成果物に記載された全操作がユースケースとしてカバーされているか
- **命名**: 名前・DTOに画面の語彙が混入していないか。名詞なき読み取り合成がないか
- **DTO完全性**: 入力DTO・出力DTOのフィールドがすべて定義されているか
- **アーキテクチャ適合性**: ビジネスロジックがユースケース層に漏れていないか（ドメイン層の振る舞いを呼ぶだけになっているか）
- **エラー網羅性**: エラーケースが網羅されているか
- **単一責任**: 1つのユースケースが複数の独立した操作を含んでいないか

問題があれば修正し、再レビュー。再レビュー時は前回のレビューファイルを参照せず、チェックリストに沿って一から評価する。クリーンになるまで繰り返す。
